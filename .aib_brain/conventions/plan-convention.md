Purpose
-------
Define a strict, deterministic convention for creating and maintaining iteration plans that guide execution for a single AIB request iteration. The plan SHALL translate clarified requirements into concrete, verifiable tasks and guardrails, minimizing ambiguity and preventing divergent implementations.

Scope
-----
This convention applies to the `\<ITERATION_ID>-plan.md` file created under a single request folder in `.aib_memory/requests/<request-folder>/`. It governs:
- File naming and location
- Document structure and required sections
- Allowed content formats and schemas
- Operation rules (how it is created, updated, validated, and consumed)
- A canonical set of disambiguation questions that MUST be asked and resolved before execution

Deterministic File Name and Location (Normative)
------------------------------------------------
- File name: `\<ITERATION_ID>-plan.md`
  - `ITERATION_ID` is a two-digit numeric token matching `^[0-9]{2}$` (e.g., `01`, `02`).
- Location: `.aib_memory/requests/<request-folder>/`
- Exactly one plan file per iteration.
- If regenerated, the file MAY be replaced entirely; regeneration MUST be idempotent for the same inputs.

Creation & Update Rules (Normative)
-----------------------------------
- The plan is generated by the `create-plan` action for the **resolved** request and **resolved** iteration.
- Prerequisites:
  - A request folder exists with `request.md` and `iterations.md`.
  - The target iteration exists in `iterations.md` (state may be `Active` or `Completed`; generation SHOULD occur when `Active`).
- Inputs (read-only):
  - `request.md`
  - The current iteration’s `\<ITERATION_ID>-analysis.md` (if present)
  - The current iteration’s `\<ITERATION_ID>-questionnaire.md` with user selections/edits (if present)
  - `.aib_memory/references.md` for documentation paths and edit permissions
- Outputs (write):
  - `\<ITERATION_ID>-plan.md` (this file)
- On validation failure, generation MUST abort with a human-readable error and MUST NOT produce partial files.

Document Structure (Normative)
------------------------------
The plan document MUST contain the sections below in the exact order. Headings MUST be level 2 (`##`) unless explicitly stated. Subsections MAY use lower levels.

Formatting rule (Normative)
--------------------------
- The plan output MUST NOT use Markdown tables.
- Where a structured schema is required (dependencies, interfaces, config keys, milestones, documentation touchpoints), represent each record as a single bullet item using a deterministic key/value pattern.

1. ## Overview
   - Concise summary (≤ 10 lines) of what will be implemented this iteration and why.
   - Linkage statements referencing the relevant parts of `request.md` and the current iteration’s `analysis` / answered `questionnaire` (by section titles or IDs, not hyperlinks).

2. ## Scope of Work
   - **In Scope**: bullet list
   - **Out of Scope**: bullet list
   - **Assumptions**: bullet list (only assumptions not already captured in analysis)
   - **Constraints**: bullet list (e.g., OS, tooling, privacy/compliance, edit permissions from references)

3. ## Work Breakdown Structure (WBS)
   A numbered task list (1., 2., 3., …) where each task uses the following schema:

   ```
   ### Task <N>: <Task Name>
   **Intent:** <single-sentence goal>  
   **Inputs:** <files, configs, parameters, data preconditions>  
   **Outputs:** <artifacts produced or changed; file paths or product components>  
   **Procedure:** <ordered, concise steps>  
   **Done Criteria:** <objective pass/fail checks>    
   **Dependencies:** <Task IDs or external>  
   **Risk Notes:** <if any>  
   ```

   - Tasks MUST be **vertically sliceable**; each produces at least one verifiable output.
   - Every output MUST be referenced by a relative path or component identifier consistent with the product documentation structure.
   - Every “Done Criteria” MUST be objectively testable (binary pass/fail).

4. ## Dependencies & Interfaces
   - **Internal task dependencies:** bullet list of records. Each record MUST include:
     - `From Task: <N>`
     - `To Task: <N>`
     - `Dependency Type: FS|SS|FF|SF`
     - `Critical: Y|N`
     - `Notes: <text>`

     Example record format:
     - From Task: 1 | To Task: 2 | Dependency Type: FS | Critical: Y | Notes: ...

   - **External interfaces (systems, data, modules):** bullet list of records. Each record MUST include:
     - `Interface: <name>`
     - `Direction: In|Out`
     - `Protocol/Contract: <text>`
     - `Version: <text>`
     - `Notes: <text>`

     Example record format:
     - Interface: Data API X | Direction: In | Protocol/Contract: REST | Version: v2 | Notes: ...

5. ## Environment & Configuration
   Specify runtime environments, configuration keys, and secrets handling **without** inlining any actual secret values.
   - **Environments:** Dev/Test/Stage/Prod (as applicable)
   - **Config Keys:** bullet list of records. Each record MUST include:
     - `Key: <NAME>`
     - `Scope: Env|Global`
     - `Default: <value>`
     - `Allowed Range/Values: <text>`
     - `Source: <file/path>`
     - `Change Control: <text>`

     Example record format:
     - Key: FOO_TIMEOUT_MS | Scope: Env | Default: 30000 | Allowed Range/Values: 5000–60000 | Source: ./config/... | Change Control: PR to ...

   - **Secrets Handling:** reference the enterprise mechanism and rotation rules (do not store secrets here).

6. ## Testing Strategy (This Iteration)
   - **Test Types:** Unit, Integration, E2E, Performance, Security (as applicable)
   - **Coverage Targets:** (if defined)
   - **Data/Fixtures:** where they come from and how to generate safely
   - **Test Execution:** command(s) or procedure summary
   - **Acceptance Evidence:** artifacts captured (logs, screenshots, reports)

7. ## Observability & Quality Gates
   - **Key Metrics/Logs/Alerts** required to validate success of this iteration’s changes (names, thresholds).
   - **Quality Gates:** explicit pass/fail criteria that MUST be met before marking tasks or the iteration as done (e.g., all unit tests ≥ threshold, no critical vulnerabilities, DQ checks pass).

8. ## Documentation Touchpoints
   - Enumerate which product documentation files are impacted, with their relative paths as defined in `.aib_memory/references.md`. For each:
     - **Doc Path**
     - **Change Type:** create/update/no-change
     - **Update Trigger:** which task completes it
     - **Edit Allowed:** Y/N (from references)
     - **Notes**

   - If an impacted doc has `edit_allowed = N`, the plan MUST instruct to raise a follow-up request or obtain permission rather than editing.

9.  ## Milestones
    - **Milestones:** bullet list of records. Each record MUST include:
      - `Milestone: <ID or name>`
      - `Description: <text>`
      - `Due: <date/time or trigger>`
      - `Depends On: <Task IDs or external>`
      - `Exit Criteria: <text>`

      Example record format:
      - Milestone: M1 | Description: ... | Due: ... | Depends On: Task 1 | Exit Criteria: ...

10. ## Risks & Mitigations
    - A prioritized list of risks with **probability**, **impact**, and **mitigation/contingency**.

11. ## Acceptance & Handover
    - **Iteration Acceptance Criteria:** bullet list referencing Done Criteria and Quality Gates
    - **Handover Artifacts:** concrete files/locations to be shared with stakeholders (builds, reports, docs updates)
    - **Post-Iteration Follow-ups:** items to carry into next iteration, if any

Validation Rules (Normative)
----------------------------
A plan is **valid** if and only if:
- File name and location conform to this convention.
- All mandatory sections are present in order and non-empty (Overview, Scope of Work, Decision Gates, WBS, Dependencies & Interfaces, Environment & Configuration, Testing Strategy, Observability & Quality Gates, Documentation Touchpoints, Milestones, Risks & Mitigations, Acceptance & Handover).
- “Decision Gates” contains explicit answers to all canonical questions **relevant** to the iteration. If a question is not applicable, it MUST be marked `N/A` with a one-line justification.
- Every WBS task defines objective **Done Criteria** and at least one **Output** artifact.
- Documentation Touchpoints only reference paths present in `.aib_memory/references.md` and respect `edit_allowed`.
- No secrets are embedded; only references to secret stores/mechanisms are allowed.
- The plan has zero internal contradictions (e.g., out-of-scope items used as required inputs).

Operational Guidance (Informative)
----------------------------------
- Keep “Overview” and “Scope of Work” succinct to aid decision-making.
- Prefer fewer, larger tasks only if each still yields a verifiable artifact; otherwise split into smaller slices.
- Keep WBS readable: prefer ≤ 12 tasks per iteration and keep each task procedure to ≤ 6 steps unless strictly necessary.
- Use clear, stable identifiers for tasks (e.g., `Task 3`) and reference them in Dependencies.
- Align test commands and metrics with existing project standards to avoid bespoke one-offs.
- If any canonical question cannot be answered, halt finalization and escalate—do not proceed with execution.
