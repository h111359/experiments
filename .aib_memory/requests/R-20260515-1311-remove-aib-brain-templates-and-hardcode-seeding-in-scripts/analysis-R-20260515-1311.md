## Input Interpretation

The developer requests the complete removal of the `.aib_brain/templates/` directory from the AIB framework. The directory currently holds one file — `requests_register-template.md` — which is read by `initialize.py` via the `load_template()` helper in `common.py` to seed `.aib_memory/requests_register.md` during workspace initialization.

The stated intent is twofold: first, the register's format and validation rules should be authoritative in the existing `requests_register-convention.md` convention file rather than in a duplicate template file; second, the seeding logic should be hardcoded directly in `initialize.py`, consistent with the pattern already used for all other seeded artifacts (`context.md`, `input.md`, `instructions.md`). Once the template is removed and the seed is hardcoded, the `load_template` helper becomes dead code and should be deleted from `common.py` along with its test coverage in `test_tools_common.py`. Test fixtures that copy the templates folder to temporary workspaces must also be cleaned up.

## Research Results

### Workspace Findings

**Template usage confirmed as single-callsite:** `grep -r "load_template"` across the workspace confirms exactly two call sites: `initialize.py` (production, line 65) and `test_tools_common.py` (test, line 279). No other script, prompt, or convention file loads templates programmatically. Removing the function and the template file is safe.

**Register seed content is already hardcoded in test fixtures:** `tests/conftest.py` defines `REGISTER_CONTENT` as an explicit inline string (`"# Requests Register\n\n| request_id | title | folder | state | created_at | closed_at |\n| --- | --- | --- | --- | --- | --- |\n"`). This diverges from the template, which includes a "Validation rules" commentary block below the table. The test fixture represents the intended minimal seed; the hardcoded string in `initialize.py` should match this format.

**Three test files copy the templates folder:** `tests/conftest.py`, `tests/test_initialize.py`, and `tests/test_lifecycle_e2e.py` each contain a `TEMPLATES_DIR` constant and a loop that copies `*.md` files from `.aib_brain/templates/` to a temporary workspace. After removal, these loops will glob zero files and the `mkdir` calls will create an unnecessary empty directory. Both should be removed.

**`ensure_workspace` does not validate the templates folder:** `common.py`'s `ensure_workspace` only checks for the existence of `.aib_brain/`. Removing the `templates/` subfolder has no impact on workspace validation.

**`menu.py` contains "brain templates" in user-visible text:** Line 792 of `menu.py` refers to "re-seed it from the new brain templates" in the upgrade prompt. This should be updated to "brain assets" for post-removal accuracy.

**`context.md` has an "AIB Templates" component entry:** The component map in `context.md` lists `- **AIB Templates** (.aib_brain/templates/)`. This entry must be removed.

**`initialize.py` `_run_upgrade` comment references "brain templates":** Line 269 contains `# Step 4 — Re-seed .aib_memory/ from brain templates.` — this is an internal comment that should be updated to "brain assets" for consistency.

### Industry Knowledge

**Hardcoded seed strings over external resource files (Twelve-Factor App):** The Twelve-Factor App methodology recommends treating configuration and seed data as constants in code rather than as external resource files when the data is stable, small, and under developer control. For a 4-line Markdown table seed that never changes independently of the codebase, an inline constant is the correct vehicle. External file loading should be reserved for data that needs to be configurable without code changes.

**Convention-driven schema as single source of truth (BABOK, IEEE 29148):** Both BABOK and IEEE 29148 recommend that the canonical definition of any structured artifact reside in a single authoritative source. The `requests_register-convention.md` already fulfills this role for the register schema. Maintaining a separate template file that duplicates a subset of the convention violates the Single Source of Truth principle and creates synchronisation risk (the template's validation rules could drift from the convention's normative rules).

### Feasibility Notes

The request is technically straightforward. The `load_template` function is a thin wrapper around `Path.read_text`; replacing its only call with an inline string introduces no algorithmic complexity. All test changes are mechanical deletions of boilerplate. No external dependencies or API contracts are affected. Risk is assessed as negligible.

### Expert Observations

The scope is well-calibrated — neither too large nor too small. The templates folder is a genuine source of accidental complexity: it creates the impression that AIB uses a template-based seeding system, when in practice only one template ever existed and all other seeds were inline. Removing it reduces cognitive overhead for future maintainers.

One minor risk worth noting: the template currently contains a "Validation rules" block below the Markdown table. If any developer or tool has ever been relying on that block's presence in `requests_register.md` (e.g., as human-readable documentation inside the file), they will lose it after the hardcoded seed takes over on fresh initializations. Inspection of the workspace's live `requests_register.md` confirms the validation rules block is absent — it was never part of the seeded file in practice, so no regression occurs.

## Implementation Alternatives

### Alternative A — Hardcode register seed inline in `initialize.py` (chosen)

Replaces the `load_template` call with a Python string literal directly in `_seed_memory`. Matches the pattern used by all other seeds in the same function.

- **Trade-offs:** Minimal change surface; no new abstractions; consistent with existing code style. No downside.
- **Codebase impact:** `initialize.py` (1 line changed, 1 import removed); `common.py` (function deleted); `templates/` folder deleted; 3 test files updated; 1 test class deleted; `menu.py` text updated; `context.md` updated.

### Alternative B — Read seed from `requests_register-convention.md`

Parse the convention file at initialization time to extract the table skeleton and use it as the seed. Achieves the stated goal of "defined from the respective convention file".

- **Trade-offs:** Adds runtime file-read dependency on a convention file; requires a convention-parsing routine that does not currently exist; fragile against convention formatting changes. The convention file is not designed for machine extraction.
- **Codebase impact:** Larger than Alternative A; introduces a new parsing utility with no other consumer.
