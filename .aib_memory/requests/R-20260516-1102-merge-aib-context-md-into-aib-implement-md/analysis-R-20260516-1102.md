## Input Interpretation

The developer requests a consolidation of the AIB prompt surface. Currently AIB exposes three developer-facing prompts: `aib-analysis.md`, `aib-context.md`, and `aib-implement.md`. The `aib-context.md` prompt is responsible for generating or fully replacing `.aib_memory/context.md` by scanning the workspace and synthesising its content under a convention-governed structure. This prompt is already invoked from within `aib-implement.md` as step 7, making it a sub-workflow of the implement phase rather than a standalone concern for most use cases.

The request asks that the context-generation logic currently housed in `aib-context.md` be merged into `aib-implement.md` — verbatim, preserving all phases (Option A chosen) — under a clearly labelled `## Context Generation Sub-flow` section. After the merge, `aib-context.md` is deleted and AIB retains exactly two prompt files: `aib-analysis.md` and `aib-implement.md`.

In addition, the request identifies a **brownfield use case**: a workspace that already contains product artefacts but has not yet been initialised with AIB memory (i.e., `.aib_memory/context.md` is absent or empty). The fix is to detect an absent or empty `context.md` at the start of the implement workflow and automatically execute the context-generation sub-flow before applying any implementation changes.

The request also asks for a menu update: when `context.md` is absent or empty, the menu should display a tooltip directing the Developer to execute `aib-implement.md` to populate context. Workspace research confirms that `menu.py` already implements `_is_context_empty()` and already renders an extra guidance line when the function returns `True`; the current guidance text references `aib-context.md`. The change required is to update that string to reference `aib-implement.md`.

The end state is: two prompts (`aib-analysis.md`, `aib-implement.md`), full brownfield support embedded in the implement workflow, and menu guidance for the empty-context state referencing `aib-implement.md`.

## Research Results

### Workspace Findings

**Prompt coupling:** `aib-implement.md` step 7 currently reads `Execute .aib_brain\prompts\aib-context.md`, making the implement prompt a direct orchestrator of the context prompt. This coupling already treats `aib-context.md` as a sub-workflow; the merge formalises this by inlining the sub-flow.

**aib-context.md structure:** The prompt is organised in six phases — Phase 1 (Preflight), Phase 2 (Supplementary read / workspace sources), Reverse-Engineering Evidence Collection, Phase 4 (Synthesis), Phase 5 (Write output), and Phase 6 (Post-write Validation) — plus a context-window management section and a safety block. Total content: approximately 200 lines of detailed instruction. All phases must be copied verbatim under Option A.

**`aib-implement.md` current step 7:** A single-line reference: `Execute .aib_brain\prompts\aib-context.md`. This is the integration point to be replaced by an inline section reference.

**Menu existing empty-context handling:** `menu.py` already defines `_is_context_empty(workspace: Path) -> bool` and uses it in `render_menu` to print an additional guidance line:

```
Context file is empty — run context generation first: Execute `.aib_brain/prompts/aib-context.md`
```

This line is rendered orthogonally to the `_GUIDANCE_MESSAGES` state machine (as a separate conditional, not as a state key). The change required is to update the text of this line to reference `aib-implement.md`. A new key in `_GUIDANCE_MESSAGES` must NOT be added — the existing test `TestGuidanceMessagesKeys` asserts exactly 7 keys and would fail.

**Test `test_context_empty_line_shown_when_context_absent`:** In `tests/test_menu.py`, `TestRenderMenuGuidance.test_context_empty_line_shown_when_context_absent` asserts `"aib-context.md" in output`. After updating `render_menu`, this assertion must be updated to `"aib-implement.md" in output`.

**No existing tests for `_GUIDANCE_MESSAGES` key count below 7 or above 7:** The test is `assert set(_menu._GUIDANCE_MESSAGES.keys()) == expected` where `expected` is the exact set of 7 keys. Adding or removing any key breaks this test.

### Industry Knowledge

**Monolithic vs. modular prompt design:** LLM-facing prompt files are most maintainable when each file has a single clear responsibility. However, sub-workflows that are always invoked together with a parent can be inlined to reduce the number of artefacts a developer must navigate. The OpenAI and Anthropic prompt-engineering guidance supports reducing external file references mid-execution — each reference is a potential failure point. Applicability: high. Merging the context sub-flow removes one runtime external reference.

**Brownfield initialisation in automation frameworks:** Tools like Cookiecutter, Yeoman, and AI coding assistants implement a "bootstrap on first run" pattern — detect missing configuration, auto-generate it, then proceed. This is preferable to halting or producing degraded output. Applicability: high. The requested behaviour (auto-generate context if absent) follows this standard approach exactly.

**Prompt size and cognitive load:** Longer prompt files are harder to navigate when context windows are limited. When embedding a large sub-flow, a clearly labelled section boundary (e.g., `## Context Generation Sub-flow`) mitigates this by providing a navigable anchor. Applicability: medium. The merged file will be longer, but section labelling reduces the navigability penalty.

### Feasibility Notes

All changes are to Markdown prompt files and one Python menu script. No third-party dependencies are introduced. The existing test for `_is_context_empty` and `render_menu` empty-context rendering confirms the feature is already in place; only the referenced prompt filename changes.

**Risk:** aib-context.md content loss during inline merge — mitigated by verbatim copy requirement (Option A). Full file must be read before editing `aib-implement.md`.

**Risk:** `_GUIDANCE_MESSAGES` key count violation — avoided by not adding a new state key; updating only the string inside the existing `_is_context_empty` conditional in `render_menu`.

### Expert Observations

The request is well-scoped and low-risk. The main implementation risk was the menu change: the developer's description ("add a tooltip") might have been interpreted as adding a new `_GUIDANCE_MESSAGES` key, which would break an existing test. Workspace research resolved this — the correct approach is to update the existing guidance string in `render_menu`. No architectural changes are needed; the feature already exists, only its target reference changes.

## Implementation Alternatives

### Alternative A: Full Verbatim Inline (CHOSEN)

Copy all phases from `aib-context.md` verbatim into `aib-implement.md` under a clearly labelled `## Context Generation Sub-flow` section.

- **Trade-offs:** Preserves 100% of original behaviour with zero regression risk. Increases `aib-implement.md` size by approximately 200 lines. The section is self-contained and unambiguous.
- **Codebase impact:** Delete `aib-context.md`; append the sub-flow section to `aib-implement.md`; update step 7 to reference the inline section; add brownfield check at step 5b.

### Alternative B: Condensed Inline

Write a condensed version of the context-generation logic inside `aib-implement.md` capturing the key behavioural steps without reproducing every sub-rule verbatim.

- **Trade-offs:** Smaller file; more readable at a glance. Risk of behavioural regression if nuances from the original phases are omitted.
- **Codebase impact:** Same as Alternative A but with authored content rather than a verbatim copy.

### Alternative C: Internal Section Reference

Keep context generation as a short enumeration of high-level steps in `aib-implement.md`, relying on the AI agent's background knowledge.

- **Trade-offs:** Minimal file change. High regression risk for deterministic file-first frameworks. Not viable.

## Request Context

### Background

The AIB framework currently exposes three developer-facing prompts: `aib-analysis.md`, `aib-context.md`, and `aib-implement.md`. The `aib-context.md` prompt generates or replaces `.aib_memory/context.md` by scanning the workspace and synthesising product knowledge. It is already invoked as step 7 of `aib-implement.md`, meaning it is effectively a sub-routine of the implement workflow. The three-prompt surface creates unnecessary cognitive load, and the internal invocation of `aib-context.md` from within `aib-implement.md` creates an external file dependency during runtime.

### Scope

- `.aib_brain/prompts/aib-context.md` — merge content verbatim into `aib-implement.md`; delete the file.
- `.aib_brain/prompts/aib-implement.md` — embed the context-generation sub-flow verbatim; add brownfield detection step (5b); update step 7 reference.
- `.aib_brain/tools/menu.py` — update the existing empty-context guidance string in `render_menu` to reference `aib-implement.md` instead of `aib-context.md`.
- `tests/test_menu.py` — update `test_context_empty_line_shown_when_context_absent` assertion to match the new guidance string.
- `.aib_memory/context.md` — update domain knowledge and glossary to reflect the two-prompt architecture and brownfield capability.

### Out of scope

- `.aib_brain/prompts/aib-analysis.md` — no changes.
- All other tool scripts in `.aib_brain/tools/` — no changes required.
- Convention files — no changes.
- CI workflows and release bookkeeping scripts — not affected.
- Adding new keys to `_GUIDANCE_MESSAGES` — explicitly excluded to avoid breaking the existing key-count test.
