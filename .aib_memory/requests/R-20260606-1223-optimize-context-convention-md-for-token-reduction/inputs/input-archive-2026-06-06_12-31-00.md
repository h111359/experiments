## Input

To compress the resulting `context.md` file and save tokens without losing any meaning, you need to update `context-convention.md` to prioritize **information density** over human readability and rigid formatting. 

Here are the specific changes you should make to your `context-convention.md` file:

### 1. Eliminate Empty Subsections and Stub Notices
Currently, the convention forces the AI to output all 22 headings and write dummy statements (e.g., `AN-I-8f3b2a1c: No analytics requirements...`) even when there is no data. This wastes hundreds of tokens.
*   **Change:** In the **Section 2 — Statements** and **Non-Applicable Subsection Format** sections, change the rules to: *"Subsections that have no statements MUST be entirely omitted. Do not output empty headings. Do not output informational stubs explaining missing data."*

### 2. Simplify the Statement ID format (Drop the Hash)
Random hex strings (`8a86981d`) are terrible for tokenization (they often cost 3-5 tokens each). Since the edit scripts can just hash the text dynamically or use line matching, you don't need the hash in the text itself.
*   **Change:** In **Atomic Statement Format**, remove the `<HASH>` requirement.
*   **Old:** `- FN-R-8a86981d: The system must download...`
*   **New:** `- FN-R: The system must download...` *(or use a simple sequential number like `FN-R-1`)*

### 3. Mandate "Telegraphic" Language
Grammatically complete sentences use "stop words" (the, is, a, must be) that consume tokens without adding semantic value.
*   **Change:** In **Atomic Statement Format**, replace *"MUST be a single grammatically complete sentence"* with: *"MUST use concise, telegraphic phrasing. Omit unnecessary articles (a, an, the) and helper verbs. Prioritize density."*
*   **Old:** `- FN-R: The user interface must display a loading spinner when the data is fetching.`
*   **New:** `- FN-R: UI displays loading spinner during data fetch.`

### 4. Condense the File Inventory (Section 3)
Listing the full path for every single file on a new line duplicates directory names massively (e.g., `src/app/models/user.py`, `src/app/models/auth.py`). 
*   **Change:** In **Section 3 — Workspace File Inventory**, switch the format from flat bullet points to a YAML-style or indented tree structure. Also, make descriptions optional for self-explanatory file names.
*   **Old:**
    `- \`src/app/\` - Core application folder.`
    `- \`src/app/user.py\` - Handles user data.`
    `- \`src/app/auth.py\` - Handles login.`
*   **New:**
    ```text
    src/app/: Core application folder
      user.py: Handles user data
      auth.py: Handles login
    ```

Here are a few more advanced optimizations you can make to `context-convention.md` to squeeze out even more tokens:

### 5. Remove Redundant Area Prefixes
Currently, the convention forces you to repeat the Area Code (`FN`, `UI`, `DB`) on every single line, even though it's already under that heading. 
*   **Change:** Tell the AI to drop the Area Code from the statement ID since it is implied by the heading.
*   **Old:** 
    ```markdown
    ### FN — Functional
    - FN-R: System handles login.
    - FN-I: Login uses OAuth.
    ```
*   **New:**
    ```markdown
    ### FN
    - R: System handles login.
    - I: Login uses OAuth.
    ```

### 6. Remove Boilerplate Wrapper Headings
The convention mandates structural headings like `# Application Context` and `## Section 2 — Statements`. LLMs do not need these human-readable section wrappers to understand the file.
*   **Change:** Delete the requirement for `## Section 2 — Statements` and `## Section 3 — Workspace File Inventory`. Just use top-level headings for the areas (e.g., `# FN`, `# DB`, `# Files`). 

### 7. Add an "Aggressive Pruning" Rule
Context files naturally grow over time as features are added, but humans rarely delete old context. 
*   **Change:** Add a strict rule in the convention: *"Aggressively delete historical context, past design iterations, and resolved open issues (O-type statements). Do not retain deprecated features. The context must strictly represent only the CURRENT state of the application."*

### 8. Inline/Group Tightly Coupled Details
The "Atomic Statement" rule often forces related facts onto separate lines, multiplying the bullets and ID prefixes.
*   **Change:** Allow grouping of highly cohesive properties into a single statement using inline lists or brackets.
*   **Old:**
    ```markdown
    - DB-R: User table has an email column.
    - DB-R: User table has a password_hash column.
    - DB-R: User table email column must be unique.
    ```
*   **New:**
    ```markdown
    - DB-R: User table columns: email (unique), password_hash.
    ```

### 9. Minimized Header Metadata (Section 1)
Section 1 currently asks for `Status`, `Version`, and `Last Updated`. remove them entirely.  If versioning is needed, rely on Git rather than inline text.


### 10. Strip Unnecessary Markdown Formatting
Humans use bolding `**text**`, italics `_text_`, and inline code ticks `` `code` `` to make things readable. LLMs do not need these visual cues; they understand the text semantically.
*   **Change:** Add a rule: *"Do not use bolding, italics, or inline code formatting (backticks). Output plain text only."* Every pair of backticks or asterisks costs tokens.

### 11. Implement an Acronym/Alias Dictionary
If your app has long names like "User Authentication Microservice" or "Payment Gateway Webhook", repeating them across dozens of statements drains tokens.
*   **Change:** Add a rule to define short aliases at the top of the file and use them exclusively. 
    *   *Example at top:* `Aliases: UAM=User Auth Microservice, PGW=Payment Gateway Webhook`
    *   *Statement:* `- R: UAM validates token before routing to PGW.`


### 12. Cap File Inventory Depth
If your project gets large, the File Inventory will become massive, listing hundreds of minor helper files.
*   **Change:** Add a rule: *"Only list directories and core architectural files. Ignore tests, config files, assets, and minor utilities in the inventory unless they contain critical business logic."* 

### 13. Drop the "O" (Open/Unresolved) Statement Type
The convention defines `O` for unresolved questions or known issues. 
*   **Change:** Stop storing open questions in the permanent context. Keep the context file strictly as a "Source of Truth" for what is currently built. Open questions should live in issues, task lists, or the prompt itself (e.g., `input.md`), not taking up space in the core memory bank.
*   

### 14. Shift to Pseudo-Code / Logic Notation
LLMs are heavily trained on code and understand logic operators perfectly. Natural language requires many filler words to explain conditions. 
*   **Change:** Instruct the AI to use pseudo-code, arrows, and logic operators (`->`, `&&`, `||`, `==`, `!=`) instead of sentences for logic and flow.
*   **Old:** `- R: If the user does not have the admin role, the system must return a 403 Forbidden error.` (21 tokens)
*   **New:** `- R: role!=admin -> HTTP 403.` (7 tokens)

### 15. Standardize Developer Abbreviations
Force the AI to use standard, universally understood developer shorthand instead of full words.
*   **Change:** Add a rule: *"Always use standard abbreviations: req (request), res (response), auth (authentication), db (database), env (environment), config (configuration), err (error), msg (message), btn (button)."*
*   **Old:** `- R: The authentication response message contains the configuration.`
*   **New:** `- R: auth res msg contains config.`

### 16. Rule of "No Negative Space" or Standard Defaults
Often, context files explicitly state things that are industry standard or explicitly state what the system *doesn't* do.
*   **Change:** Add a rule: *"Do not state standard defaults (e.g., 'API returns JSON', 'Uses REST'). Do not document negative space (e.g., 'No caching is implemented') unless it overrides a standard expectation. Only document custom business logic and deviations from standard practices."*

### 17. Condense Routing / Endpoints into Tables/Lists
If your context file describes API endpoints or UI routes, do not use atomic statements. Use a densely packed route map.
*   **Change:** Mandate that APIs and Routes must be listed in a single dense block.
*   **Old:**
    ```markdown
    - R: The system exposes a GET /users endpoint.
    - R: The GET /users endpoint requires auth.
    - R: The system exposes a POST /users endpoint to create users.
    ```
*   **New:**
    ```text
    Routes:
      GET /users (auth)
      POST /users (create)
    ```
