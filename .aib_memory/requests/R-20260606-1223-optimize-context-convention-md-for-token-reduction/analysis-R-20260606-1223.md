# Analysis: R-20260606-1223

**Request ID:** R-20260606-1223
**Request title:** Optimize context-convention.md for token reduction

---

## Overview

### Background

The developer has identified that the current `context-convention.md` produces overly verbose `context.md` files. Each analysis or implement prompt run loads `context.md` into the model context window; a token-heavy file increases inference cost and reduces the available context budget for task-specific content. The input proposes 17 targeted changes to the convention, each eliminating a source of redundant tokens: stub notices for empty sections, 8-character hex hashes in statement IDs, grammatically complete sentence requirements, flat file inventory format, redundant area prefixes repeated under their own heading, boilerplate wrapper headings, historical context accumulation, markdown formatting characters, and more.

### Scope

- `context-convention.md` — primary deliverable; all 17 changes apply to this file.
- `verify-context.py` — must be updated to validate the new format; its hardcoded regex and subsection list become invalid under the new convention.
- `hash-text.py` — becomes unused if statement IDs no longer require hashes; file may be deprecated or retained as an optional utility.
- `edit-context.py` — must be updated to address statements by text-matching rather than hash-based ID lookup.
- `tests/test_context_formatting_rules.py` — assertions reference old format strings (`<AREA>-<TYPE>-<HASH>`, section heading literals); must be updated.
- `tests/test_verify_context.py` — `VALID_CONTEXT` fixture uses old statement format; must be updated.
- `context.md` — must be regenerated after the convention is updated to comply with the new rules.

### Out of scope

- `aib-refresh-context.md` — the prompt already delegates format rules to `context-convention.md`; no changes anticipated unless a specific format directive in the prompt contradicts the new convention.
- All other `.aib_brain/` convention files — not referenced by items 1–17.
- Any product source code, CI workflows, or non-AIB-memory artifacts.
- The 17 changes do not introduce new areas or statement types beyond those already defined.

## Input Interpretation

### User Original Inputs

The developer submitted 17 numbered changes to `context-convention.md` grouped under the theme "compress the resulting context.md file and save tokens without losing any meaning." The changes address:

1. Eliminate empty subsections and stub notices: omit headings with no statements entirely.
2. Drop the `<HASH>` from statement IDs (e.g., `FN-R:` or `FN-R-1:` instead of `FN-R-8a86981d:`).
3. Mandate telegraphic language: omit articles and helper verbs; prioritize density.
4. Condense file inventory to indented tree format rather than flat bullets.
5. Drop the redundant area prefix from statement IDs within their area heading (result: `- R: text` not `- FN-R: text`).
6. Remove boilerplate `## Section 2 — Statements` and `## Section 3 — Workspace File Inventory` wrapper headings; use `# FN` / `# Files` style headings for areas instead.
7. Add an "aggressive pruning" rule: delete historical context, past design iterations, and resolved open questions; retain only current state.
8. Allow grouping tightly coupled facts into a single statement using inline lists.
9. Remove Section 1 header metadata (Status, Version, Last Updated) and rely on Git for versioning.
10. Strip markdown formatting: no bold, italic, or backtick inline code; plain text only.
11. Implement an Aliases section at the top of the file for long repeated names.
12. Cap file inventory depth: list only directories and core architectural files.
13. Drop the "O (Open/Unresolved)" statement type.
14. Use pseudo-code and logic operators (`->`, `&&`, `||`, `==`, `!=`) for logic and flow statements.
15. Standardize developer abbreviations (req, res, auth, db, env, config, err, msg, btn).
16. Add a "no negative space" rule: do not state standard defaults or what the system does not do unless it overrides a standard expectation.
17. Condense routing/endpoint descriptions into a dense inline block format.

### AI Interpretation

The request requires a comprehensive rewrite of `context-convention.md` to shift the design goal from human readability to AI-consumption token efficiency. The deliverable is an updated convention that mandates a denser, more compressed output format for `context.md`. The new format is characterized by:

- Area sections with simplified headings (area code only, no numbering or long labels), promoted to H2 level.
- Statement IDs reduced to `<TYPE>: text` or optionally `<TYPE>-<seq>: text`; hash and area prefix removed.
- Empty area sections omitted entirely rather than represented by stub notices.
- Telegraphic prose: no articles, no helper verbs, no grammar padding.
- No inline markdown formatting characters (no backticks, no bold, no italic).
- File inventory as indented tree rather than flat bullets, capped to directories and key files.
- An optional top-of-file Aliases dictionary for repeated long names.
- Explicit pruning rule removing historical and deprecated content.
- Grouped statements allowed for tightly coupled facts.
- Pseudo-code notation for logic flows.
- Standard abbreviation enforcement.
- No-negative-space rule suppressing default-confirming statements.

This change also has cascading implications for the tool layer: `verify-context.py` currently hard-codes the old format (`AREA-TYPE-HASH` regex, 22 specific subsection headings, section heading literals), and `edit-context.py` currently uses the hash component for statement addressing. Both tools require updates to remain functional under the new format.

## Research Results

### Workspace Pattern-Scan Findings

**verify-context.py is tightly coupled to the old format.** The script defines `STATEMENT_PATTERN = re.compile(r"^- ([A-Z]{2})-([A-Z])-([0-9a-f]{8}): (.+)$")`, an explicit list of 22 subsection headings (`EXPECTED_SUBSECTIONS`), and checks for the literal strings `## 2. Statements` and `## 3. Workspace File Inventory`. Every one of these will become invalid after the convention update. The tool must be fully rewritten to match the new format.

**test_context_formatting_rules.py has five assertions that will break.** Specifically:
- `assert "<AREA>-<TYPE>-<HASH>" in content` — format string removed by items 2+5.
- `assert "## 2. Statements" in content` — wrapper heading removed by item 6.
- `assert "## 3. Workspace File Inventory" in content` — wrapper heading removed by item 6.
- `assert "MUST NOT exceed" in content and "H3" in content` — heading cap rule may survive but must be verified.
- Statement-type table assertions (`| N |`, `| R |`, etc.) — type table may be preserved or restructured.

**test_verify_context.py uses VALID_CONTEXT fixture with old format.** The baseline fixture `VALID_CONTEXT` contains `PO-I-a1b2c3d4:` formatted statements and the full 22-subsection heading set. The fixture and associated tests must be rewritten to use the new compressed format.

**hash-text.py becomes unused.** The tool exists solely to compute 8-char SHA-256 hashes for statement IDs. If statement IDs no longer contain hashes, this tool has no callers. It should be deprecated or removed.

**edit-context.py addressing scheme becomes invalid.** The tool currently targets statements by their `AREA-TYPE-HASH` key. Without the hash, it must switch to text-matching (e.g., matching by the statement text content) or sequential number addressing if `TYPE-N` sequential IDs are adopted. The input explicitly notes: "the edit scripts can just hash the text dynamically or use line matching."

**No "O" statement type exists in the current convention.** The type table defines N, R, C, E, L, U, A, D, I — nine types. There is no "O" type. Item 13 references a non-existent type. No deletion is needed; the change is a no-op.

**Item 6 implies a heading hierarchy change.** The current convention mandates H1 for document title only. Items 6 proposes area headings like `# FN`. This would create multiple H1 headings, violating common Markdown conventions. The intent is most naturally interpreted as: remove the `## Section 2` and `## Section 3` wrappers, promote area headings from H3 (`### 2.N Area — AREA`) to H2 (`## AREA`), keeping H1 solely for the document title.

**Item 9 (Section 1 metadata) partially applies.** The current convention's Section 1 does not define Status, Version, or Last Updated fields. The only timestamp is in the auto-generated preamble comment (`> **Auto-generated** by aib-refresh-context.md on <timestamp>`). Item 9's spirit is to remove this inline metadata and rely on Git instead. Removing the preamble block is the correct interpretation.

**Item 10 (no backticks) conflicts with current formatting rule 8.** Rule 8 states: "File paths and code identifiers MUST be wrapped in backticks." Item 10 explicitly overrides this. The new convention must delete rule 8 and replace it with a prohibition on inline code formatting.

**Item 17 does not require Markdown tables.** The input example shows an indented block (`Routes:\n  GET /users (auth)`) not a `|col|` table. This is compatible with retaining the no-tables rule.

### Industry Knowledge

**Token compression research (prompt engineering literature, 2023–2025):** Multiple studies on compressed prompting (LLMLingua, PromptCompressor, Selective Context) demonstrate that semantic content can be preserved with 30–70% token reduction through removal of stop words, redundant labels, and grammatical filler. Telegraphic phrasing is the dominant method. Applicability: HIGH. Items 3, 7, 15, and 16 directly implement this approach. Trade-off: compressed text is harder for humans to audit without tooling assistance.

**Structured knowledge representation (AI agent memory design, 2024):** Research on "agent memory" formats (MemGPT, LangChain memory modules, custom KB formats) shows that flat key–value or labeled list formats are more token-efficient than nested Markdown prose. Section headers repeated at every level ("## Section 2 — Statements → ### 2.1 Project overview - PO → - PO-N-xxxxx:") triple-count the area code. Applicability: HIGH. Items 5, 6, and 2 directly address this redundancy. Trade-off: the area context must be inferred from the heading, which requires the model to maintain heading scope across many tokens.

**Unique ID design in content-addressed stores (software engineering literature):** Systems that use content-addressed IDs (e.g., SHA hashes) trade human readability for collision resistance. For files edited by AI agents, line-matching or sequential numbering are valid alternatives when the dataset is small (dozens to hundreds of entries). The SHA-256 hash was added in AIB for uniqueness guarantees; text-based matching is sufficient at this scale. Applicability: HIGH. Supports items 2 and 5. Trade-off: text-matching is brittle if statement text changes; sequential IDs require re-numbering on insert/delete.

### AI Agent Critique

- **context-convention.md** — The current convention is detailed and well-structured but designed for human readability. The 22 mandatory subsection headings generate boilerplate in every generated context.md file regardless of content. The 8-char hash adds 9–10 tokens per statement line (the dash, 8 hex chars, colon, space) across potentially hundreds of statements. Formatting requirements (backticks, bold, mandatory sentence grammar) add tokens with no semantic benefit for AI consumption.
- **context-convention.md, Completeness Principle** — The principle states "Statements MUST be detailed enough that the full workspace context can be reconstructed." Telegraphic statements still satisfy this if all facts are present; the principle is not violated by shortening. However, the Granularity Principle ("split complex statements") conflicts with Item 8 (allow grouping). These two principles will need to be reconciled explicitly in the updated convention.
- **context-convention.md, Uniqueness Invariant** — Without the hash, the convention must define an alternative uniqueness guarantee. Text-matching-based CRUD assumes statement text is stable; if text changes, the old entry cannot be found. The plan must describe how edit-context.py handles this edge case.
- **context-convention.md, Formatting Rules section** — Rule 6 ("Bold text MUST be used only for term definitions and critical callouts within Section 1") conflicts with Item 10 (strip all markdown formatting). If item 10 is applied globally, rule 6 must be deleted.
- **verify-context.py** — `EXPECTED_SUBSECTIONS` is a hardcoded list of all 22 headings. Even with the old convention, this list needed manual update when areas were renamed. After the format change, the entire validation approach must change to accommodate dynamic area headings.
- **test_context_formatting_rules.py** — Tests are convention-specific, not format-agnostic. They will require a rewrite rather than incremental modification. The test for `<AREA>-<TYPE>-<HASH>` format will need to check for the new `<TYPE>: text` or `<TYPE>-<num>: text` pattern.
- **input.md** — Item 13 references an "O (Open/Unresolved)" statement type that does not exist in the current convention. This may indicate the user is familiar with another convention that uses "O" for open issues. No action is needed, but this should be confirmed.
- **input.md** — Item 6 proposes `# FN` (H1) for area headings. Multiple H1 headings in a single Markdown document is non-standard. The intent should be interpreted as H2 (`## FN`) for area headings, preserving the single H1 document title `# Product Context`.
- **context.md** — The current file references statement IDs extensively (e.g., `{Related to: PO-N-4f8a2c01}`). After format change, all `{Related to: ...}` references will use the new ID scheme. A full context.md regeneration will resolve this; partial edit would introduce inconsistency.
- **Granularity vs. Grouping conflict** — Current convention mandates atomic statements ("one fact per statement"). Item 8 allows grouping tightly coupled facts. The updated convention must define when grouping is permitted to prevent misuse. A "closely coupled" definition is needed (e.g., "properties of a single named entity").

**DP-01 — Final statement ID format after applying items 2 and 5**

*Context:* Item 2 drops the hash from `AREA-TYPE-HASH` → `AREA-TYPE`. Item 5 drops the area prefix since it is implied by the heading → `TYPE`. Items are sequential; together they produce `- <TYPE>: text`.

*Alternatives:*

- **Alt-A: `<TYPE>: text` (no number)** — Shortest form. Duplicate TYPE entries in the same section are distinguished only by content. `edit-context.py` must match by text content.
- **Alt-B: `<TYPE>-<seq>: text` (sequential number)** — Adds a sequential counter per area section. Provides a stable short ID. Requires renumbering on insert; breaks in-file references if not updated atomically.

*Resolution:* `resolve-autonomously`

Rationale: The input shows `- R: System handles login.` (no number) as the target format for Item 5, with sequential numbers listed only as an optional alternative in Item 2 ("or use a simple sequential number like `FN-R-1`"). The final format with items 2+5 combined is Alt-A: `- <TYPE>: text`. The input explicitly states "the edit scripts can just hash the text dynamically or use line matching," confirming text-based addressing as the intended addressing mechanism. Alt-A is selected. Source: `input.md ## Input`, Items 2 and 5.

---

**DP-02 — Area heading level after removing wrapper headings (Item 6)**

*Context:* Item 6 removes `## Section 2 — Statements` and `## Section 3 — Workspace File Inventory`. Area subsections currently at H3 level (`### 2.1 Project overview - PO`) become the primary organizational units.

*Alternatives:*

- **Alt-A: Promote to H2 (`## PO`, `## CM`, ...`## Files`)** — Consistent with current H1-only document title. Standard Markdown hierarchy. Compatible with current heading depth cap of H3 for sub-areas.
- **Alt-B: Promote to H1 (`# FN`, `# Files`)** — Matches the literal example in Item 6. Creates multiple H1 headings in a single document, which is non-standard for Markdown and breaks most parsers' table-of-contents generation.

*Resolution:* `resolve-autonomously`

Rationale: Item 6 shows `# FN` as an example, but multiple H1 headings in a Markdown file is non-standard and would break the existing heading hierarchy rule (only `# Product Context` should be H1). The intent is clearly to simplify headings, and Alt-A (H2) achieves this while maintaining valid Markdown hierarchy. Source: `input.md ## Input`, Item 6; reasoning from current convention heading rules.

---

**DP-03 — Preamble metadata removal (Item 9)**

*Context:* Item 9 says "remove Status, Version, and Last Updated entirely." The current convention's Section 1 does not define these fields. The only timestamp metadata is the auto-generated preamble: `> **Auto-generated** by aib-refresh-context.md on <timestamp>.`

*Alternatives:*

- **Alt-A: Remove the auto-generated preamble block** — Eliminates the preamble entirely. Saves 3 lines (~25–30 tokens per generation). Version tracking via Git only.
- **Alt-B: Retain the preamble timestamp, document that Status/Version/Last Updated are not used** — Preserves audit trail without adding fields the user didn't ask for. No token saving.

*Resolution:* `resolve-autonomously`

Rationale: Item 9 explicitly states "rely on Git rather than inline text" for versioning. The preamble timestamp is the only inline metadata. Alt-A aligns with the stated intent. The preamble also uses backtick formatting (contradicted by Item 10) and bold formatting (also contradicted by Item 10). Removing it satisfies both Item 9 and Item 10. Source: `input.md ## Input`, Items 9 and 10.

**DP-04 — Backtick formatting in Section 3 file inventory (Item 10 vs. current rule 8)**

*Context:* Current formatting rule 8: "File paths and code identifiers MUST be wrapped in backticks." Item 10: "Do not use bolding, italics, or inline code formatting (backticks). Output plain text only."

*Alternatives:*

*Resolution:* `resolve-autonomously`

Rationale: Item 10 explicitly says "Output plain text only" with no exceptions. The developer's intent is to strip all formatting characters globally. Alt-A is selected. Source: `input.md ## Input`, Item 10.

**DP-07 — Granularity Principle vs. Grouping (Item 8)**
