## Input Interpretation

The Developer has identified a regression in the GitHub Actions CI workflow `.github/workflows/aib-semver-patch-bump-and-log.yml`. The workflow is failing on pull request #106 (branch `103-clean-up---obsolete-code-concepts`) because that branch carries a manually-applied SemVer marker of `v1.3.0`, while the base branch (`main`) is at `v1.2.22`.

The root cause is in `scripts/release_bookkeeping.py`: the existing mismatch-check logic only recognises the head marker as valid when it equals either the base marker (`v1.2.22`) or a PATCH bump thereof (`v1.2.23`). Any MINOR or MAJOR manual bump on the PR branch triggers the error `SemVer marker state mismatch between base and PR branch`.

The required fix must update the version-comparison logic to accept head markers that represent any version strictly greater than the base version, treating them as pre-applied manual bumps and using the branch marker directly as the target version for log and archive creation.

## Research Results

### Workspace Findings

**Confirmed root cause — CI log evidence:**
The GitHub Actions log for run `25960596429` contains the literal error:
```
ERROR: SemVer marker state mismatch between base and PR branch. Base marker is 'v1.2.22', but PR branch marker is 'v1.3.0'. Rebase the PR branch onto the latest base branch and rerun.
```
The failure occurs at the `Run release bookkeeping` step. All subsequent non-`always()` steps are skipped.

**Affected code path in `scripts/release_bookkeeping.py`:**
Lines responsible for the failure (pre-fix):
```python
target_version = base_version.bump_patch()
target_marker = target_version.to_marker()
if head_marker not in (base_marker, target_marker):
    raise ReleaseBookkeepingError(...)
```
The guard assumes only a PATCH bump is valid. No comparison infrastructure existed on `Version` — the class was `@dataclass(frozen=True)` without `order=True`, so Python tuple-based ordering was unavailable.

**Version class structure:**
`Version` is a frozen dataclass with `(major: int, minor: int, patch: int)` declared in that order. Adding `order=True` produces lexicographically correct comparison via `(major, minor, patch)` tuples — semantically correct for SemVer where major takes the highest precedence.

**Test coverage gap:**
The 10 pre-existing integration tests for `release_bookkeeping.py` cover only the PATCH-bump and idempotent-rerun scenarios; no tests covered MINOR/MAJOR pre-bumped markers or backwards-marker detection.

### Industry Knowledge

**SemVer 2.0.0 specification (semver.org):** Version precedence is determined by comparing major, minor, and patch numerically from left to right. A pipeline that only validates PATCH increments is non-compliant with SemVer's intended usage of MINOR and MAJOR bumps for breaking changes and new features. The correct gate is: "head version ≥ base version" — with auto-PATCH applied only when they are equal.

**Applicability:** The current workspace's CI model corresponds directly to this specification. The fix aligns the script with SemVer's precedence rules by using ordered comparison instead of membership testing against a hard-coded two-element set.

**Python dataclass `order=True` (PEP 557, Python 3.7+):** When `order=True` is applied, Python auto-generates `__lt__`, `__le__`, `__gt__`, and `__ge__` based on the order of field declaration. For `(major, minor, patch)` declared in that order, this produces semantically correct SemVer ordering (tuple lexicographic = SemVer numeric precedence).

**Applicability:** Zero implementation overhead; single keyword addition to the existing class decorator is sufficient. No external library is required.

### Feasibility Notes

The fix is a small, self-contained change to a single class decorator and a single logic block in one Python file, plus a display-name update in the workflow YAML. All existing tests continue to pass after the change. Three new tests explicitly cover the MINOR, MAJOR, and backwards-marker scenarios. Feasibility risk: negligible.

### Expert Observations

The original design was intentionally constrained to PATCH-only bumps (the workflow was named "PATCH bump"). This was appropriate for a product in its early lifecycle, but becomes a blocker the first time a MINOR or MAJOR increment is needed. The fix is minimal and reversible. No refactoring is introduced — only the minimum change to unblock the failing PR.

## Implementation Alternatives

### Alternatives

1. **Version-comparison branching (chosen):** Add `order=True` to `Version`, replace the membership test with `<`/`==`/`>` comparisons. When `head > base`, use head as target directly. Simple, self-documenting, and idiomatic Python.

2. **Workflow-level conditional skip:** Add a bash pre-check in the workflow YAML to detect a pre-bumped marker and skip or short-circuit the Python script. Avoids touching `release_bookkeeping.py` but pushes version logic into shell, splits the logic across two files, and is harder to test.

3. **Accept any head marker unconditionally:** Remove the mismatch check entirely. Simpler but removes the safety net against backwards-marker regressions, which are a real risk on manual branches.

## Request Context

### Background

The CI workflow `aib-semver-patch-bump-and-log.yml` automates SemVer PATCH bumping, version log creation, and `.aib_brain/` archiving on every pull request targeting `main`. It was designed under the assumption that all version increments would be auto-applied PATCH bumps by CI. However, PR #106 includes a manually-applied MINOR bump to `v1.3.0` (from `v1.2.22`), which triggers a hard error in the version mismatch check added to prevent state inconsistencies.

### Scope

- `scripts/release_bookkeeping.py` — `Version` dataclass and mismatch check logic.
- `.github/workflows/aib-semver-patch-bump-and-log.yml` — `name:` display field.
- `tests/test_release_bookkeeping.py` — 3 new regression tests.

### Out of scope

- Renaming or relocating the workflow file.
- Supporting MINOR or MAJOR auto-bumping by CI (CI still only auto-applies PATCH; MINOR/MAJOR remain manual pre-branch tasks).
- Changes to `close-request.py`, `initialize.py`, or any other tool script.
