# Analysis — R-20260523-0728

## Overview

**Request ID:** R-20260523-0728
**Request title:** Restructure aib-analyze.md to remove redundancy

### Background

The `aib-analyze.md` prompt has accumulated structural redundancy across three areas: (1) output behavioral rules for the analysis document are split between section 6.1 and `analysis-convention.md`; (2) plan document structure and formatting rules are duplicated between section 6.2 and `plan-convention.md`; (3) sub-flows for the Auto-Request Creation Branch and Answer Application Sub-flow are separated into section 7, requiring the reader to jump between sections 5 and 7 to understand the linear execution. Additionally, GC-03 ("Q-blocks in first cycle only") encodes a behavioral constraint that no longer reflects actual prompt behavior.

### Scope

- Remove GC-03 from `aib-analyze.md` and remove all supporting statements that encode the same rule in sections 6.3.3 and 7.2.
- Merge section 6.1 (Analysis Document output rules) into step 5.5 (Generate Analysis), replacing duplicated write-behavior rules with a reference to `analysis-convention.md` and retaining only prompt-process-specific rules not covered by the convention.
- Merge section 6.2 (plan.md Updates) into step 5.9 (Plan Generation), replacing duplicated WBS schema and formatting rules with a reference to `plan-convention.md` and retaining only process and behavioral rules that the convention does not define.
- Restructure section 6 to contain Q-block specification only (previously section 6.3), promoting sub-sections 6.3.1 → 6.1, 6.3.2 → 6.2, 6.3.3 → 6.3.
- Inline the Auto-Request Creation Branch (section 7.1) into step 5.1 and the Answer Application Sub-flow (section 7.2) into step 5.4; remove section 7 entirely (including 7.3 Re-run Behaviour Summary).
- Update all internal cross-references within `aib-analyze.md` that point to sections 6.1, 6.2, 6.3, 7.1, 7.2, and 7.3 after the restructuring.
- Update `analysis-convention.md` cross-reference to `aib-analyze.md section 6.3` to reflect the new section number after renumbering.
- Update `tests/test_analysis_prompt_structure.py` — specifically `TestPlanSchemaFieldsInAnalysisPrompt.test_outputs_field_present` which checks for `#### Outputs` in `aib-analyze.md`; after WBS schema is removed from `aib-analyze.md`, this assertion must be removed from that class and retained only in `TestPlanSchemaFieldsInPlanConvention`.
- Update `.aib_memory/context.md` to reflect the new prompt structure.

### Out of scope

- Changes to `aib-implement.md`, `aib-refresh-context.md`, or any other prompt file.
- Changes to `plan-convention.md` content beyond confirming it already holds the structural rules being de-duplicated from `aib-analyze.md` (no new content needed).
- Changes to `analysis-convention.md` beyond the cross-reference update.
- Addition of new behavioral rules not implied by the refactoring.
- Changes to the test suite beyond removing/updating the `#### Outputs` assertion in `TestPlanSchemaFieldsInAnalysisPrompt`.

## Input Interpretation

The developer requests a structural refactoring of `.aib_brain/prompts/aib-analyze.md` to eliminate content redundancy and clarify separation of concerns between the process prompt and the convention files. The request comprises five coordinated changes:

1. **Remove GC-03**: The constraint "Q-blocks are generated only when this is the first analysis run for the active request" is behaviorally inaccurate and must be removed from the global constraints section along with all statements in the prompt body that express the same rule.

5. **Merge section 7 into section 5**: The Auto-Request Creation Branch (7.1) and Answer Application Sub-flow (7.2) must be inlined at the precise execution points in section 5 where they are triggered (steps 5.1 and 5.4 respectively). The Re-run Behaviour Summary (7.3), explicitly marked non-authoritative, is removed. All "see section 7.x" cross-references become self-contained in section 5.

### Industry Knowledge

1. **Don't Repeat Yourself (DRY) principle** (Hunt & Thomas, *The Pragmatic Programmer*, 1999; repeated in most software engineering curricula): Every piece of knowledge must have a single, unambiguous, authoritative representation. The current duplication of WBS schema between the prompt and the convention file creates maintenance risk — future schema changes require updating two files. The refactoring directly applies DRY by making `plan-convention.md` the sole definition of the schema. *Applicability: high — directly addressed by this request.*

2. **Separation of Concerns** (Dijkstra, 1974; formalized in structured design): Behavioral specifications (when to act, how to sequence steps) and structural specifications (what a document looks like) should reside in separate artifacts with clean interfaces. Mixing process steps and output format specs in a single section (`## 6. Output Specifications`) conflates concerns that are more legibly expressed in their respective locations (process in section 5 steps, format in convention files). *Applicability: high — the core motivation for merging sections 6.1 and 6.2 into the process steps.*

3. **Inline sub-procedure documentation** (DITA concept-task-reference model, IBM; Microsoft Writing Style Guide "procedures" section): Technical writing standards recommend inlining sub-procedures at their invocation points rather than placing them in appendix-like sections. A reader following the step-by-step flow in section 5 should not need to jump to section 7 to execute the branch. Inlining reduces cognitive load, eliminates forward-reference errors, and makes the flow auditable in sequence. *Applicability: high — directly motivates the merging of section 7 into section 5.*
