## Input Interpretation

The developer observes that requirements statements and other bulleted/list-based entries in `.aib_memory/context.md` frequently span multiple sentences or clauses, producing dense entries that are difficult to scan and read. The developer requests that an instruction be added to restrict each bullet or list entry to a maximum of one to two sentences; any entry requiring more text must be split into multiple separate entries rather than combined into a single long item.

Under the AIB framework, `.aib_memory/context.md` is an auto-generated artifact produced entirely by the `aib-context.md` prompt execution. Its content is governed by `.aib_brain/conventions/context-convention.md`, which the prompt reads as its authoritative formatting and structure source. A persistent, enforceable length restriction must therefore be expressed (a) as a normative rule in `context-convention.md`, and/or (b) as an explicit directive in the `aib-context.md` Phase 4 formatting checklist — the part of the prompt that explicitly enumerates formatting requirements the AI Agent must apply during every generation run.

## Research Results

### Workspace Findings

**Rule 16 is already present in `context-convention.md`.** The convention's Formatting Rules section contains:

> "16. Each bullet or list item MUST NOT exceed two sentences. If the content of an entry requires more than two sentences to express, it MUST be split into two or more separate bullet items rather than combined into a single long entry."

This rule was introduced as part of a prior request (consistent with the R-20260515-0710 "Consistency audit" timeline) and therefore no convention change is required for this request.

**The `aib-context.md` Phase 4 formatting checklist is incomplete.** The explicit "Formatting requirements (MUST enforce for every section)" block in Phase 4 of `aib-context.md` contains three explicit bullets:

- No Markdown tables (enforcing rule 11).
- One blank line between list items (enforcing rule 12).
- No bold as a section/field label (enforcing the bold-restriction rule).

It does **not** contain an explicit bullet for rule 16. The Phase 4 checklist is the AI Agent's highest-salience reminder list during generation — omitting rule 16 from it increases the risk that the rule is silently skipped despite being in the convention.

**The test suite covers only rules 11–13.** `tests/test_context_formatting_rules.py` tests for the presence of the no-tables rule, blank-line rule, and heading-depth cap in `context-convention.md`, and verifies their corresponding Phase 4 prompt text. Rules 14, 15, and 16 have no test coverage. The existing test structure (one test per rule per class) provides a clear pattern for extension.

**Current `context.md` entries violate rule 16.** Multiple requirement entries (for example FR-003, FR-004, FR-005) in the active `context.md` exceed two sentences per bullet. These were generated before rule 16 was introduced and will be corrected on the next `aib-context.md` execution once the prompt checklist is updated.

### Industry Knowledge

**Documentation readability research (Nielsen Norman Group; plain-language writing standards):** List items that exceed one to two sentences create cognitive load during scanning. Studies consistently show that readers process bulleted lists for key facts — not continuous prose — and that items longer than two sentences are re-read significantly more than shorter items. This finding directly validates the 1–2 sentence cap as the threshold most beneficial to human readers.

**Style guide alignment (Microsoft Writing Style Guide; GNU documentation conventions):** Both guides recommend limiting each list item to one idea and, where possible, one sentence. Two sentences is a well-established practical maximum that accommodates a qualifier or condition while preventing list items from becoming prose paragraphs. Applicability to this workspace is direct: the AIB framework is a developer-facing technical tool where rapid comprehension of requirement statements is a primary usability concern.

### Feasibility Notes

No technical uncertainty exists. Rule 16 is already in the convention; the implementation reduces to: (1) adding a single bullet to the `aib-context.md` Phase 4 checklist, and (2) extending the test file. Both changes are low-risk, reversible, and constrained to well-understood files. The only scope boundary to manage is that fixing *existing* long entries in `context.md` requires re-running `aib-context.md` — this is a separate prompt action and is outside implementation scope.

### Expert Observations

The scope is appropriately narrow. The convention rule is already in place; the work is making enforcement visible to the AI Agent and verifiable via tests. This is a well-scoped follow-through on prior work.

The most significant risk is that the developer continues to see the dense entries in the current `context.md` after implementation, which might create the impression that the rule is still not enforced. The documentation task in the plan explicitly calls for a `aib-context.md` regeneration run as the post-implementation step to close this gap.

No simplification opportunities beyond what is described. No maintainability concerns. The change follows the established pattern in the test file exactly.

## Implementation Alternatives

### Alternative A — Convention-only change (no action needed)

Rule 16 is already in `context-convention.md`. The `aib-context.md` prompt reads the convention at generation time and could in principle apply rule 16 without an explicit checklist item.

- **Trade-offs:** Simple; no files changed. However, the checklist's purpose is to surface critical rules explicitly for the AI Agent — omitting rule 16 from it silently lowers enforcement salience compared to the three explicitly listed rules.
- **Codebase impact:** Zero changes. Rule 16 remains unverified by tests.

### Alternative B — Add Phase 4 checklist bullet + test coverage (selected)

Add an explicit bullet for rule 16 to the `aib-context.md` Phase 4 formatting checklist, and add test coverage for rule 16 in `test_context_formatting_rules.py`.

- **Trade-offs:** Minimal change; consistent with established patterns; verifiable. Requires updating two files. Existing long entries in `context.md` are not immediately fixed.
- **Codebase impact:** `aib-context.md` (one bullet added to Phase 4), `test_context_formatting_rules.py` (two new test methods).

### Alternative C — Alternative B + immediate context.md remediation

All of Alternative B, plus regenerate `context.md` immediately by re-running `aib-context.md`.

- **Trade-offs:** Complete end-to-end fix; developer sees results immediately. Requires running `aib-context.md` as a separate prompt action during implementation.
- **Codebase impact:** `aib-context.md`, `test_context_formatting_rules.py`, and `context.md` are all updated.

## Request Context

### Background

The developer observed that `.aib_memory/context.md`, AIB's auto-generated workspace product-context document, contains requirement statements and other list entries that are multi-sentence, multi-clause prose blocks rather than concise single-topic bullets. This density makes the document difficult to scan quickly, defeating its purpose as a rapid-orientation reference for both human developers and the AI Automation Agent. The developer requests that an instruction be put in place to restrict each bullet or list entry to a maximum of 1–2 sentences.

### Scope

- `.aib_brain/prompts/aib-context.md` — add one explicit enforcement bullet to the Phase 4 "Formatting requirements" checklist for the 2-sentence limit (rule 16).
- `tests/test_context_formatting_rules.py` — add two new test methods: one verifying rule 16 is present in `context-convention.md`, one verifying the enforcement instruction is present in `aib-context.md` Phase 4.
- `logs/next_version_changes.md` — append a change bullet.
- `.aib_brain/README.md` — reviewed and updated if discrepancies with `context.md` are found.

### Out of scope

- Changes to `.aib_brain/conventions/context-convention.md` (rule 16 is already present).
- Direct editing of `.aib_memory/context.md` to remediate existing long entries (auto-generated; must be addressed via `aib-context.md` re-run).
- Changes to any other prompt, convention, or tool script.
- Coverage extension for rules 14 or 15 (adjacent but outside this request's stated goal).
