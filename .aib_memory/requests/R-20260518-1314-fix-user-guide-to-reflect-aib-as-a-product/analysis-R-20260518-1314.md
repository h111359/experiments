## Overview

**Request ID:** R-20260518-1314
**Request title:** Fix user guide to reflect AIB as a product

### Background

The `.aib_brain/user_guide.html` file is the self-contained, browser-viewable reference for AIB users. During a review, the developer identified that one of the prerequisites listed in Section 2 — "Git available on PATH (required for CI release bookkeeping)" — is accurate only within the AI_Builder development repository (where CI pipelines require Git to execute release bookkeeping scripts), but is not a valid user-facing prerequisite for AIB as a deployable product. Users who install AIB's `.aib_brain/` folder into their own workspace do not require Git on their PATH to use the AIB prompts and tools locally. The developer requested a comprehensive audit to identify and correct all similar errors where the user guide conflates the AI_Builder project's specific environment with the product's general requirements.

### Scope

- `.aib_brain/user_guide.html` — audit and correction of all content that describes the AI_Builder development project rather than AIB as a general product. Six discrete errors were identified and are addressed in the plan.

### Out of scope

- Changes to any file other than `.aib_brain/user_guide.html`, except for automated test additions and any required `context.md` documentation updates.
- Structural or layout changes to the user guide (accordion structure, CSS, JavaScript, section order).
- Adding new content sections not related to correcting the identified errors.

## Input Interpretation

The developer requests that `.aib_brain/user_guide.html` be audited and corrected to accurately represent AIB as a deployable product framework, distinct from the AI_Builder development repository in which AIB itself is built and maintained.

The specific defect identified is the listing of "Git available on PATH (required for CI release bookkeeping)" as a user-facing prerequisite. This statement is factually valid only for the AI_Builder project: the repository hosts CI workflows (`scripts/release_bookkeeping.py`) that require `git` to be available in the CI runner. However, when AIB is used as a product — that is, when a user or team installs `.aib_brain/` into their own repository — they do not need Git on their local PATH to invoke AIB prompts and tools. Git is a CI infrastructure concern, not a workstation developer prerequisite.

The developer's instruction to "revise for other similar errors and fix them all" extends the scope to a full audit of the user guide for any other content that carries AI_Builder project-specific assumptions rather than product-generic descriptions. This includes glossary entries, folder structure descriptions, and any annotation text that would mislead a user who has installed AIB into a workspace that is not the AI_Builder repository.

## Research Results

### Workspace Findings

**User guide content audit (`user_guide.html`):**

Six distinct product-vs-project conflation errors were found in `.aib_brain/user_guide.html`:

1. **Section 2 — Prerequisites (line 383):** `<li><strong>Git</strong> available on PATH (required for CI release bookkeeping).</li>`. Git is required only in the CI environment that runs `scripts/release_bookkeeping.py`; it is not needed by developers using AIB locally.

2. **Section 9 — Folder Structure (`.aib_brain/` tree, line 623):** `README.md ← This guide's source of truth`. The README.md is a quick-start overview that explicitly points to `user_guide.html` as the full guide (confirmed by reading `.aib_brain/README.md`). The annotation "This guide's source of truth" is misleading — it implies README.md is the authoritative reference, when `user_guide.html` is the complete user-facing document.

3. **Section 9 — Folder Structure (`.aib_brain/` tree):** `user_guide.html` is absent from the folder tree listing. The file exists in `.aib_brain/` (confirmed by `list_dir`) but is not shown.

4. **Section 9 — Folder Structure (`.aib_memory/` tree, line 631):** `logs/ ← Version logs and curated change bullets`. The `.aib_memory/logs/` directory actually contains tool execution and action logs (e.g., `aib-action-20260515-110601-close-request-py.log`, `aib-action-20260516-120329-close-request-py.log`). Version logs and curated change bullets live in the workspace-root `logs/` folder, not in `.aib_memory/logs/`.

5. **Section 11 — Glossary, CI entry (line 713):** `Continuous Integration: automated pipeline that runs on pull request events targeting <code>main</code>`. The branch name `main` is specific to the AI_Builder repository's configuration. As a product, AIB can be installed in repositories that target any branch (e.g., `master`, `develop`, `release`).

6. **Section 11 — Glossary, VCS entry:** `Version Control System (Git in this workspace)`. The phrase "in this workspace" is AI_Builder-specific. As a product, AIB is VCS-agnostic from the user's perspective; the VCS glossary entry should be product-neutral.

**Cross-reference check — context.md vs user_guide.html:**

- `context.md` Technical Constraints states: "`git` must be available on PATH in CI for release bookkeeping." This correctly scopes Git as a CI requirement. The user guide's prerequisite section contradicts this framing by presenting Git as a user-workstation requirement.
- `context.md` glossary entry for "VCS": "Version Control System (Git in this workspace)" — this is the same AI_Builder-specific phrasing found in the user guide. This entry in `context.md` is also project-specific, though correcting `context.md` is not the primary scope of this request.
- `context.md` glossary entry for "CI": "Continuous Integration: automated pipeline that runs on pull request events" — no branch specification in context.md; the user guide's `targeting main` addition is not present in context.md and is a user-guide-specific inaccuracy.

**`.aib_brain/README.md` content:** The README.md begins with "Open user_guide.html in a browser for the full interactive user guide." This confirms that README.md is a pointer and quick-start document, not "This guide's source of truth" for the user guide content.

### Industry Knowledge

**Finding 1 — Prerequisites scoping (Write the Docs community standard):**
Developer tool documentation should list only what users need to install or configure on their local workstations. CI/CD infrastructure requirements (runner environment variables, tokens, binary availability in CI) belong in a separate "CI Setup" or "Infrastructure" section, not in user prerequisites. Mixing CI requirements with user prerequisites creates a false dependency and may discourage or block adoption. Applicability: directly applicable — the Git prerequisite is a CI concern and should not appear in the user-facing Prerequisites section.

**Finding 2 — Neutral product language in documentation (OpenAPI / CNCF documentation patterns):**
Product documentation distributed for adoption in diverse environments should avoid environment-specific constants (branch names, server addresses, organisation names) in generic definitions unless those constants are configurable. Glossary terms in particular should define concepts, not document configuration specifics. Applicability: directly applicable to the CI glossary entry ("targeting `main`") and VCS glossary entry ("in this workspace").

**Finding 3 — Folder structure documentation accuracy (Diátaxis framework — Reference section principles):**
Reference documentation (such as folder structure tables and glossaries) must be accurate and up to date with the actual state of the product. Inaccurate reference entries undermine user trust and create debugging friction when users inspect their workspace and find a mismatch between the documentation and reality. Applicability: directly applicable to the `.aib_memory/logs/` description mismatch and the missing `user_guide.html` entry.
