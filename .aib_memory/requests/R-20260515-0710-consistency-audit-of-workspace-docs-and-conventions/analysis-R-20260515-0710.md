## Research Results

### Workspace Findings

**Finding A — analysis-convention.md defines 4 mandatory sections; context.md and the component map entry for aib-analysis.md describe 8:**

`analysis-convention.md` section 4 lists exactly four mandatory sections:
1. Executive Summary
2. Files Read During This Analysis Run
3. Research Results
4. Implementation Alternatives

`context.md` (FR-004, component map, and the aib-analysis.md description) consistently refers to **8 mandatory sections**: Executive Summary, Files Read During This Analysis Run, Research Results, Best Practices, External Benchmarking, Minimal Spikes and Experiments, Implementation Alternatives, AI Copilot Suggestions.

`aib-analysis.md` section 6.2 defines generation instructions for only the same 4 sections that the convention names. There is no generation guidance for Best Practices, External Benchmarking, Minimal Spikes and Experiments, or AI Copilot Suggestions in the current prompt. This makes `context.md` stale with respect to the actual prompt and convention.

Impact: an agent reading `context.md` as authoritative context (as prescribed by both prompts) may attempt to generate 8 sections, while the convention and prompt only require 4.

**Finding B — Q-block `*(recommended)*` marker: direct contradiction between context.md and aib-analysis.md:**

`context.md` (FR-007 and the component map description of `aib-analysis.md`) states: "Each Q-block includes a `> **Why this matters:**` impact line, mutually exclusive options with a `*(recommended)*` marker" and "a `*(recommended)*` marker on the preferred choice".

`aib-analysis.md` section 7.5.3 Q-block format explicitly states: "Do NOT include a `*(recommended)*` marker on any option — the developer chooses without AI steering."

These two sources are diametrically opposed. An agent using `context.md` as context for implementing changes to the prompt would introduce a `*(recommended)*` marker that the prompt explicitly forbids.

**Finding C — "Decision Points Catalog" vs unnamed "Decision Points table":**

`context.md` (FR-004 and component map) references `### Decision Points Catalog` as the name of a required subsection within `## Implementation Alternatives` of the analysis document. Neither `aib-analysis.md` nor `analysis-convention.md` name this table using a heading — both refer only to "a **Decision Points table**" inline without requiring a named heading.

Impact: an agent resolving via context.md would try to produce or validate a `### Decision Points Catalog` subsection heading that neither the prompt nor the convention requires.

**Finding D — `Category` column in Decision Points table: in context.md but absent from prompt and convention:**

`context.md` (FR-004): "The `Category` field MUST be one of `Architecture`, `Data Model`, `UX/UI`, `Integration`, `Performance`, `Testing`, or `Documentation`."

`aib-analysis.md` section 7 and `analysis-convention.md` section 4.4 both define the Decision Points table with three columns only: `Decision Fork`, `Tag`, `Rationale / Resolution`. No `Category` column is defined anywhere outside `context.md`.

Impact: an agent enforcing context.md's FR-004 would reject analysis documents that lack a `Category` column, even though the prompt and convention never ask for one.

**Finding E — `requests_register-convention.md` timestamp format: specification text vs worked examples are inconsistent:**

The convention's Required Table Schema says: "ISO local timestamp... format `YYYY-MM-DD HH:MM:SS ±HHMM` (24h, with seconds and timezone offset)."

The "Worked Example Rows" use the format `2026-03-07 13:42` (no seconds, no timezone offset). The actual register uses the full ISO format matching the specification text. The example contradicts the specification within the same file.

**Finding F — `aib-analysis.md` section 6.1 note on analysis generation when triggered from `aib-implement.md`:**

Section 6.1 states: "Always generated unless triggered from `aib-implement.md`." This literal reading implies the analysis document is *not* generated when `aib-analysis.md` is auto-triggered from `aib-implement.md`. However, without the analysis, the Plan in `request.md` would also not be generated, leaving implement with no execution scope.

The trigger guard in section 8 (Standard Flow Final Step) is correctly about suppressing `finalize-input.py` when triggered from implement. Section 6.1 appears to incorrectly extend that restriction to the analysis document itself.

**Finding G — "AI Copilot Suggestions" section restriction scope:**

`context.md` says the AI Copilot Suggestions section "is a reasoning-only artifact, MUST NOT be read or acted on by `implement`." This implies the rest of the analysis document *may* be read by implement.

`analysis-convention.md` section 1 and `aib-analysis.md` section 5 both state "The analysis document is NOT an implementation driver. `implement` MUST NOT read the analysis document" — applying the restriction to the entire document, not just one section.

**Finding H — context.md version string is stale:**

`context.md` Product Identity says: "Current version: **v1.2.8** (active request R-20260514-2159 in progress; next release pending)." Request R-20260514-2159 is Closed per `requests_register.md`. This is expected aging of a synthesized document, but it will confuse agents that treat context.md as current-state truth.

### Industry Knowledge

**I1 — Single Source of Truth Principle (documentation consistency):** Established software documentation practice (e.g., Diátaxis framework, DITA architecture) dictates that any behavioral requirement should have exactly one canonical source. When two documents describe the same behavior, one should be authoritative and the other should defer to or cite it explicitly. In this workspace, `context.md` is synthesized from workspace sources and is explicitly described as fully auto-replaced on each run — making it a derived artifact, not an authoritative source. The authoritative sources are the prompt files and convention files. `context.md`'s FR-004/FR-007 entries should reflect the prompts, not define competing requirements.

Applicability: the inconsistencies found (Findings A–D) are a direct consequence of `context.md` being generated in isolation from `.aib_brain/` (the `aib-context.md` non-goals explicitly exclude reading prompts). The pattern of defining FRs in `context.md` that may diverge from actual prompt behavior is a structural risk.

**I2 — Specification by example (worked examples in conventions):** Technical writing best practice (per IEEE 830 and Google Developer Documentation Style Guide) requires that examples within a specification document be consistent with the specification's normative text. Inconsistent examples in the same document are a known source of implementation errors. Finding E (timestamp format) is a classic example of this anti-pattern.

Applicability: `requests_register-convention.md` should have its worked examples updated to match the normative timestamp format.

### Feasibility Notes

This is a purely read-only audit producing a documentation artifact. No technical uncertainty affects the scope. All required files were readable. The only uncertainty was the missing explicit `### Scope` section in the original `input.md`, which was resolved by inferring scope from the audit instructions. This assumption is recorded below.

### Expert Observations

The scope is well-bounded. The most impactful issues are the direct contradictions between `context.md` and the actual prompts (Findings A, B, C, D). These are high-risk because both `aib-analysis.md` and `aib-implement.md` read `context.md` as part of their preflight — an AI agent can simultaneously read context.md (which says 8 sections, recommended markers, Decision Points Catalog) and then read the prompt (which says 4 sections, no recommended markers, no named heading). The result is ambiguity at execution time.

The simplest remediation for all four context.md findings is a single run of `aib-context.md` after the next time the prompts stabilize. The prompt itself correctly excludes `.aib_brain/` from reading, which is why context.md drifts — fixing this is an architectural decision that should be raised rather than silently corrected.

Finding F (misleading note in section 6.1 of aib-analysis.md) is a low-risk documentation error since in practice the auto-request creation branch always needs to generate the analysis. But it could confuse a future maintainer editing the prompt.
