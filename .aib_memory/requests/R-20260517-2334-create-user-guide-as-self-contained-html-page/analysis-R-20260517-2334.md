# Analysis: R-20260517-2334

## Overview

**Request ID:** R-20260517-2334

**Request title:** Create user guide as self-contained HTML page

### Background

The AIB framework currently ships with a `README.md` inside `.aib_brain/` that documents its structure and daily workflow. However, this README is a plain Markdown file — not easily navigable, not visually guided, and not suitable for sharing or browsing outside of a code editor. The developer has requested a richer, browser-friendly user guide delivered as a single self-contained HTML file (with all CSS and JavaScript embedded inline), which will serve as the primary human-facing documentation artifact for AIB end-users. Supplementary media assets (icons, images) may reside in an adjacent `.aib_brain/media/` folder if needed, though the developer preference is to keep the main file self-contained. Icons must be sourced from open-source, free-of-charge libraries or generated inline.

### Scope

- New file `.aib_brain/user_guide.html` — a single self-contained HTML page with all CSS and JavaScript embedded (no external CDN or network dependency at runtime).
- Optional new folder `.aib_brain/media/` — for any supplementary media assets if binary files are required (icons sourced externally and saved locally).
- Content to cover: AIB introduction, prerequisites, quick start (menu), daily flow, prompt invocations (analyze, implement, refresh-context), Q&A mechanism, `instructions.md` directives, workspace structure, concepts/glossary.
- Interactive navigation using JavaScript as implied by the request ("with css and JS inside").
- SVG icons sourced from an open-source MIT-licensed icon library (e.g., Heroicons, Feather Icons), embedded inline or saved to `.aib_brain/media/`.

### Out of scope

- Modifications to any existing `.aib_brain/` prompts, conventions, or tool scripts.
- Changes to `.aib_memory/` or any workspace source code, tests, or CI workflows.
- Multi-page HTML output (a single self-contained file is explicitly required).
- Backend or server-side rendering; the guide is a purely static artifact.
- Automated test coverage of the HTML content structure (the guide is a documentation artifact, not executable code under test).

### Industry Knowledge

- **Self-contained HTML documentation (MDN, W3C):** Embedding CSS and JS inline in a single HTML file is a well-established pattern for portable documentation (e.g., single-file dashboards, offline-first docs). The trade-off is increased file size, but for documentation of this scale (< 300 KB), this is negligible. The pattern is used widely in open-source projects (e.g., Swagger UI standalone, single-file testing harnesses). Applicability: High — exactly matches the request requirement.

- **Accessible navigation patterns (WCAG 2.1, WAI-ARIA):** Tabbed interfaces with ARIA roles (`role="tablist"`, `role="tab"`, `role="tabpanel"`) provide strong screen-reader support and keyboard navigability. Single-page scroll with scroll-spy is simpler but requires careful heading structure for accessibility. Accordion patterns (disclosure widgets with `aria-expanded`) are also accessible when implemented correctly. Applicability: Medium — the guide targets developers (less likely to have accessibility constraints) but following ARIA best practices is low-cost and adds robustness.

- **SVG icon embedding (CSS Tricks, Smashing Magazine):** Inline SVG icons provide styling control (color, size via CSS variables), eliminate HTTP requests, and work offline. Icon libraries like Heroicons (MIT), Feather Icons (MIT), and Tabler Icons (MIT) publish their icons as raw SVG files, making inline embedding straightforward. The preferred authoring pattern is to copy the `<svg>` markup directly into the HTML. Applicability: High — this directly addresses the "download free icons / generate yourself" requirement in the most self-contained way.

## Implementation Alternatives

### Decision Fork 1: Navigation Pattern

**Applies to:** The page-level UI structure task — the top-level organization of content within the HTML file.

**Why alternatives exist:** The developer requested "with CSS and JS inside" (implying interactive navigation) but did not specify the navigation paradigm. Three distinct patterns are viable, each with materially different UX, JS complexity, and accessibility characteristics.

**Alternative A: Tabbed interface**
- Description: Each major topic (Quick Start, Daily Flow, Prompts, Q&A, Concepts/Glossary) is a separate tab; clicking a tab hides all other panels and shows the selected one.
- Trade-offs: Clean section separation, low scroll fatigue, common in developer documentation; requires JS tab-switching logic; all content not visible without tab interaction; deep-linking to sections requires extra effort.
- Codebase impact: ~60–80 lines of JS for tab management + ARIA attributes; moderate CSS for active-tab styling.

**Alternative B: Single-page scroll with sticky side-navigation**
- Description: All content flows vertically; a fixed left or top navigation bar highlights the active section using scroll-spy (IntersectionObserver API).
- Trade-offs: All content visible in sequence (good for first-time readers); no JS-gating of content; scroll-spy is elegant but requires care with IntersectionObserver thresholds; can feel long for reference lookups.
- Codebase impact: ~40–60 lines of JS for scroll-spy; simpler CSS layout; natural document structure.

**Alternative C: Accordion sections**
- Description: Each major section is a collapsible panel (expand/collapse); all section headers visible at all times; clicking a header expands that section.
- Trade-offs: Compact overview of all sections; quick access to any section; common in FAQ/help formats; less suitable for sequential learning flow.
- Codebase impact: ~30–40 lines of JS for accordion toggle + ARIA; straightforward CSS for collapsed/expanded states.

### Decision Points

- **Tag:** `resolve-autonomously` (resolved via Q&A Answer Application Sub-flow)
