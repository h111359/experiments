## Input Interpretation

The AIB developer has reviewed the proposed solution from the prior analysis run for request R-20260519-1209 and determined it is insufficient. The proposed fix — adding a "Read-scope restriction" paragraph to section 4.8 step 1 and replacing the "active session context" phrase in section 6.2 — addresses the root cause at the prompt-instruction level but provides no structural guarantee that the generated output actually conforms to the convention. The developer identifies two requirements:

1. A broader review of the analysis generation logic to evaluate whether the proposed read-scope restriction is the correct approach or whether a fundamentally different mechanism is needed.

2. The addition of a post-generation verification step within `aib-analyze.md` that programmatically validates the output `analysis-<request_id>.md` conforms to `analysis-convention.md` after every run — catching duplication or structural violations regardless of their root cause.

The developer's intent is to establish a defense-in-depth strategy: the read-scope restriction prevents the most common source of duplication (full prior analysis in context window), while the verification step catches any instance of non-conformance regardless of origin.

## Research Results

### Workspace Findings

**Review of the analysis generation logic:**

The current `aib-analyze.md` prompt generates the analysis through the following flow on re-run after Q&A:

1. Section 4.8 (Answer Application Sub-flow) is entered when `## Questions` are present in `input.md`.

2. Step 1 reads the full `analysis-<request_id>.md` to extract `## Input Interpretation` for creating `plan.md` when absent (deferred-creation state).

3. Steps 2–4 apply Q-block answers, remove `## Questions`, and continue to the standard analysis flow.

4. Section 6.1–6.2 govern the output writing: full content replacement of the analysis file, with generation instructions for each mandatory section.

**The fundamental weakness in the proposed fix:**

The read-scope restriction approach (adding text-based constraints) is a "tell the AI not to do X" strategy. It is better placed (at the point of reading rather than the point of writing), but it shares the same fundamental limitation as the existing section 6.1 prohibition: it depends on the AI reliably self-constraining against content already loaded into its context window. Both are prompt-level instructions that cannot physically prevent the AI from retaining loaded content.

**What actually causes duplication:**

The duplication manifests in two patterns:

- Pattern A: Section headings appearing twice (the old heading + the new heading both present in the output file).

- Pattern B: Content from the prior analysis appearing within a newly generated section alongside fresh content.

Both patterns stem from the same mechanism: the AI uses the old analysis content as a "template" or "starting point" rather than generating from scratch, and fails to fully discard the prior text.

**Why verification is the correct structural defense:**

A post-generation verification step operates on the actual output file after it has been written. It can deterministically check:

- Each mandatory section heading appears exactly once.

- No section heading is duplicated.

- The section order matches the convention.

- The file does not contain tell-tale markers of copy-paste duplication (e.g., two `## Overview` headings, two `## Research Results` headings).

This check is independent of what was in the AI's context window. It catches the defect regardless of whether the cause was the old analysis being loaded, the AI hallucinating a second pass, or any other unforeseen trigger.

**Existing convention support for verification:**

`analysis-convention.md` section 4 defines the mandatory structure (5 sections in exact order). This provides a clear, deterministic specification against which the output can be verified. The verification needs only to:

1. Parse the output file for level-2 headings (`##`).

2. Assert the headings match the expected set exactly (allowing only the 5 mandatory headings plus their content).

3. Assert no heading appears more than once.

### Industry Knowledge

**Finding 1 — Output validation in AI agent workflows (LangChain Structured Output, Guardrails AI, LMQL):**

AI agent frameworks increasingly implement post-generation validation as a standard practice. LangChain's "Structured Output" pattern, Guardrails AI's "Rail" specifications, and LMQL's type constraints all operate on the same principle: after the AI produces output, a deterministic validator checks it against a schema before the output is accepted. Failed validations trigger regeneration or error reporting. Applicability: HIGH — the verification step requested by the developer follows this established industry pattern. The validator in this case checks Markdown section structure rather than JSON schema, but the principle is identical.

**Finding 2 — Defense in depth for prompt reliability (NIST AI 100-1, Microsoft Responsible AI practices):**

Prompt-level constraints are "soft" controls — they depend on model compliance. Best practice for high-reliability AI agent systems is to layer soft controls (prompt instructions) with hard controls (deterministic validation). Prompt constraints reduce the probability of error; post-generation validation eliminates the impact of any remaining errors by catching them before they propagate. Applicability: HIGH — combining the read-scope restriction (soft control) with a verification step (hard control) aligns with defense-in-depth principles.

**Finding 3 — Self-verification in multi-step AI workflows (Chain-of-Verification, "Let's verify step by step" prompting):**

Research on LLM self-verification (Dhuliawala et al., 2023; Weng et al., 2023) demonstrates that explicit verification steps — where the model checks its own output against stated criteria — significantly reduce hallucination and formatting errors. The key requirement is that verification criteria are objective and deterministic (not subjective). Section heading presence/absence is fully deterministic and therefore ideal for self-verification. Applicability: HIGH — the verification step can be implemented as a self-check within the prompt without requiring external tooling, though an external script would provide stronger guarantees.

### AI Agent Critique

- **aib-analyze.md section 4.8 step 1 — missing read-scope restriction:** The instruction reads the full `analysis-<request_id>.md` without restricting what may be retained. The read-scope restriction should be added as a preventive measure (soft control) regardless of whether verification (hard control) is also added.

- **aib-analyze.md section 6.2 — "active session context" phrase is ambiguous:** The phrase "as understood from the active session context on re-run" invites the AI to draw from the full context window. This should be replaced with a precise reference to the extracted Input Interpretation text.

- **aib-analyze.md — no post-generation verification exists:** The prompt writes the analysis file and proceeds to plan updates without any structural check on the written content. This is the primary gap the developer wants filled.

- **analysis-convention.md section 4.3 — stale reference to `request-<request_id>.md`:** The convention says "when creating `request-<request_id>.md` on re-run" — this should reference `plan-<request_id>.md` since the rename in v1.3.0. Quality issue unrelated to current request scope.

- **context.md — GC identifier list is incomplete:** The "Execute analysis workflow" entry in `context.md` references "GC-01 through GC-06" but the actual prompt contains GC-07. The context description should be updated.

- **plan-R-20260519-1209.md — plan does not include verification step:** The current plan has 5 tasks focused on the read-scope restriction approach only. It needs revision to include the post-generation verification step the developer requested.

- **aib-analyze.md section 6.1 — prohibition phrasing relies on temporal ordering:** The sentence "The fact that the prior analysis file was read during this run… does NOT authorize retaining any of its content in the output" is a backward-looking prohibition. It is correctly placed as a belt-and-suspenders check but cannot prevent content that was already loaded from influencing generation.

## Implementation Alternatives

### Decision Fork: Verification mechanism type

**Applies to:** The new post-generation verification step to be added to `aib-analyze.md`.

**Why alternatives exist:** The verification could be implemented as either a prompt-level self-check instruction (the AI verifies its own output before proceeding) or as a tool-script invocation (a Python script performs the structural validation).

**Alternative A — Prompt-level self-verification instruction (recommended):** Add a new section (e.g., section 6.4 "Post-Generation Verification") to `aib-analyze.md` that instructs the AI to re-read the just-written `analysis-<request_id>.md`, count level-2 headings, verify each mandatory heading appears exactly once in the correct order, and halt with an error if verification fails (rewrite from scratch in the same run).

- Trade-offs: Simple to implement (no new script needed); stays within the existing prompt-only architecture; the AI is both generator and verifier which may create a conflict, but verification criteria are fully deterministic (heading presence/count) so subjective blind spots are unlikely; no additional maintenance burden.
- Codebase impact: Addition of one new sub-section in section 6 of `aib-analyze.md`; one test asserting the verification language is present.

**Alternative B — Tool-script verification:** Create a new Python script (e.g., `.aib_brain/tools/verify-analysis.py`) that parses the analysis file, checks section structure against the convention, and exits non-zero on failure. The prompt invokes this script after writing the analysis file.

- Trade-offs: Deterministic external validation (no model compliance dependency); reusable across runs; adds a new tool script to maintain; requires registration in section 2.2 External Dependencies; increases implementation scope.
- Codebase impact: New Python script + prompt modification to invoke it + test for the script itself + test for prompt reference.

- [ ] Option A: Prompt-level self-verification *(recommended)*
- [ ] Option B: Tool-script verification

### Decision Fork: Retain read-scope restriction alongside verification

**Applies to:** Whether to keep the read-scope restriction in section 4.8 step 1 and the "active session context" phrase replacement in section 6.2 as part of this fix, or rely solely on the verification step.

**Why alternatives exist:** The verification step catches non-conformance after it occurs but does not prevent it. The read-scope restriction prevents the most common trigger (full prior analysis in context). These are complementary but the developer questioned whether the read-scope restriction is "the correct solution."

**Alternative A — Defense in depth — keep both (recommended):** Retain the read-scope restriction (section 4.8) and phrase replacement (section 6.2) as preventive measures AND add the verification step as a detective/corrective measure.

- Trade-offs: Maximum reliability (prevent + detect); slightly more complex prompt; two layers of defense; aligns with industry best practices (NIST AI 100-1 defense-in-depth).
- Codebase impact: All changes from the prior plan plus the new verification section.

**Alternative B — Verification only:** Remove the read-scope restriction and phrase replacement from scope; rely solely on the post-generation verification step to catch and report non-conformance.

- Trade-offs: Simpler prompt change; relies entirely on detection rather than prevention; the AI may waste a generation cycle producing non-conformant output that then gets flagged; does not address the "active session context" ambiguous phrase.
- Codebase impact: Only the verification section is added; no changes to section 4.8 or 6.2.

- [ ] Option A: Defense in depth — keep both *(recommended)*
