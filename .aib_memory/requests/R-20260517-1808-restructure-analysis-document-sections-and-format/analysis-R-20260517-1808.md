## Input Interpretation

The AIB Developer has submitted seven structural and content directives to modify how the AIB Automation Agent generates the `analysis-<request_id>.md` artifact. These directives operate on two files: `analysis-convention.md` (the normative structure specification) and `aib-analyze.md` (the generation prompt). `context.md` also requires update because it references the old six-section list by name.

**Directive 1 — Relocate `## Request Context` to position 1.** The `## Request Context` section (currently the last of six mandatory sections) must be moved to the first position in the analysis document. This places the human-readable background and scope information at the top, immediately orienting the reader before any deep analytical content.

**Directive 2 — Move `**Request ID:**` and `**Request title:**` into `## Request Context`.** The two identity fields from `## Executive Summary` must be relocated into the repositioned `## Request Context` section, so that all request-identification information lives in the same top-level section.

**Directive 3 — Remove `## Executive Summary`.** After the two identity fields are moved (Directive 2), the `## Executive Summary` section retains only the "High-level purpose" item. That remaining item must be removed, and the section must be eliminated entirely. The identity fields satisfy the identification role previously served by Executive Summary.

**Directive 4 — Rename `## Request Context` to `## Overview`.** The repositioned, enriched section described in Directives 1–2 must be renamed to `## Overview`. The resulting `## Overview` section contains: Request ID, Request title, Background, Scope, and Out of scope — as a unified human-facing entry point for the analysis document.

**Directive 5 — Replace `### Feasibility Notes` and `### Expert Observations` with `### AI Agent critique`.** The two level-3 sub-sections within `## Research Results` that currently cover technical feasibility and expert developer perspective must be replaced by a single `### AI Agent critique` sub-section. The new sub-section has a substantially broader scope: it must review all workspace files read during the analysis run (not only those related to the current request), compare the requested changes against best practices, identify potential issues of all types (misalignment, inconsistencies, logical errors, redundancies, misplaced content, unclear wording, broken cross-references, format drift, etc.), and present every finding as a bullet list entry regardless of whether the finding relates to the current request scope. The intent is for the AI to act as an expert reviewer who surfaces all quality concerns encountered across the workspace artifacts it has read.

**Directive 6 — Restructure `## Implementation Alternatives` to per-decision-point format.** The current format mandates three named wholistic alternatives (Alternative A, B, C) representing different overall implementation approaches. This is too abstract. The new format must enumerate specific decision forks encountered in the request scope. For each fork, the prompt and convention must require: (a) identification of the specific task or step where the decision applies; (b) two or more named alternative approaches for that specific decision, with trade-offs and codebase impact; and (c) the resolution (tag `ask` or `resolve-autonomously` per existing classification rules in section 7.3.2 of `aib-analyze.md`). This ties alternatives directly to actionable decisions rather than wholistic implementation strategies.

**Directive 7 — Convert Decision Points from table to heading/sub-heading list.** The markdown table format of the current `### Decision Points` (or "Decision Points table") is inconvenient for revision — long cell text wraps poorly and editing requires understanding table syntax. The new format must use a heading/sub-heading list structure so each fork is easy to read and edit.

### Industry Knowledge

**Document-first information architecture (Information Mapping, Robert Horn; Nielsen Norman Group "F-pattern reading" research).** Placing high-salience, orienting information at the top of a document reduces cognitive load and accelerates comprehension. Moving the Overview (what the request is, why it exists, what is in scope) to position 1 directly applies this principle. Human reviewers of analysis artifacts most often want to verify scope and context before reading research or alternatives; the current position-6 placement of Request Context forces them to scroll past the analytical bulk to find it.

**Per-decision-point decision records (MADR — Markdown Architectural Decision Records, Michael Nygard's ADR format; Thoughtworks Technology Radar).** MADR structures each decision as a standalone record with named options, pros/cons per option, and a rationale statement. The current "Alternative A/B/C" format in AIB is a simplified version of this pattern, but applied at the macro (whole-request) level rather than per-decision. Shifting to per-decision-point records within `## Implementation Alternatives` aligns AIB with MADR's established practice and makes each decision independently traceable to the Q-blocks it generates.

**AI code/document review scope (GitHub Copilot code review research, Amazon CodeWhisperer, 2024 Gartner Emerging Tech research on AI-Augmented DevOps).** Effective AI review tools surface all observable quality concerns in the files they touch, not just those relevant to the requested change. The developer's `### AI Agent critique` concept mirrors "opportunistic review" behaviour documented in AI code review literature: because the AI reads the full file anyway, the marginal cost of surfacing unrelated quality issues is low, while the benefit (catching issues that would otherwise require a separate review cycle) is significant.

### AI Agent critique

The following issues were found during this analysis run across all files read. Issues are listed regardless of whether they relate to the current request scope.

**In `analysis-convention.md`:**

- Section 4 numbered list uses bold markers like `**[REQ]**` next to section names, but the body descriptions (sections 4.1–4.6) do not consistently echo this notation. This creates a minor formatting inconsistency between the summary list and the detailed sub-sections.

- Section 4.5 (`## Implementation Alternatives`) mandates "List the 3 most appropriate implementation approaches" — the number 3 is arbitrary and conflicts with scenarios where fewer or more alternatives exist. This is the issue the current request targets, but notably the convention wording is more prescriptive than the aib-analyze.md prompt wording, creating a subtle divergence between the two files.

- Section 4.5 states "At least one alternative MUST be documented even if only one viable approach exists" — after the request changes, this rule becomes unclear in the per-decision-point format. The rule should be restated as: "At least one named alternative per decision fork MUST be documented."

- Sections 6–10 are absent from the convention file. The convention jumps from section 5 (Maintenance Rules) to section 11 (Formatting Requirements). This looks like accidental section deletion or a numbering gap. Should be investigated and, if intentional, documented.

- Section 11 rule: "Make a list of sub-bullets if the bullet text is split by symbols like `;`" — the guidance "symbols like `;`" is vague. It is unclear whether `,`, `:`, `-`, or other delimiters also trigger the rule. Should be made explicit (e.g., "if a bullet contains a semicolon-separated list of items, reformat each item as a sub-bullet").

**In `aib-analyze.md`:**

- `context.md` at line 25 states the global constraints section covers "GC-01 through GC-06", but `aib-analyze.md` actually defines GC-01 through GC-07. GC-07 (No implementation writes) is documented in the prompt but omitted from the `context.md` description. This is a stale reference in `context.md` that will be corrected on the next context refresh only if the refresh synthesizes it from the actual prompt content.

- Section 4.3 (Step 3): "If `plan.md` is absent and no `## Questions` section exists in `input.md`, halt with the GC-04 error message." — The GC-04 error message template is `ERROR: Cannot read mandatory file <path>. Execution halted.` For the case where `plan.md` is absent (not a read failure but an existence gap), this message is slightly misleading; "cannot read" implies I/O failure rather than absence. The message could be made more precise.

- Section 7.3.2 references the developer's own `input.md ## Input` text as a valid source for `resolve-autonomously` tagging, but in the Auto-Request Creation Branch, `input.md` is archived before the analysis produces its Decision Points table. The AI must rely on the archived content (accessible only via GC-01-exempt `## Input Interpretation`). This creates a potential traceability gap: if the AI misinterprets the developer's intent in `## Input Interpretation`, a fork may be incorrectly tagged `resolve-autonomously`. The prompt should explicitly permit the AI to cite `## Input Interpretation` content as a proxy for the archived `input.md` when tagging forks.

- Section 8.1 defines the "non-stub" check using a literal seed string with `\n` escape sequences embedded inline in a Markdown document. While readable to humans who know the convention, an AI agent reading this for the first time might misinterpret the `\n` as literal characters. The definition could be improved by showing the seed as a fenced code block.

**In `plan-convention.md`:**

Affects the "File Location & Naming" section and possibly the "Lifecycle & Editing Rules" section. Should be fixed.

## Implementation Alternatives

The following alternatives represent materially different approaches to the structural changes requested.

### Alternative A — Minimal surface change (rename + reorder only)

Implement only the section renaming, reordering, and content moves (Directives 1–4) in `analysis-convention.md` and `aib-analyze.md`. Defer the `### AI Agent critique` definition change (Directive 5) and the Implementation Alternatives restructure (Directives 6–7) to a separate request.

- **Trade-offs:** Lower risk of breaking existing analysis patterns; faster to implement. However, leaves the most substantive improvements (AI Agent critique, per-decision alternatives) pending. Requires a follow-up request.

- **Expected codebase impact:** Moderate changes to `analysis-convention.md` and `aib-analyze.md`; minimal test changes.

### Alternative B — Full scope in one pass (all seven directives)

Implement all seven directives in a single implementation run: restructure the convention and the prompt together, add tests, and refresh context.md.

- **Trade-offs:** Full alignment between convention, prompt, and tests in one iteration. Higher change surface but no harder to implement given that all changes are to Markdown text files. No follow-up request needed.

- **Expected codebase impact:** `analysis-convention.md`, `aib-analyze.md`, `tests/test_analysis_prompt_structure.py`, and `context.md` (via refresh).

### Alternative C — Convention-only first, prompt update deferred

Update `analysis-convention.md` first (section structure), then update `aib-analyze.md` in a subsequent run. This avoids simultaneous changes to two inter-dependent files.

- **Trade-offs:** The convention and prompt would be temporarily out of sync between the two runs, which is a risk if any analysis is run in the interim. The convention is normative; having it describe a structure the prompt doesn't yet produce would generate non-compliant analysis files.

- **Expected codebase impact:** Only `analysis-convention.md` in the first run.

## Request Context

### Background

The current `analysis-<request_id>.md` structure places the human-oriented context (`## Request Context`) last, after all AI reasoning content. Reviewers who want to quickly understand what a request is about must scroll past research, alternatives, and decision tables. The `## Executive Summary` section at the top is too thin (three bullet items) to provide meaningful orientation. Additionally, the `### Feasibility Notes` and `### Expert Observations` sub-sections are narrowly scoped to the current request, missing the opportunity for the AI to surface quality issues it encounters across all workspace files it reads during analysis.

The `## Implementation Alternatives` format (three named wholistic alternatives) does not connect well to the Q-blocks it is supposed to drive: the alternatives are abstract strategies, not specific decision forks, making it harder for the developer to understand what is actually being asked.
