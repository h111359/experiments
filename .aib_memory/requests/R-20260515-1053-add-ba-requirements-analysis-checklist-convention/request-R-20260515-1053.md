## Goal

Add a business-analysis requirements validation convention to `.aib_brain/conventions/` that acts as an acceptance gate before implementation planning begins, ensuring every work request is sufficiently clear and complete from a user-requirements perspective. Integrate this gate into the `aib-analysis.md` prompt so the AI evaluates all mandatory checklist items against each active request and surfaces gaps as Q-blocks when they cannot be resolved by a documented assumption.

## Background

AIB currently processes requests through analysis and implementation without a formal, principled BA gate. Technically under-specified work items can reach implementation without clear goals, stakeholder identification, or testable acceptance criteria — resulting in solutions that may not meet actual user needs. Established requirements-engineering frameworks (BABOK, IEEE 29148, INVEST, SMART) define principled acceptance standards that can be expressed as extensible checklists. Embedding such a gate at analysis time aligns AIB with industry best practice and reduces rework risk.

## Scope

- A new convention file `requirements-analysis-convention.md` in `.aib_brain/conventions/` containing a structured, extensible BA checklist.

- The convention covers at minimum: Goal Clarity, Stakeholder Identification, Business Value, Scope Definition, Out-of-Scope Declarations, Constraints and Assumptions, Success Criteria, and Context Adequacy.

- Integration of the convention into `aib-analysis.md`: the prompt reads the convention file during preflight and evaluates every mandatory checklist item against the active `request.md` and `input.md`.

- Gate evaluation results are surfaced in the `## Research Results` section of the analysis document under a dedicated **Requirements Gate Evaluation** sub-heading.

- When a mandatory checklist item is unmet and cannot be satisfied by a reasonable documented assumption, the gap is tagged `ask` in the Decision Points table and a corresponding Q-block is raised in `input.md`.

- Enforcement rule: the `## Plan` in `request.md` MUST NOT contain a WBS when unresolved Q-blocks exist (plan deferral — already enforced by the existing stub mechanism); implementation MUST NOT proceed without a generated plan (already enforced by `aib-implement.md`).

- The convention file MUST include an Extension Guide section documenting how future maintainers can add new checklist items or categories without invalidating existing references.

## Out of scope

- Changes to any other convention file (coding conventions, context-convention, implementation-convention).

- UI or menu changes to `menu.py`.

- CI/release bookkeeping workflow changes.

- Generation of a per-request standalone "requirements document" artifact — the gate evaluation lives inside the analysis document only.

- Changes to `aib-implement.md` or `close-request.py`.

## Constraints

- The convention file must be plain Markdown using Markdown checkbox syntax (`- [ ]`) for checklist items.
- The convention file must be extensible: new items may be appended to existing categories and new categories may be added without breaking references to prior items.
- Must remain model-agnostic and vendor-agnostic (no tool-specific references).
- `aib-analysis.md` reads three convention files already; the new file is added as a fourth mandatory read in preflight step 8.
- No new Python tool scripts are required; the change is entirely in Markdown prompt and convention files.

## Success criteria

- `requirements-analysis-convention.md` exists at `.aib_brain/conventions/requirements-analysis-convention.md` with at minimum 8 checklist categories.
- `aib-analysis.md` lists `requirements-analysis-convention.md` as a mandatory preflight read (section 4.6 step 8) and documents the halt condition if the file cannot be read.
- When `aib-analysis.md` runs, it evaluates every mandatory checklist item against the active request and outputs a **Requirements Gate Evaluation** sub-section inside `## Research Results` of the analysis document.
- Any unmet mandatory item that cannot be resolved by a reasonable documented assumption generates a Q-block in `input.md`.
- The Extension Guide section in `requirements-analysis-convention.md` provides clear instructions for adding new items and categories.
- All existing test suites pass without modification.
