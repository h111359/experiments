## Research Results

### Workspace Findings

**`aib-analysis.md` — existing GC constraints:** The Global Constraints section currently defines GC-01 through GC-05. GC-01 reads: "`inputs/input-archive-*.md` files in request folders MUST NOT be read or referenced during any phase of this prompt." This is the closest existing safeguard, but it only covers `input-archive-*.md` files under the `inputs/` subfolder of a request — it does not cover `request-<id>.md`, `analysis-<id>.md`, `implementation-<id>.md`, or any other artifact that is moved to the request subfolder by `move-request-artifacts.py` when a request closes.

**`aib-implement.md` — Execution Requirements:** Under `## Rules / Execution requirements`, the only prohibitive statement is "MUST NOT read `inputs/input-archive-*.md` files from any request folder." Same gap as in `aib-analysis.md`.

**Two-phase placement rule (convention):** From `request-convention.md` and `analysis-convention.md`: during the Active phase all artifacts live at `.aib_memory/` root with ID-suffixed names; during the Archived phase they are moved to `.aib_memory/requests/<request-folder>/`. The register's `state` field distinguishes Active from Closed. This means an AI agent can always determine whether a file inside `.aib_memory/requests/` belongs to a Closed request by checking the register before reading it — or more simply, by treating ALL files inside closed-request subfolders as off-limits by default.

**Existing test coverage:** `tests/test_analysis_prompt_structure.py` checks for the presence of "Global Constraints" as a heading in `aib-analysis.md`, and tests for several structural elements. It does not verify individual GC entries. Adding a regression test for GC-06 would close this gap and prevent accidental deletion.

**Impact on self-referential behavior:** An AI agent running `aib-analysis.md` for a new request could theoretically search the workspace for "similar prior requests" and read closed request files to inform its analysis. GC-01 only blocks input archives; all other closed-request artifacts (especially `implementation-<id>.md` and `request-<id>.md`) remain unguarded.

### Industry Knowledge

**Least-privilege principle (OWASP / information security):** A well-established security and reliability pattern is to scope read access to only the files an agent legitimately needs for its current task. Closed request artifacts are not needed — they represent a different completed scope and are explicitly stale. Prohibiting their access is an application of least-privilege: the agent cannot act on stale data it was never given permission to read.

**Prompt injection / context poisoning (AI safety literature):** Closed request files can contain scope instructions, task descriptions, and implementation instructions that were valid at the time but are now superseded. If an AI agent reads them as context, it risks "context poisoning" — incorporating outdated instructions into its current reasoning. This is a well-recognised failure mode in agentic AI workflows (analogous to prompt injection via stale context). Making the prohibition explicit and non-negotiable ("hard constraint") is the standard mitigation pattern used in frameworks like LangChain guardrails and OpenAI Assistants API file-scoping features.

The new constraint references a path pattern (`.aib_memory/requests/<folder>/` for Closed requests) that is already established in the existing conventions and tool scripts. Adding a new GC entry and a parallel rule in `aib-implement.md` is low-risk.

### Expert Observations

The request scope is well-calibrated: narrowly focused on two prompt files and one test file. The fix is minimal — one new GC line in `aib-analysis.md`, one new rule line in `aib-implement.md`, and a supporting regression test. There is no risk of scope creep. The main risk is inconsistent phrasing between the two prompts, which should be mitigated by deriving both constraints from the same language. The broader risk (if not addressed) is that future AI agents running these prompts might exploit the gap by reading prior closed-request plan and implementation files under the guise of "workspace research," inadvertently applying stale constraints or overwritten decisions.

## Implementation Alternatives

### Alternative A — New standalone GC-06 in aib-analysis.md + parallel Execution rule in aib-implement.md

Add a GC-06 entry to the Global Constraints list in `aib-analysis.md` and a matching rule under Execution requirements in `aib-implement.md`. Both entries prohibit reading any file inside a Closed request's subfolder during any phase of execution.

Trade-offs:
- Benefit: Consistent with existing GC naming scheme; easy to reference by ID in other sections if needed.
- Benefit: Additive — does not modify GC-01 wording.
- Drawback: None identified.

Expected codebase impact: Two lines added (one per prompt file); one test method added.

### Alternative B — Extend GC-01 in aib-analysis.md to cover all closed-request artifacts

Broaden GC-01 to cover all files inside closed-request subfolders, not just `inputs/input-archive-*.md`.

Trade-offs:
- Benefit: Fewer GC entries.
- Drawback: Violates request Constraints ("MUST NOT change the existing GC-01 constraint wording; the new constraint is additive").

Expected codebase impact: Single line changed in `aib-analysis.md`; existing tests that assert specific GC-01 wording could break.
