### Background

During a second analysis run (the post-Q&A re-run), the AI agent wrote the new `analysis-<request_id>.md` content as a prepend or insert rather than a full replacement of the existing file. The resulting document contained duplicated sections. Two root causes were identified: (1) `analysis-convention.md` section 3 uses the word "amend" to describe the re-run write operation, which is naturally interpreted as "modify, append to, or patch" rather than "fully replace from scratch"; (2) `aib-analyze.md` section 6.1 states "Full content replacement" but provides no explicit prohibition against appending or prepending, and gives no guidance for the specific situation where the old analysis file was read moments before the write.

All other write contracts in AIB use "replace," "overwrite," or "fully replace." `plan-convention.md` reads "Re-runs of `aib-analyze.md` fully replace the active copy … without merging." The asymmetry is the primary defect.

**`analysis-convention.md` section 3 heading scope mismatch — confirmed:** The heading "## 3. The final substantive bullet after the fix will be a write-behavior prohibition — categorically different from naming and location. The proposed rename accurately describes the full contents of the section.

**`aib-refresh-context.md` and `docs/aib-refresh-context-AIB_version.md` — narrow disambiguation confirmed:** Both files carry the identical bullet "MUST produce full content replacement of `.aib_memory/context.md` on each execution (not append)." The "(not append)" clause rules out one mode of incorrect writes but not prepending or partial edits. For `aib-refresh-context.md`, the Answer Application Sub-flow does not exist — the prompt does not read the prior `context.md` before overwriting it — so the prior-read authorization sentence from `aib-analyze.md` section 6.1 is inapplicable; only the parenthetical extension is needed.

**Both `aib-refresh-context.md` files are identical in the relevant line:** Both `.aib_brain/prompts/aib-refresh-context.md` and `docs/aib-refresh-context-AIB_version.md` carry the same wording. Both must be updated consistently.

**No cross-references to the "File Naming & Location" heading name exist outside `analysis-convention.md`:** A workspace search confirms no other file references the section 3 heading by name. Heading rename has zero cross-reference impact.

**No tests target this language:** Confirmed — no test in `tests/` asserts the specific text of the full-replace rule in any of the four target files.

### Industry Knowledge

**Idempotent write principle:** A write-from-scratch contract is standard for generated artifacts in specification-driven and code-generation pipelines (cf. OpenAPI Generator, Swagger Codegen, Terraform provider generation). The artifact on disk must equal what the current run produces from scratch — prior state is irrelevant. Prohibiting all partial-write modes (append, prepend, partial edit) closes every known deviation path. Applicability: directly supports all four changes.

**"Explicit is better than implicit" (specification authoring principle):** Prohibitions stated as MUST NOT clauses are enforceable; those implied by positive statements are not. RFC-style specification language (BCP 14) distinguishes MUST, SHOULD, and MAY precisely because the strength of the obligation matters. "(not append)" is a parenthetical aside — weaker than a MUST NOT statement. Applicability: supports upgrading the parenthetical in both `aib-refresh-context.md` files and adding the MUST NOT prohibition in `aib-analyze.md` and `analysis-convention.md`.

**Section heading accuracy (information architecture principle):** A section heading is a navigational contract. Readers who need to find write-behavior rules will not think to look under "File Naming & Location." Renaming to include "& Write Behavior" makes the heading a complete description of its contents and reduces the chance that future editors introduce similar gaps. Applicability: directly supports the heading rename.
