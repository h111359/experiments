## Overview

**Request ID:** R-20260606-1517  
**Request Title:** Make plan self-sufficient for implement phase

### Background

Currently the AIB implement prompt (step 4 of `aib-implement.md`) reads `context.md` to obtain workspace-level product knowledge before executing the plan. This creates an implicit coupling between the plan artifact and the workspace memory store: a plan written today may produce a different implementation tomorrow if `context.md` changes. It also means that the plan cannot be handed to a human engineer or a fresh AI session without also providing `context.md`. The developer has requested that the plan become the single authoritative source for everything needed during implementation, eliminating the dependency on `context.md` at execution time.

### Scope

- Modify **step 9 of `.aib_brain/prompts/aib-analyze.md`** to produce self-sufficient plans: procedure steps must include exact file paths, context update tasks must include exact atomic statements rather than vague references, and the plan must contain sufficient background for execution without additional files.
- Modify **`.aib_brain/prompts/aib-implement.md`** to remove step 4 (context.md read) and add an explicit prohibition on reading `context.md` during implementation; allow only scripted editing of `context.md` as directed by the plan.
- Modify **`.aib_brain/conventions/plan-convention.md`** to: (a) remove `#### Risk notes` from the task schema; (b) mandate plan self-sufficiency with no reliance on `context.md` at execution time; (c) require context update tasks to include exact atomic statements.
- Update **`.aib_brain/README.md`** to reflect the revised plan structure and implementation constraints.
- Update **`.aib_memory/context.md`** to reflect revised data-flow, process, and functional requirements statements about the plan/implement lifecycle.

### Out of scope

- Changes to the analysis document structure or `analysis-convention.md`.
- Changes to `context-convention.md`, `requirements-analysis-convention.md`, `q-block-convention.md`, or any other convention file not listed in scope.
- Changes to tool scripts (Python files under `.aib_brain/tools/`).
- Changes to CI workflows under `.github/`.
- Any changes to `.aib_memory/context.md` content outside the statements that describe the plan/implement lifecycle.

## Research Results

### Workspace Pattern-Scan Findings

1. **aib-implement.md Step 4** currently reads `context.md` unconditionally and injects its content as "unified workspace product context". This is the primary coupling that the request targets. If removed, the plan must supply all context previously delegated to `context.md`.

2. **plan-convention.md — task schema** currently mandates `#### Risk notes` as a sub-field. The user explicitly requests its removal. There are no other references to `#### Risk notes` in the plan convention or in aib-analyze.md step 9 that would need corresponding updates beyond these two files.

3. **aib-analyze.md — step 9 (S09)** currently says: "Generate or recreate `plan-<request_id>.md` based on `.aib_memory/analysis-<request_id>.md` and project memory in `.aib_memory/context.md`." This phrasing explicitly designates `context.md` as a source. The step needs rewording so the plan is derived from the analysis and the user input only, and so all file paths and atomic context-update statements are embedded directly in the plan tasks.

4. **plan-convention.md — Operational Workflow** states "implement MUST rely on `plan-<request_id>.md` as its authoritative source" but does not explicitly forbid reading `context.md`. Adding an explicit prohibition closes the gap.

5. **plan-convention.md — Content Rules** states every plan must include "a task to update `.aib_memory/context.md` and any other documentation files affected by the request, reflecting changes made and any discovered discrepancies. The documentation task MUST specify for each step: (1) the target file path, (2) what to change, and (3) an acceptance test." The user's request tightens this: the plan must include exact atomic statements (e.g., specific `edit-context.py` command arguments with the exact text to insert or delete), not just descriptions of what to change.

6. **context.md** — the DF (Data Flow) section states: "Analysis generates `analysis-REQUEST_ID.md` and updates `plan-REQUEST_ID.md` with Plan and Decisions sections." The PR (Process) section states: "Implement workflow reads plan, executes tasks in order, runs tests, and closes request." The FN section includes "AIB MUST update `plan-REQUEST_ID.md` with Plan and Decisions section after analysis generation." These statements are all accurate and need only minor amendments to capture the self-sufficiency mandate.

7. **README.md** describes `plan-<id>.md` as "Request plan and decisions. Active copy lives at `.aib_memory/plan-<id>.md`; moved to request subfolder after implementation." No mention of self-sufficiency or context.md prohibition — this is where the update is needed.

### Industry Knowledge

1. **Self-contained work orders (PMI PMBOK Guide, 7th Ed. — "Work Package" concept):** A work package (plan task equivalent) is considered complete when it contains its own acceptance criteria, output definition, and execution steps such that the assigned team can execute it without requesting additional clarification. This directly supports the user's requirement for plan self-sufficiency and is a well-established project management principle.

2. **"Definition of Ready" in Agile/Scrum (Scaled Agile Framework, SAFe):** The Definition of Ready requires that every user story (plan task equivalent) be "Detailed Enough" — meaning the team has enough information to estimate and begin implementation without re-contacting the product owner. Mandating exact file paths and atomic context-update statements in plan tasks is a direct application of this principle to AI-assisted development.

3. **Explicit specification vs. implicit knowledge coupling (Design Patterns — Principle of Least Surprise, POLS):** Decoupling the plan from `context.md` follows the same reasoning as interface segregation in software: the consumer (implement) should depend only on the artifact that directly provides what it needs (the plan), not on a broader context store that may change independently. Removing the implicit read of `context.md` from `aib-implement.md` reduces surface area for subtle bugs caused by stale or updated context.

## Decision Register

### Decision Points

**DP-01 — Plan self-sufficiency mechanism: how should the `## Goal` section be extended?**

*Where it applies:* plan-convention.md — `## Goal` section definition; aib-analyze.md step S09 plan generation instructions.

*Why alternatives exist:* The user requires the plan to be self-sufficient for a human engineer without additional context. The current `## Goal` section only mandates "a concise, execution-oriented description." Adding background context requires either expanding the `## Goal` definition or introducing a new `## Background` section.

**Alternative A — Expand `## Goal` definition** *(chosen)*  
Mandate that `## Goal` include: (1) what the implementation must achieve, (2) the motivating background (why the change is needed), and (3) the impacted components. This satisfies self-sufficiency without adding a new mandatory section to the four-section structure.

Trade-offs: Simpler change; fewer structural impacts. May make `## Goal` longer than "a few sentences" as the convention currently suggests.

**Alternative B — Add `## Background` as a fifth mandatory section**  
Add a dedicated `## Background` section before `## Goal` for rationale and context.

Trade-offs: Clean separation of concerns, but breaks the four-section structure mandated by the convention and requires renumbering.

*Resolution:* `resolve-autonomously` — Developer input states "The plan should be able to be provided to a human engineer without any additional context." Alternative A satisfies this with minimal structural disruption. Plan-convention.md already mandates four sections; adding a fifth section would require more pervasive changes across all existing plans and the convention's validation rules. Expanding `## Goal` is the least-disruptive path that achieves the stated outcome.

**DP-03 — Context update task: exact statement specification**

*Where it applies:* plan-convention.md — Content Rules; aib-analyze.md step S09.

*Why alternatives exist:* The current convention says context update tasks must specify "the target file path, (2) what to change, and (3) an acceptance test" — vague enough to allow non-atomic descriptions. Self-sufficiency requires that context update task steps include exact `edit-context.py` command arguments (exact text of statements to insert or delete) so implement never needs to read context.md.

**Alternative A — Require exact `edit-context.py` command arguments** *(chosen)*  
Mandate that every context update procedure step include the exact `edit-context.py` invocation with the `--select`, `--insert`, or `--delete` arguments spelled out, so implement can run the command verbatim without reading context.md first.

Trade-offs: More work for aib-analyze.md (must look up existing context.md statements during analysis to formulate exact change commands). Self-sufficient plan. Implement never reads context.md.

**Alternative B — Require human-readable description of changes**  
Retain the current "what to change" description approach; implement is responsible for translating this into `edit-context.py` calls, which still requires reading context.md to understand current state.

Trade-offs: Less analysis work but breaks self-sufficiency. Requires implement to read context.md to understand what currently exists and formulate the correct edit command.

*Resolution:* `resolve-autonomously` — Developer input explicitly states: "Only editing context.md should be allowed if necessary for implementation, but it should be done according to the tasks instructions and using the python scripts, not reading the context.md." Alternative A directly implements this: the plan carries the exact commands so implement executes them verbatim. The analysis phase (when context.md IS available and CAN be read) is the right place to determine exact statement changes.
