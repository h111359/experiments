## Overview

- Request ID: R-20260521-1709
- Request title: Restructure aib-analyze.md for clarity and readability

### Background

The developer reports that the current `.aib_brain/prompts/aib-analyze.md` prompt has accumulated organisational drift that makes it hard to read and verify: numbered headings whose numeric labels no longer match the cross-references used in the body (e.g. body text says "section 4.7" or "section 2.1" but the heading numbering mixes conceptual and implementation ordering); a monolithic Section 4 (Mandatory Preflight) of roughly 200 lines that bundles the main sequence together with two complete sub-flows (Auto-Request Creation Branch, Answer Application Sub-flow); output specifications located in Sections 6 and 7 that the execution flow in Section 4 must jump forward and back to consult; constraints scattered between the Global Constraints block, Analysis Requirements (Section 5), and the per-section invariants inside the output contracts; mixed abstraction in Section 7 where format definitions live alongside procedural logic (Q-block generation rules, decision classification); and burial of the two sub-flows at depth 3 under "Preflight", which makes them hard to reference and discoverable only through internal navigation. The developer wants a restructured prompt that preserves identical behaviour while reorganising topics, chapters, and structure for the most appropriate human-readable arrangement.

### Scope

- Restructure `.aib_brain/prompts/aib-analyze.md` only — re-number, re-group, and re-locate sections; consolidate constraints; promote the two sub-flows to first-class top-level sections; co-locate execution-procedure phases with explicit forward references to a dedicated output-specifications chapter; align all internal cross-references (e.g. `section 4.7`, `section 7.3.2`, `GC-01..GC-07`) with the new numbering.

- Verify that all literal phrases and headings referenced by regression tests in [tests/test_analysis_prompt_structure.py](tests/test_analysis_prompt_structure.py) remain present after restructure (Execution Model Summary, Global Constraints, Failure Handling, GC-06, "All 4 mandatory sections", "Decision Points", "Input Interpretation", "deferred-creation", "`## Overview`", "Requirements Gate Evaluation", "all three convention files", "recursively", "archive the pre-reset `input.md` content", "If stub-equivalent: skip archive creation for this standard-flow reset.", "evaluate whether `.aib_memory/input.md` is in a non-stub state", `#### Outputs`, etc.).

- Update [.aib_memory/context.md](.aib_memory/context.md) only where it describes the structural layout of `aib-analyze.md` (e.g. the FR-004 wording naming "Section 4 organised into four labeled phases", references to "Section 7.3 split into three sub-sections", and similar) so that the published structure matches the new numbering.

### Out of scope

- No behavioural changes to the analysis workflow: every Preflight step, sub-flow, halt condition, Q-block rule, finalisation invocation, and output contract MUST remain functionally identical. No new behaviour, no removed behaviour, no different defaults.

- No edits to convention files (`analysis-convention.md`, `plan-convention.md`, `requirements-analysis-convention.md`) beyond what is strictly required to keep cross-references valid.

- No edits to `aib-implement.md`, `aib-refresh-context.md`, tool scripts, or tests beyond cross-reference alignment.

## Input Interpretation

The developer requests a structural re-organisation of the AIB analysis prompt action `.aib_brain/prompts/aib-analyze.md` to improve human readability and reviewability while strictly preserving its current execution semantics. The current prompt's defects, as enumerated by the developer, are: numbering drift between heading labels and in-body cross-references; a monolithic Preflight section that mixes the linear execution sequence with two distinct sub-flows; output specifications located far from the execution phases that consume them, forcing the reader to jump back and forth; constraints scattered across multiple sections (Global Constraints, Analysis Requirements, per-section invariants); mixed abstraction inside the Plan/Decisions/Q-block chapter (Section 7), where output format definitions are interleaved with procedural rules; and burial of the two sub-flows at depth-3 nesting under Preflight, hindering standalone reference.

The desired outcome is a reorganised prompt whose chapter sequence reflects a clean separation of concerns: (1) Objective; (2) Execution Model (phase list + authoritative invariants); (3) Global Rules (constraints + failure handling); (4) Inputs, Outputs & Dependencies; (5) Execution Procedure (phase-by-phase, with explicit forward references to the Output Specifications chapter); (6) Output Specifications (analysis document, plan document, Q-block rules); (7) Sub-flows (Auto-Request Creation Branch and Answer Application Sub-flow as standalone procedures, plus Re-run Behaviour Summary fused in as Section 7.3); (8) Completion Confirmation. The developer explicitly invited a counter-proposal and resolved via Q&A to adopt an 8-chapter layout: fusing `Re-run Behaviour Summary` into `Sub-flows` as Section 7.3 and moving `Context-Window Management` into `Global Rules` as Section 3.3. No ASCII flow diagram is added to Chapter 2. The old Section 5 analysis-quality invariants are split across three destinations: quality rules to the Analysis Generation phase sub-section inside Chapter 5, Q-block/Requirements Gate link to Section 6.3 (Q-block Rules), and Requirements Gate Evaluation surfacing instruction to Section 6.1 (Research Results generation instructions).

The hard constraint is behavioural equivalence: every Preflight step, sub-flow branch, halt condition, output contract, Q-block rule, finalisation invocation, and ordering invariant defined in the current prompt MUST be preserved verbatim in semantics — only their location, grouping, and presentation may change. All internal cross-references MUST be updated to match the new numbering, and all literal strings depended upon by the regression suite in `tests/test_analysis_prompt_structure.py` MUST remain present. The work also requires synchronising `.aib_memory/context.md` paragraphs that describe the prompt's structural layout so that the published context remains accurate.

### Industry knowledge

- **Diátaxis documentation framework (https://diataxis.fr/, Procida 2017–present)** — Procedural reference material is most navigable when it cleanly separates four documentation modes: tutorials, how-to guides, reference, and explanation. Applied to an AI prompt action that mixes execution procedure (how-to) with output schemas (reference) and rules (explanation), the strongest readability gain comes from physically clustering each mode in its own chapter and providing forward pointers — exactly the developer's proposed split between Execution Procedure (Section 5) and Output Specifications (Section 6). This pattern is directly applicable: it reduces the back-and-forth jumping the developer cited as a defect.

- **Microsoft Style Guide for technical reference (https://learn.microsoft.com/en-us/style-guide/reference) and Google developer documentation style guide (https://developers.google.com/style/reference)** — Both recommend co-locating rules with the operation they govern (e.g. put the input contract immediately next to the API surface). For a prompt where some rules apply globally and others apply per-step, the recommended pattern is a "global rules" chapter at the front (so the reader sees cross-cutting constraints once) plus inline per-step rules where applicable. The developer's proposed Section 3 (Global Rules) plus per-phase invariants in Section 5 conforms to this pattern. Applicability: high; the current `aib-analyze.md` already follows it partially (Global Constraints + per-section invariants), so the restructure mainly involves moving the Failure Handling table next to Global Constraints and trimming duplicate constraint statements inside Section 5.

- **OpenAI Cookbook prompt-engineering guidance (https://cookbook.openai.com/articles/related_resources, "Anthropic Prompt Engineering best practices" mirror, https://docs.anthropic.com/en/docs/build-with-claude/prompt-engineering)** — For long, multi-phase agentic prompts, the recommended layout is: (1) Role/Objective, (2) High-level execution model with explicit phase numbers, (3) Per-phase instructions with cross-references to a single source of truth for output schemas, (4) Standalone "branch" or "fallback" procedures listed after the main flow, (5) Final-step / completion confirmation at the end. The developer's proposed 10-chapter layout matches this pattern one-to-one. Applicability: high; this is the dominant pattern in production agent prompts.

- **RFC 2119 / BCP 14 plus IETF RFC document organisation conventions** — Normative documents place authoritative invariants at the top of each chapter (Introduction-then-Normative pattern) and provide a navigational summary appendix that disclaims its own non-normative status. The current Re-run Behaviour Summary already follows this pattern; the restructure should preserve the disclaimer when the summary chapter is renumbered.

- **Sphinx / Read-the-Docs convention for procedural manuals** — Place "Cookbook" or "Recipes" (i.e. sub-flows that interrupt the linear flow under specific triggers) in a dedicated chapter after the main reference, with cross-links from the trigger point. This is precisely the developer's proposal for Sub-flows (Section 7). Applicability: high.

## Implementation Alternatives

### Fork A — Top-level chapter layout

The developer proposed a 10-chapter layout (Objective; Execution Model; Global Rules; Inputs/Outputs/Dependencies; Execution Procedure; Output Specifications; Sub-flows; Re-run Behaviour Summary; Context-Window Management; Completion Confirmation) and explicitly invited a counter-proposal. The two viable alternatives are: (1) adopt the proposed 10-chapter layout verbatim; (2) adopt a refined 9-chapter variant that fuses Re-run Behaviour Summary into Sub-flows (since the summary is essentially a navigational appendix about re-runs that the sub-flows already cover) and moves Context-Window Management into Global Rules.

- Alternative A1 — Adopt the developer's 10-chapter proposal verbatim. Trade-offs: matches the developer's mental model exactly; minimal review friction; produces 10 top-level chapters which is at the upper end of comfortable navigation. Codebase impact: rewrite `aib-analyze.md` with 10 `##` chapters; update all cross-references and update context.md FR-004 prose.
- Alternative A2 — Adopt a refined 9-chapter variant (merge Re-run Behaviour Summary into Sub-flows as Section 7.3, and Context-Window Management into Global Rules as Section 3.3). Trade-offs: tighter chapter count, fewer top-level entries to scan; risk of deviating from the developer's clearly articulated proposal; the Re-run Summary's "non-authoritative" disclaimer is harder to enforce when nested under Sub-flows.

### Fork B — Treatment of analysis-quality invariants currently in Section 5

The current Section 5 ("Analysis Requirements") holds invariants that span two concerns: (a) analysis-quality rules (e.g. "MUST NOT ask the user for information you can collect yourself") and (b) Q-block / decision-fork rules (e.g. Requirements Gate Evaluation surfacing). Under the new layout these invariants must land somewhere.

- Alternative B1 — Move all Section 5 invariants into a single new sub-section 5.5 "Phase 4 — Analysis Generation: invariants" inside the Execution Procedure chapter. Trade-offs: keeps invariants close to where they are applied; risk of long sub-section. Codebase impact: one sub-section in Section 5.5.
- Alternative B2 — Split: analysis-quality invariants go into Section 5.5 (Phase 4); Q-block-related invariants migrate into Section 6.3 (Q-block Rules); the Requirements-Gate-Evaluation surfacing rule lives in Section 6.1 (Analysis Document → Research Results sub-section). Trade-offs: rules co-located with their target output (cleaner separation of concerns per Diátaxis); requires careful migration to avoid losing any rule.

### Fork C — Visual aid (ASCII flow diagram) for the Execution Model

The developer's stated goal is "easy for human to read and verify". A standard technique for improving phase-flow comprehension is an ASCII flow diagram showing the six phases plus the two interrupting sub-flows.

- Alternative C1 — Add a small ASCII phase-flow diagram inside the new Section 2 (Execution Model), in addition to the numbered phase list. Trade-offs: improves at-a-glance comprehension; uses ~20 extra lines; may drift if phases change.
- Alternative C2 — Keep Section 2 text-only (numbered phase list + invariants block, as in the current Execution Model Summary). Trade-offs: matches the current prompt's stylistic register; nothing new to maintain.
