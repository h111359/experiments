**Convention content summary:** `requirements-analysis-convention.md` defines eight checklist categories (Goal Clarity, Stakeholder Identification, Business Value, Scope Definition, Out of Scope, Constraints & Assumptions, Success Criteria, Context Adequacy), each with annotated `- [ ]` items. It includes an Applicability section, a pass threshold definition, and an Extension Guide. The Applicability section explicitly excludes _"Internal AIB framework maintenance tasks that have no user-facing product change (e.g., pure refactoring with no behavioral delta)"_ — this exclusion does NOT apply to the present request because adding the gate changes AI agent behavior during analysis (observable, user-facing change).

**Impacted files:** `.aib_brain/prompts/aib-analysis.md` (3 targeted edits), `tests/test_analysis_prompt_structure.py` (1 new test class with 3 tests), `.aib_memory/context.md` (re-generation after implementation to reflect new convention reads).

**Prior art in workspace:** The pattern of listing convention files in Section 2.2 and reading them in Step 8 was established when `analysis-convention.md` and `request-convention.md` were first introduced. Adding a third entry follows the identical pattern — no new mechanism is required.

### Industry Knowledge

1. **Requirements Engineering Gate Application (BABOK v3 / IEEE 29148:2018):** Both frameworks treat requirements review as an explicit, structured activity with defined entry and exit criteria that must occur before design begins. BABOK v3 Section 7 (Requirements Analysis and Design Definition) specifies that all requirements must satisfy quality characteristics (complete, consistent, correct, feasible, modifiable, prioritized, testable, unambiguous, verifiable) before architectural or implementation work starts. Applicability: the convention directly mirrors these characteristics via its eight categories. Wiring it into the analysis prompt ensures every request is evaluated against these criteria before a WBS is generated.

2. **Prompt Engineering — Explicit Context Registration (OpenAI Cookbook / LangChain agent patterns):** In AI-driven development agents, context must be explicitly injected via prompt instructions; implicit or ambient context is not loaded. A convention file not listed in the prompt's inputs section and not referenced in a reading step will not be consulted by the agent, regardless of its content. This is equivalent to dead code. The fix — adding a table row and a reading instruction — is the canonical pattern used in this codebase (see `analysis-convention.md` and `request-convention.md` as precedents).

### Feasibility Notes

No technical uncertainty. The change is purely editorial: three targeted additions to `aib-analysis.md`. The convention file exists, is well-formed Markdown, and is already covered by the Failure Handling wildcard. No tooling, CI, or template changes are required.

### Expert Observations

The scope is correctly sized — minimal and self-contained. The only non-trivial design choice is _where_ to add the gate application instruction in the prompt: it must appear after the convention is read (step 8 or later) but before analysis sections are written. Section 5 (Analysis Requirements) is the natural home because it governs what analysis content is mandatory. Placing the instruction there is consistent with the existing invariants listed in that section. No simplification opportunities exist; the change is already as small as it can be.

## Implementation Alternatives

### Alternative A — Add reference to Section 2.2, Step 8, and Section 5

Minimal, targeted changes: (1) add a row to the Section 2.2 External Dependencies table, (2) extend Step 8 to read three convention files instead of two, (3) add a gate application paragraph in Section 5. No structural reordering.

**Trade-offs:** Consistent with all precedents in the codebase; minimal diff; zero risk of breaking existing prompt logic.

**Codebase impact:** 3 edits to `aib-analysis.md`, 1 new test class in `test_analysis_prompt_structure.py`.

### Alternative B — Create a new dedicated Phase/Step for requirements gate

Add a new numbered step (e.g., step 8a or step 9a) inside Section 4 (Preflight) specifically for requirements gate evaluation, separating it from convention reads.

**Trade-offs:** More visible in the prompt, but adds complexity to an already-detailed Preflight sequence. The convention is a read-and-apply resource, not a separate execution phase; conflating it with phased execution logic would misrepresent its role.

**Codebase impact:** Larger diff; risk of numbering conflicts with existing step references in `context.md` and tests.
