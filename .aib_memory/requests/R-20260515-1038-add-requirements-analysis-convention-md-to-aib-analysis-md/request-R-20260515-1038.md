## Goal

Reference `.aib_brain/conventions/requirements-analysis-convention.md` explicitly in `.aib_brain/prompts/aib-analysis.md` so the analysis prompt loads and applies the requirements gate convention during every analysis run.

## Background

Request R-20260515-0947 created `.aib_brain/conventions/requirements-analysis-convention.md` — a structured, checklist-driven gate for evaluating whether a work request is sufficiently clear and complete before implementation planning begins. However, `aib-analysis.md` does not currently list this convention in its inputs, does not read it during Phase 4 (Context Enrichment), and does not instruct the AI to apply the gate checks as part of analysis. The convention exists but is effectively dead because no prompt references it.

## Scope

- `.aib_brain/prompts/aib-analysis.md` — add `requirements-analysis-convention.md` to the External Dependencies table (section 2.2), instruct the AI to read it in Preflight step 8 (convention reads), and specify how/when to apply the gate during the analysis run.

## Out of scope

- Changes to `.aib_brain/conventions/requirements-analysis-convention.md` content.

- Changes to any other prompt (`aib-implement.md`, `aib-context.md`, etc.).

- Retroactive application of the gate to already-closed requests.

## Constraints

- The `aib-analysis.md` prompt structure (numbered sections, labeled phases, GC identifiers) MUST be preserved; no sections may be renumbered or reorganised beyond what the request requires.

- The gate evaluation output MUST NOT be written as a separate top-level section in the analysis document; it MUST be surfaced within the existing `## Research Results` or `## Implementation Alternatives` sections, or as Q-blocks where the gate detects ambiguity.

- The convention MUST be read in the same step as the other conventions (Preflight step 8) to maintain the existing reading order.

## Success criteria

- SC-1: `aib-analysis.md` section 2.2 (External Dependencies) lists `requirements-analysis-convention.md` with its path and purpose.

- SC-2: `aib-analysis.md` Preflight step 8 explicitly names `requirements-analysis-convention.md` as one of the convention files to read (alongside the existing two).

- SC-3: `aib-analysis.md` contains a prose instruction specifying how and where the requirements gate results are surfaced in the analysis output.

- SC-4: Automated tests in `tests/` validate that the updated prompt text contains the expected references (path and usage instruction).

## Assumptions

- A1: The requirements gate evaluation results are surfaced in `## Research Results` of the analysis document — not as a separate top-level section — consistent with `analysis-convention.md` section 4.3.
  - Risk if false: The gate output could appear in an unexpected location, making it harder for reviewers to locate and act on.

- A2: The Failure Handling table wildcard `*.md` in `aib-analysis.md` already covers `requirements-analysis-convention.md`; no new halt-condition row is needed.
  - Risk if false: A missing convention file would not be caught by the halt gate — but the wildcard syntax makes this low risk.

- A3: `context.md` will be regenerated by running `aib-context.md` after implementation; it does not need manual editing.
  - Risk if false: `context.md` would temporarily reflect stale convention-read information — a low-severity documentation lag.
