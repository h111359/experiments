## Input Interpretation

The developer observes that `## 0. Brownfield Pre-Preflight` in `aib-analyze.md` is positioned as a section that executes before all Preflight phases, despite its sole concern being the availability of `context.md` — a resource consumed exclusively within Phase 4 — Context Enrichment (steps 6–9). The developer requests that the brownfield check be repositioned as an integral part of Phase 4, removing the conceptual anomaly of a pre-preflight section whose trigger condition and remediation action are entirely about context enrichment, not register-state or input-acquisition concerns.

The structural change is behavioral-neutral: steps 1–5 of the Mandatory Preflight (register-state resolution, request file read, attachment read, and input-option read) carry no dependency on `context.md`. The brownfield check can therefore be deferred to Phase 4 without altering when or how any other step executes.

## Research Results

### Workspace Findings

**1. Current Section 0 placement and referential structure**

`aib-analyze.md` Section 0 is declared as:
```
> **Trigger:** Executes before Preflight Step 1. MUST complete before any preflight step begins.
```
It instructs: "If absent or empty: execute `aib-refresh-context.md`… After execution completes, continue to section 4 (Mandatory Preflight)." And: "If present and non-empty: continue directly to section 4 (Mandatory Preflight)."

Internally, Section 0 cross-references "section 4 (Mandatory Preflight)" as its continuation target in both branches (absent/empty and present/non-empty). After relocation, these references must be rewritten to "continue to step 6" (or "continue with the next step within Phase 4") because the code is now inside Phase 4, not transitioning toward it.

**2. Phase 4 current description**

Phase 4 is labeled:
```
### Phase 4 — Context Enrichment
_Covers steps 6–8 (within section 4.6): context read, additional developer-flagged file reads, and convention reads._
```
After absorbing the brownfield check, the description must be updated to mention the brownfield context check as the phase's opening action and the step range updated to reflect it now logically covers step 6 (which includes the check) through step 9.

**3. Execution Model Summary — Section 0 not listed**

The `## Execution Model Summary` in `aib-analyze.md` lists six phases (Preflight, Branch handling, Analysis generation, Request enrichment, Q-block generation, Finalization). Section 0 is absent from this list. This confirms that the brownfield check was never intended as a seventh autonomous phase; it belongs within an existing phase. Phase 1 (Preflight / Context Enrichment) is the natural home.

**4. Test suite — no test guards Section 0 position**

`tests/test_analysis_prompt_structure.py` contains regression tests for structural sections (`Execution Model Summary`, `Global Constraints`, `Failure Handling`, `GC-06`, attachments language, etc.). No test asserts the presence of `## 0. Brownfield Pre-Preflight` as a level-2 heading. Relocating the content does not break any existing automated test.

A new regression test should be added to:
- Assert `## 0. Brownfield Pre-Preflight` is absent as a level-2 heading (prevents regression to the old position).
- Assert `aib-refresh-context.md` is still referenced within `aib-analyze.md` (confirms the brownfield logic was not accidentally deleted).

**5. `context.md` — references Section 0 as a distinct feature**

The `context.md` domain knowledge section states:
> "Add brownfield pre-preflight step (section 0) to `aib-analyze.md` that auto-executes context generation when `context.md` is absent or empty."

And the FR-004 description in the Requirements section states:
> "The prompt structure includes: an `## Execution Model Summary` section…; Section 4 (Preflight) organized into four labeled phases…"

Both references must be updated to reflect that the brownfield check now lives inside Phase 4 — Context Enrichment, not as a separate pre-preflight section.

### Industry Knowledge

**Finding 1 — Fail-late pattern (phase cohesion principle)**
In workflow automation literature (notably in Apache Airflow's DAG design guidance and Martin Fowler's *Patterns of Enterprise Application Architecture*), a widely recommended practice is to colocate prerequisite checks with the phase that consumes the resource being checked. Running a context-population check before register-state resolution violates phase cohesion: it introduces a dependency check into a phase boundary that the dependency does not affect. Moving the check to the phase that reads `context.md` aligns with the "fail-late but fail-right" principle — gates are placed where the resource is consumed, not at the earliest possible moment unless the resource is needed at that earliest moment.

*Applicability:* Directly applicable. Steps 1–5 of the Mandatory Preflight do not consume `context.md`. The brownfield check's earliest valid position is Phase 4.

**Finding 2 — Guard-clause at point of use (Clean Code / Robert C. Martin)**
Guard clauses — conditional checks that ensure preconditions are met — should be placed as close as possible to the point where the guarded resource is first used. Section 0 acts as a guard for `context.md`; `context.md` is first used in step 6. Placing the guard immediately before step 6 is the canonical pattern, analogous to null-checks placed immediately before dereference, not at function entry when the reference is only used deep inside a conditional branch.

*Applicability:* Directly applicable. Relocating Section 0's logic to immediately precede step 6 satisfies this pattern.

### Feasibility Notes

The change is a pure textual restructuring of `aib-analyze.md`. No tool scripts are modified. No register state or artifact schema changes. No test behavior changes beyond adding a new regression test. Risk of implementation failure: negligible. The only technical uncertainty is whether any test fixture or closed-request document embedded a string check on `## 0. Brownfield Pre-Preflight` — workspace scan confirms this is not the case.

### Expert Observations

The developer's instinct is correct and the fix is overdue. The `## 0. Brownfield Pre-Preflight` label creates a visual and structural anomaly: it implies the check is a prerequisite to preflight itself, when in fact the entire Preflight phases 1–3 run correctly without `context.md`. Placing it before the Objective section (`## 1. Objective`) further compounds the confusion by interrupting the document's top-level narrative flow (Objective → Execution Model → Constraints → Inputs → Preflight).

After relocation to Phase 4, the prompt reads as a coherent linear sequence: Preflight starts with state resolution (Phase 1), acquires inputs (Phase 2), applies mutations (Phase 3), then enriches context (Phase 4) — and Phase 4 naturally opens with "ensure context.md exists." The non-recursion guarantee note is preserved in-place within Phase 4, exactly where it is relevant (adjacent to the `aib-refresh-context.md` invocation instruction).

The implementation is low-risk, high-clarity, single-file, and directly testable.

## Implementation Alternatives

### Alternatives Enumerated

**Alternative 1 — Inline into step 6 as an opening conditional**
The brownfield check logic is folded directly into the prose of step 6. Step 6 becomes a compound step: first, ensure `context.md` exists (invoking `aib-refresh-context.md` if needed); then read `context.md`. The non-recursion guarantee note is embedded within step 6's text.

*Trade-offs:* Concise, fewest lines changed. Makes step 6 compound and harder to scan quickly. The non-recursion note is buried in body text.

*Codebase impact:* Minimal — step 6 text is extended; Section 0 is removed.

**Alternative 2 — Dedicated labeled block before step 6 (preamble to Phase 4)**
The brownfield check is extracted into a visually distinct block — a blockquote or a `> **Brownfield context check:**` callout — placed immediately before step 6 within Phase 4's section, separated from step 6 by a blank line. The non-recursion guarantee note is preserved as a distinct blockquote.

*Trade-offs:* Preserves the check's distinct visibility and keeps step 6 atomic. Slightly more lines than Alternative 1 but directly mirrors the existing Section 0 structure (trigger/purpose/steps/guarantee), making the diff minimal and the intent obvious.

*Codebase impact:* Section 0 removed; Phase 4 preamble block added; Phase 4 description updated.

**Alternative 3 — Rename and relabel as a Phase 4 named sub-section**
Add a `#### Brownfield Context Check` level-4 heading within Phase 4, before step 6, containing the current Section 0 content verbatim (except the continuation references).

*Trade-offs:* Explicit and scannable. Adds a level-4 heading to Phase 4 where none currently exist before 4.6. Slightly increases visual hierarchy complexity.

*Codebase impact:* Same as Alternative 2 plus a new `####` heading.

## Request Context

### Background

The `aib-analyze.md` prompt was extended in request R-20260516-1217 with a brownfield detection step placed before all Preflight phases (`## 0. Brownfield Pre-Preflight`). The stated rationale was ensuring `context.md` is populated before analysis proceeds. However, Preflight phases 1–3 (state resolution, input acquisition, state mutation) carry no dependency on `context.md`. The check executes too early relative to its purpose, creating a structural anomaly: a pre-preflight section whose sole concern is a resource consumed in Phase 4.

### Scope

- `.aib_brain/prompts/aib-analyze.md` — remove `## 0. Brownfield Pre-Preflight` level-2 section; integrate its logic as a preamble block within Phase 4 — Context Enrichment, immediately before step 6; update Phase 4's description line; rewrite internal cross-references from "continue to section 4 (Mandatory Preflight)" to "continue to step 6".
- `tests/test_analysis_prompt_structure.py` — add regression test asserting the level-2 `## 0. Brownfield Pre-Preflight` heading is absent and `aib-refresh-context.md` is still referenced.
- `.aib_memory/context.md` — update domain knowledge and FR-004 descriptions to remove references to "section 0" and "pre-preflight" and reflect Phase 4 integration.

### Out of scope

- `aib-refresh-context.md` — no changes; its content and behavior are unchanged.
- Any tool script (`.aib_brain/tools/`) — no changes.
- `requests_register.md` — no changes.
- Any closed-request artifacts — not read or modified.
- Behavioral changes to when or how `context.md` is generated — this is a structural repositioning only.
