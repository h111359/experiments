## Goal

Add a business requirements analysis convention file in `.aib_brain/conventions/` that defines — in checklist form — what must be verified as clear and complete in `input.md` and `context.md` before any technical or implementation decisions are made. The convention shall encode best practices from business requirements gathering and serve as acceptance criteria for input quality.

## Background

Currently AIB proceeds to technical analysis and implementation planning as soon as a request is submitted, without a dedicated gate that verifies the request is sufficiently clear from a business and user-requirements perspective. This can result in ambiguous or incomplete requests being implemented, causing rework. A structured requirements-quality checklist convention, applied before technical analysis, will reduce ambiguity and improve the quality of implemented features.

## Scope

- Create a new convention file `.aib_brain/conventions/requirements-analysis-convention.md` containing a structured checklist for requirements quality.

- The checklist MUST cover: goal clarity, stakeholder/user identification, business value justification, scope completeness, out-of-scope explicitness, success criteria measurability, constraint identification, context.md adequacy, and any cross-cutting concerns.

- Best practices from requirements engineering (e.g., BABOK, IEEE 830, Agile user-story criteria such as INVEST, SMART criteria for acceptance) MUST be incorporated as checklist categories or items.

- The convention MUST be extensible: it MUST be structured so that future checklist items and categories can be added without restructuring the file.

- The convention file MUST include a short preamble explaining its purpose and how it is used as an acceptance gate before implementation.

- The convention MUST state explicitly that it serves as acceptance criteria determining when input is of sufficient quality for implementation to proceed.

## Out of scope

- Changes to existing convention files (analysis-convention.md, request-convention.md, etc.).
- Changes to any AIB prompt files (aib-analysis.md, aib-implement.md, etc.) — integration of this convention into the prompt workflow is a separate future request.
- UI or tooling changes.
- Changes to `context.md` content.

## Constraints

- The new convention file MUST be a standalone Markdown file in `.aib_brain/conventions/`.
- The checklist format MUST use Markdown checkboxes (`- [ ]`) so items can be acted upon interactively.
- Each checklist item MUST be annotated with a brief (1–2 sentence) inline explanation describing why the criterion matters and what "satisfied" means (Q001 → Option B).
- Checklist items MUST be organized into named category groups (5–8 domain groups, each with its own items) to align with industry frameworks and enable principled future extension (Q002 → Option A).
- File naming MUST follow existing convention file naming patterns (lowercase, hyphenated).
- No changes to existing files are permitted except for documentation updates if discovered necessary.

## Success criteria

- `.aib_brain/conventions/requirements-analysis-convention.md` exists and is non-empty.
- The file contains a preamble explaining its purpose and usage as an acceptance gate.
- The checklist covers at minimum: goal clarity, stakeholder identification, business value, scope completeness, out-of-scope definition, measurable success criteria, constraints, and context.md adequacy.
- Best practices from at least two established requirements frameworks (e.g., BABOK, IEEE 830, INVEST, SMART) are incorporated and cited.
- The structure allows future items to be added without restructuring the file (e.g., categorized sections).
- All checklist items use `- [ ]` checkbox format.
- A declaration is present stating the checklist serves as an acceptance gate before implementation.

## Plan

### Task 1: Create `requirements-analysis-convention.md`

#### Intent
Author the full requirements analysis convention file with preamble, acceptance gate declaration, eight annotated category groups, and a pass threshold statement.

#### Outputs
`.aib_brain/conventions/requirements-analysis-convention.md` (new file, ~130–160 lines)

#### Procedure
1. Create `.aib_brain/conventions/requirements-analysis-convention.md` with the following structure:
   - `# Convention: requirements-analysis-convention.md` — title heading
   - `## Purpose` — one-paragraph description of the convention's role as a requirements quality gate
   - `## Applicability` — when and by whom the checklist is used
   - `## Normative Language` — BCP 14 note
   - `## Acceptance Gate Declaration` — explicit pass/fail threshold statement: all mandatory items checked (no unchecked `- [ ]` items in mandatory categories), no critical-severity gaps documented
   - Eight category sections (each `### <Category Name>`), each containing annotated `- [ ]` checkbox items:
     1. `### 1. Goal Clarity` — items: goal stated, single clear intent, implementation-free (no solution prescribed), goal traceable to a business need (BABOK: Unambiguous, Atomic; IEEE 29148: Implementation-free, Necessary)
     2. `### 2. Stakeholder & User Identification` — items: at least one named stakeholder or user role identified, stakeholder need is explicit, impacted parties listed (IEEE 29148: Traceable; BABOK: stakeholder alignment)
     3. `### 3. Business Value & Justification` — items: business value or benefit stated, request is prioritized / reason for priority explained, no equivalent existing solution unaddressed (INVEST: Valuable, Estimable; BABOK: Feasible, Prioritized)
     4. `### 4. Scope Definition` — items: scope is bounded and explicitly listed, scope is deliverable within a reasonable iteration without decomposition, no scope items contradict each other, scope does not include implementation details (INVEST: Small, Independent; BABOK: Complete, Consistent; IEEE 29148: Singular)
     5. `### 5. Out of Scope` — items: at least one out-of-scope exclusion stated, exclusions are explicit and unambiguous (BABOK: Concise; prevents scope creep)
     6. `### 6. Constraints & Assumptions` — items: all known technical constraints stated, all known business constraints stated, assumptions that could affect scope are documented, constraints are separated from functional scope items (IEEE 29148: explicit constraint separation; INVEST: Negotiable)
     7. `### 7. Success Criteria & Acceptance` — items: at least one success criterion defined, each criterion is measurable (has a concrete pass/fail test), each criterion is achievable within stated constraints, criteria are specific (not vague), criteria cover all stated scope items (SMART: Specific, Measurable, Achievable; BABOK: Testable; IEEE 29148: Verifiable)
     8. `### 8. Context Adequacy` — items: `context.md` has been read and is non-empty, product domain and key actors are described in context, relevant prior decisions or constraints from context are reflected in the request, no conflict exists between request and documented product constraints
   - `## Extension Guide` — one-paragraph note on how to add new items (append within category) or new categories (append new `###` section at end)

2. Verify all checklist items in `.aib_brain/conventions/requirements-analysis-convention.md` use `- [ ]` format and each item has a 1–2 sentence annotation on the line immediately following.

3. Verify the file contains the `## Acceptance Gate Declaration` section with an explicit pass/fail threshold statement.

4. Verify eight category sections (`### 1.` through `### 8.`) are present and non-empty.

#### Done criteria
- File `.aib_brain/conventions/requirements-analysis-convention.md` exists and is non-empty.
- File contains `## Purpose`, `## Acceptance Gate Declaration`, eight `### N.` category sections, and `## Extension Guide`.
- All checklist items use `- [ ]` format.
- Each item has an inline annotation (at least one sentence following the checkbox line).
- At least two framework citations (BABOK, IEEE 29148, INVEST, or SMART) appear in the file.

#### Dependencies
None.

#### Risk notes
- Annotation verbosity: if annotations grow beyond 2 sentences, refactor to a category-level note to preserve scannability.
- Item count: keep total items to 25–35 to avoid an unwieldy checklist.
