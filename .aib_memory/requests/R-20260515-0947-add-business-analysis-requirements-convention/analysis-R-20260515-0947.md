### Industry Knowledge

**Finding 1 — BABOK v3 (Business Analysis Body of Knowledge, IIBA):**

BABOK v3 defines eight quality characteristics for well-formed requirements: *Atomic, Complete, Consistent, Concise, Feasible, Unambiguous, Testable, and Prioritized*. The "Requirements Life Cycle Management" knowledge area specifies traceability and stakeholder-need alignment as prerequisites for requirements that can be tested against business outcomes.

*Applicability:* Maps directly to checklist categories: Goal Clarity (Unambiguous, Atomic), Business Value (Feasible, Prioritized), Scope (Complete, Consistent), Success Criteria (Testable). BABOK's quality attributes provide the theoretical justification for most items.

**Finding 2 — IEEE 29148-2018 (International Standard for Systems and Software Engineering — Life Cycle Processes — Requirements Engineering):**

IEEE 29148 defines good requirements as: *Necessary, Implementation-free, Unambiguous, Consistent, Complete, Singular, Feasible, Traceable, and Verifiable*. It requires stakeholder identification as a necessary pre-condition, explicit separation of constraints from functional requirements, and that each requirement be verifiable with a concrete pass/fail test.

*Applicability:* Stakeholder Identification category, Constraints category (explicitly separating constraints from scope), and Success Criteria category (verifiable = measurable with a pass/fail test). IEEE 29148's "Implementation-free" criterion maps to an important checklist item: the goal/scope must not prescribe a specific technical solution.

**Finding 3 — INVEST Criteria (Bill Wake, XP Community):**

INVEST (Independent, Negotiable, Valuable, Estimable, Small, Testable) evaluates user-story quality. *Valuable* checks explicit business value; *Estimable* checks bounded scope; *Small* checks deliverability within an iteration; *Testable* checks concrete done-criteria. *Negotiable* maps to explicit assumptions and constraints being stated so trade-offs can be discussed.

*Applicability:* Business Value category (Valuable, Estimable), Scope category (Small/bounded), Constraints/Assumptions category (Negotiable). INVEST bridges the gap between requirements completeness and implementation readiness — the exact purpose of the new convention.

**Finding 4 — SMART Criteria (Doran, 1981; Management by Objectives):**

SMART (Specific, Measurable, Achievable, Relevant, Time-bound) is applied to success criteria and acceptance criteria. *Measurable* requires quantitative or unambiguous qualitative tests. *Achievable* requires no known technical blocker. *Specific* requires unambiguous, non-interpretable statements.

*Applicability:* Success Criteria category. Each checklist item in that category should verify: criterion is Specific (not vague), Measurable (has a concrete test), and Achievable within stated constraints.

### Expert Observations

**Scope assessment:** Appropriately sized. A single convention file with annotated, grouped checklist items is a focused, low-risk deliverable. The main risk is over-engineering: generating an excessive number of checklist items. Target 25–35 items across 7–8 categories. Items should be high-signal, not exhaustive.

**Q001 resolution (annotated format) — design implication:** Each item will have a 1–2 sentence annotation. Annotations MUST stay concise — they are inline prose, not paragraphs. If annotations become too long, they should be refactored into a guidance note at the category level.

**Q002 resolution (grouped categories) — category design:** Eight categories are appropriate given the request scope and frameworks referenced:

1. Goal Clarity
2. Stakeholder & User Identification
3. Business Value & Justification
4. Scope Definition
5. Out of Scope
6. Constraints & Assumptions
7. Success Criteria & Acceptance
8. Context Adequacy (context.md coverage)

**Test coverage:** The convention file itself is verifiable via automated tests: file exists, preamble present, all categories present, checkbox format used, pass/fail gate declaration present. A test task MUST be included in the plan.

**Maintainability:** The convention is owned by the AIB Maintainer role. It must be self-documenting (preamble explains purpose) and must not require knowledge of BABOK/IEEE to extend — category headings provide the organizing principle.

## Implementation Alternatives

### Alternative A: Annotated, category-grouped checklist (selected)

A convention file organized into 7–8 named category sections, each containing `- [ ]` checkbox items with 1–2 sentence inline annotations. A preamble section defines purpose, usage, and the acceptance-gate pass threshold.

- **Trade-offs:** Practical usability (annotations reduce ambiguity), principled extensibility (category grouping), aligned with four industry frameworks. Moderate file size (~120–160 lines).

- **Codebase impact:** Single new file `.aib_brain/conventions/requirements-analysis-convention.md`. Supporting changes: `context.md` documentation update (separate task), automated test (separate task).

### Alternative B (superseded): Bare-checkbox flat list

A single flat list of checkboxes without annotations or categories.

- **Superseded reason:** Q001 → Option B (annotated) and Q002 → Option A (grouped) were selected. This alternative does not reflect developer choices.
