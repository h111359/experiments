## Input Interpretation

The AIB Maintainer requests the permanent removal of `.aib_brain/templates/request-template.md` — a legacy seed template that was used by `create-request.py` prior to v1.2.0 to seed an initial stub of `request.md` in the request folder. Since ADR-0006 (implemented in v1.2.0), `create-request.py` creates only the request folder and register entry; `request.md` is now AI-generated in full by `aib-analysis.md` from the developer's `input.md` content, populated according to `request-convention.md`. The template is therefore dead code: it is not read by any tool script, prompt, or test, and its continued presence creates a false impression that it plays a role in request creation.

The developer's second sentence — "The structure of the request should be defined only from the respective convention and the analysis prompt" — confirms the intended state of the system after this change: `request-convention.md` is the normative structural authority; `aib-analysis.md` is the generation mechanism. The template file has no role in either.

The scope includes: (1) deleting the file itself, and (2) removing all non-archive references to it in the active workspace. Archived `context.md` snapshots under `.aib_memory/archives/` are out of scope by design — they are immutable historical records.

### Industry Knowledge

**Finding 1 — Dead code removal (Software Engineering principle, widely documented in "Clean Code" by Martin, and the YAGNI principle from Extreme Programming):** Retaining artifacts that are explicitly superseded and have no consumers introduces maintenance overhead and cognitive noise. Engineers reading the codebase must reason about whether the artifact could be in use somewhere undiscovered. Removing it eliminates that uncertainty. Applicability: directly applicable — the file has no consumers as confirmed by workspace scan.

**Finding 2 — Single source of truth for structural schemas (IEEE 29148, BABOK):** When both a template file and a convention file define the same structure, teams face a synchronization problem: changes to one may not propagate to the other, causing drift. Eliminating the template and retaining only the convention eliminates the dual-maintenance concern. Applicability: directly applicable — `request-convention.md` is the normative authority; the template was a redundant structural definition.

### Feasibility Notes

No technical uncertainty exists. The change involves one file deletion and two string removals from a single Markdown document. The `context.md` file is auto-generated by `aib-context.md`, so the removed entries will not reappear unless `aib-context.md` is re-run — and when it is, it will scan the live workspace and naturally omit the now-deleted file. There is no risk of regression in tool scripts or tests.

### Expert Observations

**Scope:** Appropriately scoped. This is a minimal, targeted cleanup with zero risk of scope creep.

**Risks:** None material. The only risk is accidentally editing `context.md` in a way that corrupts adjacent bullets — mitigated by precise string matching in the implementation step.

**Simplification opportunity:** None beyond what is requested.

**Maintainability:** After this change, the templates folder will contain exactly one file (`requests_register-template.md`), which is actively used. Any future engineer reading the folder will understand its purpose without encountering a confusing legacy artifact.

## Implementation Alternatives

### Alternative A — Delete file + direct `context.md` edits (chosen)

Remove the template file and immediately edit `.aib_memory/context.md` to remove the two bullet entries referencing it.

- **Trade-offs:** Immediate consistency; no waiting for a future `aib-context.md` run. Requires careful string matching to avoid corrupting adjacent content in `context.md`. Low risk given two well-identified, unique strings.
- **Codebase impact:** One file deleted from `.aib_brain/templates/`; two bullet lines removed from `.aib_memory/context.md`.

### Alternative B — Delete file + deferred `context.md` update

Remove the template file only; allow a future `aib-context.md` run to regenerate `context.md` without the now-absent file.

- **Trade-offs:** Leaves stale references in `context.md` until the next context regeneration. Violates the developer's explicit instruction to "remove all references."
- **Codebase impact:** One file deleted; `context.md` temporarily inconsistent.
