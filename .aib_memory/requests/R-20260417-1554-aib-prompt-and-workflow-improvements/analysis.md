## Domain Knowledge Essentials

**AI Builder (AIB):** A minimal, model-agnostic framework for specification-driven development. Operates entirely through filesystem reads and writes — no network calls at runtime. Its "brain" (`.aib_brain/`) holds reusable, immutable-during-runtime assets; its "memory" (`.aib_memory/`) accumulates workspace-specific state.

**Request:** The atomic unit of work in AIB. Defined in `request.md` within a dedicated folder under `.aib_memory/requests/`. Lifecycle: Active → Closed. Only one Active request may exist at a time.

**"No changes" toggle:** An opt-in flag in `input.md`'s `## Options` section (`- [x] No changes — provide answer only`). When checked, the analysis run writes a timestamped answer file in the request folder and resets `input.md` with the current active request ID. It MUST NOT touch `request.md`, `analysis.md`, or any other file. Intended as a read-only Q&A mode.

**`input.md` lifecycle (corrected):** Ephemeral communication channel between developer and AI agent. Developer writes intent into `## Input`; the agent reads it, processes the analysis run, and resets the file to the seed template as the very last step. The reset must happen only after all other artifacts are fully written. After reset, `## Active request` must display the current active request ID and title. Exception: when `aib-analysis.md` is triggered from `aib-implement.md` (Auto-Analysis Branch), `input.md` must NOT be reset by the analysis run — implement manages the lifecycle.

**Standard analysis flow gap:** The current `aib-analysis.md` standard flow (Active request exists, no toggles) contains no `input.md` reset step. After every standard analysis run, `input.md` retains whatever content the developer last wrote. Fix: add a reset step as the final action of the standard flow, conditioned on "unless triggered from `aib-implement.md`".

**Auto-Request Creation Branch:** A branch of `aib-analysis.md` activated when no Active request exists and `input.md` is non-empty. Reads `input.md`, derives a title, invokes `create-request.py`, generates AI-authored `request.md` with all 14 mandatory sections (1–6 with non-empty human-derived content, 7–14 present and populated by the analysis continuation in step 8), archives `input.md`, and resets `input.md` after all artifacts are written.

**14 mandatory sections of `request.md`:** Per `request-convention.md`, all 14 sections must be present. Sections 1–6 (Goal, Background, Scope, Out of scope, Constraints, Success criteria) must be non-empty and are human-authored. Sections 7–14 (Assumptions, Plan, Testing, Documentation, Questions & Decisions, Code and Asset Scan, Internal Review, Multi-Perspective Stakeholder Review) are AI-generated and populated during the analysis run.

**Seed template for `input.md`:** Fixed content produced after reset. Currently `initialize.py` seeds with `No active request` as the placeholder; after the fix, the prompt writes the actual active request ID and title at reset time.

**`## Active request` section:** The first section of `input.md`. After the fix, must always contain the current active request ID and title (e.g., `R-20260417-1554 — AIB prompt and workflow improvements`) rather than the literal `No active request`.

**Impacted roles/personas:**

- Developer — writes `input.md` content, observes `## Active request` to confirm which request is active, uses "No changes" mode for Q&A without side-effects.

- AI Automation Agent — executes prompt workflows; must follow corrected timing, scope, wording, and section-generation rules.

- AIB Maintainer — owns prompt files, conventions, CI workflow, and documentation.

**Business processes touched:**

- Communicate user intent (`input.md` lifecycle, standard-flow reset gap fix).

- Execute analysis workflow (standard branch reset, Auto-Request Creation Branch 14-section generation).

- Execute implement workflow (no-confirmation auto-analysis branch).

- Release bookkeeping (CI `git add` fix).

## Technical Knowledge & Terms

**`aib-analysis.md`:** Prompt file governing the analysis run. Contains three execution paths: (a) standard flow (Active request, no toggles), (b) "No changes" branch (toggle checked), (c) Auto-Request Creation Branch (no Active request). Confirmed gaps: (a) standard flow has no `input.md` reset step; (b) "No changes" branch prohibition list is incomplete; (c) Auto-Request Creation Branch step 5 enumerates only 6 of 14 mandatory sections; (d) Auto-Request Branch resets `input.md` at step 7 before analysis generation at step 8 — order must be reversed.

**`aib-implement.md`:** Prompt file governing implementation. Contains an Auto-Analysis Branch that fires when no Active request exists — triggers `aib-analysis.md` and then continues. Current wording does not explicitly prohibit seeking user confirmation. When analysis is triggered from here, the analysis run must NOT reset `input.md`.

**`.github/workflows/aib-semver-patch-bump-and-log.yml`:** GitHub Actions CI workflow. Runs on PR events targeting `main`. Invokes `release_bookkeeping.py`, which produces `versions/aib_brain_vX.Y.Z.zip`. The "Commit and push" step currently runs `git add .aib_brain logs` — `versions/` is omitted, so the zip is generated but never committed.

**`release_bookkeeping.py`:** Python script that bumps the patch version, rotates the marker file, writes a per-version log, and creates `versions/aib_brain_vX.Y.Z.zip`. Script logic is correct; only the CI `git add` is missing `versions`.

**`request-convention.md`:** Governs `request.md` structure. Specifies 14 mandatory sections. Sections 1–6 are human-authored and must be non-empty; sections 7–14 are AI-generated. The analysis prompt's Auto-Request Creation Branch cites only 6 — a gap against the convention that must be fixed in Task 3.

**`edit_allowed` flag:** Column in `references.md` controlling whether AIB automation may edit a file. `Concepts.md` is listed with `edit_allowed=N`. The request scope explicitly authorizes the edit — the implement run must honor this override.

**Idempotency:** Running the same prompt multiple times on the same state converges to the same output. The `input.md` reset with active-request-ID substitution must be idempotent.

**`versions/` directory:** Tracked in VCS (no `.gitignore` rule found excluding it). Must be added to CI `git add` for the zip to be committed.

**Technologies in scope:** Markdown (prompt files, documentation), YAML (GitHub Actions workflow), Python 3.10+ (read-only dependency; no changes needed), Git/GitHub (CI commit and push model).

**Non-functional attributes:**

- Model-agnostic (NFR-001): all fixes are prose-level; no vendor-specific syntax introduced.

- Idempotent CI (NFR-005): `git add .aib_brain logs versions` is safe to re-run; staging an already-staged directory is a no-op.

### Pattern Findings

1. **"No changes" branch — wording gap, not a logic gap:** The current branch already writes `answer-<timestamp>.md` and resets `input.md` and says "Do NOT modify `request.md` or generate `analysis.md`." The prohibition list is incomplete — it names two things but does not enumerate the full set of prohibited writes. The developer's amendment confirms the reset must remain. The fix is to strengthen the wording: explicitly state the two permitted writes and prohibit all others by name.

2. **`input.md` reset timing — Auto-Request Branch:** Step 7 (reset `input.md`) precedes step 8 (proceed with standard analysis). If analysis fails after the reset, the user's original input is lost. Reversing the order (reset as the final step) eliminates this data-loss risk.

3. **`## Active request` content after reset:** Both the "No changes" branch reset and the Auto-Request Branch reset must write the active request ID and title to `## Active request`. The ID and title are already known at reset time from the preflight active-request resolution. No additional tool invocation needed.

4. **Auto-Request Branch `request.md` generation:** Step 5 references `request-convention.md` and says all 6 sections "MUST be present and non-empty" but does not enumerate headings. An explicit list and pre-write verification instruction reduce the risk of partial generation on edge cases.

5. **`aib-implement.md` no-confirmation gap:** The zero-Active-rows branch describes triggering analysis and continuing but contains no prohibition on asking the user for permission. One explicit imperative sentence closes this gap reliably across models.

6. **CI `git add` gap confirmed:** `git add .aib_brain logs` found in "Commit and push" step. `versions/` is tracked. Fix: `git add .aib_brain logs versions`.

7. **README.md gaps:** (a) "Typical Daily Flow" step 2 says "Open a request" manually, contradicting ADR-0005. (b) "No changes" toggle not documented. (c) Use case scenarios 1 and 2 still show manual open-request step.

8. **Concepts.md gaps:** (a) Invocation contract has no "No changes" toggle entry. (b) `input.md` lifecycle in the Holistic workflow section does not reflect the corrected reset-last behavior.

9. **Standard analysis flow missing `input.md` reset:** The standard flow (Active request, no toggles) has no reset step for `input.md`. After every standard run, `input.md` retains whatever the developer wrote — it is never cleared. The reset-last behavior (already being fixed for the Auto-Request Branch) must also be added to the standard flow, with the condition "unless triggered from `aib-implement.md`" to prevent double-reset.

10. **Auto-Request Branch enumerates 6 sections, convention requires 14:** Step 5 of the Auto-Request Creation Branch explicitly states 6 mandatory sections. `request-convention.md` states all 14 sections must be present. The 8 AI-generated sections (7–14) are absent from the enumeration, meaning auto-created `request.md` files may be missing them until (and if) analysis continues to step 8.

## External Benchmarking

**1. Explicit prohibition patterns in LLM agent prompts (industry pattern)**

- Open-source agent frameworks (LangGraph, AutoGen, CrewAI) address confirmation-seeking by adding explicit imperative statements at decision branch points: "Do NOT ask for user confirmation. Proceed autonomously." Relying on contextual inference is unreliable across models; a direct prohibition at the branch yields consistent behavior regardless of the underlying model.

- Key takeaway: Explicit imperative prohibition at the branch point is more robust than contextual framing. Adopted for Task 4 — one sentence added to `aib-implement.md`'s zero-Active-rows branch.

- Applicability: Adopted directly.

**2. Explicit `git add <path>` for generated CI artifacts (GitHub Actions community practice)**

- In GitHub Actions workflows that produce generated files during CI (changelogs, lockfiles, release archives), the predominant pattern is to stage output directories explicitly in `git add` rather than using `git add .`. This avoids accidentally staging unintended artifacts while ensuring all intended outputs are committed. Projects using `zipfile` or similar to produce release archives consistently add the archive output directory to the explicit stage list.

- Key takeaway: `git add .aib_brain logs versions` is safer than `git add .` (overly broad) and more complete than the current incomplete list. Explicit staging is the idiomatic practice.

- Applicability: Adopted directly for Task 5.
