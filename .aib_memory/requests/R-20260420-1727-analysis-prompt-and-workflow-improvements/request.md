## Goal

Fix a bug where `input.md` retains the old active request reference after a request is closed via `aib-implement.md`, and deliver three workflow/usability improvements to the analysis prompt and related files: (1) add scale-direction labels to the Question threshold option row in `input.md`, (2) add a recommended-answer marker to Q-blocks, and (3) add an ambiguity-detection instruction to the analysis prompt.

## Background

Four issues were reported via `input.md`:

1. **Bug — stale `input.md` after implement**: After `aib-implement.md` successfully closes a request via `close-request.py`, the `## Active request` line in `input.md` still shows the now-Closed request ID. The reset that is expected (set to "No active request") does not happen because neither `aib-implement.md` nor `close-request.py` performs it. `aib-analysis.md` correctly resets `input.md` as its final step, but `aib-implement.md` has no equivalent step.

2. **Enhancement — Question threshold scale labels**: The threshold row `- Question threshold: [ ] 1  [ ] 2  [x] 3  [ ] 4  [ ] 5` gives no indication of the scale's direction. A user unfamiliar with the scale must consult documentation to know whether 1 means "raise all questions" or "raise none". Adding short labels such as `1 (all)` and `5 (mandatory only)` resolves this at a glance.

3. **Enhancement — Recommended answer marker in Q-blocks**: Q-blocks currently provide options without indicating which the AI considers most appropriate. Adding a recommended marker to the preferred option improves decision guidance for the developer.

4. **Enhancement — Ambiguity detection instruction**: The analysis prompt does not currently instruct the AI to actively identify ambiguous or underspecified parts of the request that could lead to materially different implementation outcomes. Adding an explicit ambiguity-detection instruction will improve analysis quality.

## Scope

- Fix the stale-`input.md` bug: after `aib-implement.md` successfully closes a request, reset `input.md` to the seed template with "No active request" (decision on exact location deferred to Q001).

- Update the Question threshold option row in the `input.md` seed string within `initialize.py` to add scale-direction labels: `[ ] 1 (all)` and `[ ] 5 (mandatory only)` (values 2, 3, 4 receive no label).

- Update the same seed string in `aib-analysis.md` (used when the prompt resets `input.md`) to match.

- Update the Q-block format in `aib-analysis.md` and `request-convention.md` to include a recommended-answer marker on one option per Q-block (format confirmed by Q002).

- Add an ambiguity-detection instruction to the `## Questions & Decisions` generation section of `aib-analysis.md`.

- Update `context.md` to reflect the above changes.

## Out of scope

- Changing the Question threshold default value (currently `[x] 3`).

- Redesigning the Q-block schema beyond adding a recommended marker.

- Adding scale-direction labels for threshold values 2, 3, and 4.

- Modifying any test files or CI workflows beyond what is strictly required by the above scope.

- Changing `close-request.py` logic beyond what is needed for the chosen Q001 option.

## Constraints

- `.aib_brain/` files MUST NOT be modified by tool scripts; they are edited directly in this implementation.

- All tool scripts must remain Python 3.10+ standard-library only.

- The seed template in `initialize.py` and the reset template in `aib-analysis.md` must remain consistent after this change.

- `aib-implement.md` MUST NOT reset `input.md` before implementation is confirmed complete and `close-request.py` has run successfully.

- The Q-block format change must be backward-compatible: existing answered Q-blocks in closed requests need not be retrofitted.

## Success criteria

- SC-01: After running `aib-implement.md` to completion, `input.md` `## Active request` line reads "No active request", not the old closed request ID.

- SC-02: A freshly seeded `input.md` (via `initialize.py`) displays the threshold row as: `- Question threshold: [ ] 1 (all)  [ ] 2  [x] 3  [ ] 4  [ ] 5 (mandatory only)`.

- SC-03: After `aib-analysis.md` resets `input.md`, the threshold row contains the scale labels (same format as SC-02 with the checked value replaced accordingly).

- SC-04: Q-blocks generated by `aib-analysis.md` include a recommended-answer marker on exactly one of the options per Q-block.

- SC-05: `aib-analysis.md` generates at least one ambiguity-targeting Q-block for any request that contains underspecified parts with materially different implementation interpretations.

- SC-06: The Q-block format definition in `request-convention.md` matches the updated format in `aib-analysis.md`.

## Assumptions

- A1: The scale-direction labels for the threshold row use exactly the user-specified format: `[ ] 1 (all)` and `[ ] 5 (mandatory only)`, with values 2, 3, and 4 unlabeled.
  - Risk if false: If the user expects all values labeled or different label text, the implementation requires rework of `initialize.py` and `aib-analysis.md` seed strings.

- A2: The ambiguity-detection instruction is inserted inside the `## Questions & Decisions` generation block in `aib-analysis.md`, before the threshold-check logic (so identified ambiguities are subject to severity rating and threshold filtering).
  - Risk if false: If placement is after the threshold check, ambiguities below the threshold would be silently dropped without any record.

- A3: A recommended-answer marker applies to exactly one option per Q-block; Q-blocks where no option is clearly preferable may omit the marker.
  - Risk if false: Requiring a marker on every Q-block regardless of preference clarity would produce forced/arbitrary recommendations that mislead the developer.

- A4: The input.md reset string in `aib-analysis.md` appears in multiple inline reset instructions (standard flow final step, Auto-Request Creation Branch step 8, toggle handling). All occurrences must be updated.
  - Risk if false: If only one occurrence is updated, some code paths will reset with the old threshold row format (no labels), creating inconsistency.
