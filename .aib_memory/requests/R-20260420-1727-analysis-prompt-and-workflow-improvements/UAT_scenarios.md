## UAT-01 — Verify input.md reset after implement completes (SC-01)

**Objective:** Confirm that after a full `aib-implement.md` run that successfully closes a request, `input.md ## Active request` reads "No active request".

**Preconditions:**
- Exactly one Active request exists in `requests_register.md`.
- `input.md ## Active request` shows the Active request ID.

**Steps:**
1. Execute `aib-implement.md` in the AI coding interface on the active request.
2. Confirm the prompt completes all implementation steps, runs `close-request.py`, and outputs "--- I am done with the implementation ---".
3. Open `.aib_memory/input.md`.

**Expected outcome:**
- `## Active request` line reads: `No active request`
- `## Options` block is reset to the seed defaults (all toggles unchecked, `[x] 3` as the threshold).
- `## Input` section is empty.

**Pass/Fail:** Pass if all three conditions hold; Fail if `## Active request` still shows the old request ID.

## UAT-02 — Verify recommended marker appears in live Q-blocks (SC-04)

**Objective:** Confirm that Q-blocks generated by `aib-analysis.md` include a recommended-answer marker on exactly one option.

**Preconditions:**
- A test request with at least one decision point at or above the threshold exists (or can be created).
- `aib-analysis.md` has been updated with the recommended-marker format.

**Steps:**
1. Create a test request that contains a known decision fork (e.g., choice of output format).
2. Execute `aib-analysis.md`.
3. Open the request's `request.md` and navigate to `## Questions & Decisions`.

**Expected outcome:**
- At least one Q-block is present.
- Exactly one option per Q-block carries the recommended-answer marker (e.g., `*(recommended)*`).
- No option is marked twice in the same Q-block.

**Pass/Fail:** Pass if all conditions hold.

## UAT-03 — Verify ambiguity-detection Q-block generation (SC-05)

**Objective:** Confirm that `aib-analysis.md` generates at least one Q-block targeting an ambiguity when the request contains underspecified parts with materially different interpretation paths.

**Preconditions:**
- A test request with intentionally ambiguous language exists (e.g., "update the output format" without specifying which format or where the output is used).
- `aib-analysis.md` has been updated with the ambiguity-detection instruction.

**Steps:**
1. Create a test request with at least one ambiguous statement that has two or more plausible, mutually exclusive interpretations.
2. Execute `aib-analysis.md`.
3. Open the request's `request.md` and navigate to `## Questions & Decisions`.

**Expected outcome:**
- At least one Q-block is present whose question text explicitly references the ambiguous part of the request.
- The options in the Q-block represent mutually exclusive, realistic interpretations (not variations on the same interpretation).

**Pass/Fail:** Pass if at least one qualifying Q-block is present.
