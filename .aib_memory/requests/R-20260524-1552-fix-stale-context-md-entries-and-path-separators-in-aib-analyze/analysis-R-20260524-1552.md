## Overview

### Background

A consistency audit of the workspace (attachment `improvements_suggestions_claude-sonnet-4.6_20260524.md`) identified several stale or incorrect entries in `.aib_memory/context.md`, one incorrect test method in `tests/test_analysis_prompt_structure.py`, and mixed backslash/forward-slash path separators in `.aib_brain/prompts/aib-analyze.md`. These issues cause agents reading `context.md` to form an incorrect mental model of the analysis workflow step order, GC constraint identifiers, Appendix A location, S07 structure, and ADR-003 architectural record. The test mismatch causes a false assurance failure where a test claims to verify the "no-closed-request-reads" constraint but actually checks an unrelated GC identifier.

### Scope

The following targeted corrections are in scope:

- `.aib_memory/context.md` — correct the step order in "Analysis workflow structure" bullet (steps 6 and 7 were swapped); correct the GC constraint range (remove false "GC-03 was removed" and "GC-04 through GC-07" claims); correct the Appendix A location description; remove stale section 5.6 sub-section reference from "Analysis Q-block rules" bullet and replace with accurate S07 description; update ADR-003 to reflect that Q-block rules are now in `q-block-convention.md`; split the long run-on "Analysis workflow structure" bullet per context-convention.md Rule 16.
- `tests/test_analysis_prompt_structure.py` — rename `test_gc06_no_closed_request_reads_present` to `test_gc04_no_closed_request_reads_present` and update the docstring and assertion to check for `GC-04`.
- `.aib_brain/prompts/aib-analyze.md` — replace all backslash path separators with forward slashes in S05.2, S08.2, and S09.1.

### Out of scope

- Fixing other findings from the consistency audit (F-001 through F-018) that are not explicitly listed in the input request.
- Updating `aib-refresh-context.md` to prevent future regeneration of stale content (this is a separate request).
- Any changes to the GC constraint definitions themselves (e.g., adding GC-06/GC-07).
- Fixing the GC-05 stale cross-reference (F-002) or the aib-implement.md typos (F-006).

## Input Interpretation

The developer requests a set of targeted corrections to three files to eliminate documentation inaccuracies and code quality issues identified by an external consistency audit. The corrections are maintenance-only changes with no behavioral impact on AIB workflows:

1. **context.md step order correction** — The "Analysis workflow structure" Domain Knowledge bullet in `.aib_memory/context.md` lists step 6 as "Archive Input and Reset" and step 7 as "Quality Check". The actual `aib-analyze.md` prompt defines S06 as Quality Check and S07 as Archive Input and Reset. The bullet must be corrected to match the prompt's actual execution sequence.

2. **context.md GC range correction** — The same bullet claims "GC-03 was removed" and lists "GC-04 through GC-07". In reality, `aib-analyze.md` defines GC-01 through GC-05 with no gap; GC-03 exists ("No partial writes on halt"); GC-06 and GC-07 do not exist. The bullet must be corrected to state "GC-01 through GC-05".

3. **context.md Appendix A location correction** — The same bullet states Appendix A is "after `## 7. Completion Confirmation`", a heading that no longer exists. Appendix A appears after the `## 5. Execution Procedure` section (after step S10). The reference must be updated.

4. **context.md S07 description correction** — The "Analysis Q-block rules" bullet describes Archive Input and Reset using stale section numbering (5.6, 5.6.1, 5.6.2, 5.6.3) from a prior structural layout. The current prompt has only S07.1 (invoke finalize-input.py) and S07.2 (emit step-completion note). The stale sub-section references must be removed and replaced with an accurate description.

5. **context.md ADR-003 correction** — ADR-003 records the decision to co-locate Q-block rules in `aib-analyze.md` section 6.3. This decision was subsequently reversed: Q-block format rules were extracted to `.aib_brain/conventions/q-block-convention.md`. ADR-003 must be updated to reflect the current architecture.

6. **context.md Rule 16 compliance** — The "Analysis workflow structure" bullet is a single entry spanning more than 250 words across multiple sentences, violating context-convention.md Rule 16 (max two sentences per bullet). It must be split into multiple bullets each covering one concern.

7. **test method rename** — `test_gc06_no_closed_request_reads_present` in `tests/test_analysis_prompt_structure.py` claims to verify the "no-closed-request-reads" constraint but asserts the presence of the string `"GC-06"`. GC-04 is the actual "No closed-request reads" constraint in `aib-analyze.md`. The test method name, docstring, and assertion must be updated to reference `GC-04`.

8. **aib-analyze.md path separator fix** — Steps S05.2, S08.2, and S09.1 in `aib-analyze.md` use Windows-style backslash path separators (`\`) inconsistently alongside forward-slash paths in the same steps. All backslash occurrences in those three steps must be replaced with forward slashes for cross-platform consistency.

## Decision Register

### Decision Points

**DP-001 — How to split the "Analysis workflow structure" bullet to comply with Rule 16**

- **Tag:** `resolve-autonomously`
- **Options:**
  - Option A: Split into 4 focused bullets: (1) step sequence, (2) prompt section structure + GC range, (3) Appendix A + Answer Sub-flow, (4) step-completion notes + State transitions.
  - Option B: Compress the content into a single 2-sentence bullet at the cost of information loss.
- **Resolved by:** Option A — context-convention.md Rule 16 requires splitting when two sentences are insufficient; all four topic areas are independently meaningful and currently referenced by context-consumers. No information should be discarded; splitting is the mandated approach. The four-bullet structure maps cleanly to four distinct sub-topics already present in the original bullet.
- **Source citation:** `.aib_brain/conventions/context-convention.md` Rule 16: "it MUST be split into two or more separate bullet items rather than combined into a single long entry."

**DP-002 — What reference to use for the Q-block rules in the updated ADR-003**

- **Tag:** `resolve-autonomously`
- **Options:**
  - Option A: Reference `aib-analyze.md` step identifier `S08.2` (current prompt step naming scheme).
  - Option B: Reference the old section number `6.3` (stale; no longer accurate).
- **Resolved by:** Option A — the current `aib-analyze.md` uses step identifiers (S01–S10), not section numbers. `S08.2` is the step that references `q-block-convention.md`. Using `S08.2` is consistent with the prompt's naming conventions.
- **Source citation:** `.aib_brain/prompts/aib-analyze.md` S08.2: "For every Decision Point tagged `ask`, generate one Q-block following the instructions in `.aib_brain\conventions\q-block-convention.md`."

**DP-003 — Whether to update the "Analysis Q-block rules" bullet S07 stale text in-place or remove the sentence entirely**

- **Tag:** `resolve-autonomously`
- **Options:**
  - Option A: Replace the stale "Step 6 (Archive Input and Reset, section 5.6)..." sentence in the "Analysis Q-block rules" bullet with an accurate S07 description.
  - Option B: Remove the sentence entirely (it is out of place in the Q-block rules bullet, which should describe Q-block behavior, not archive behavior).
- **Resolved by:** Option B — the "Analysis Q-block rules" bullet concerns Q-block rules; an accurate S07 description belongs in the "Analysis workflow structure" bullet group (DP-001). Retaining S07 content in the Q-block bullet violates separation of concerns. The "Analysis workflow structure" split bullets (DP-001, Option A) will include an accurate S07 description.
- **Source citation:** Domain Knowledge bullet named "Analysis Q-block rules" in `.aib_memory/context.md` is titled for Q-block rules specifically; embedding an archive-step description in it conflates distinct topics.
