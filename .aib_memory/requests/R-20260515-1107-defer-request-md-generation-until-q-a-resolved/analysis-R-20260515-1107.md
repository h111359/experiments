## Input Interpretation

The developer's request addresses a sequencing problem in the Auto-Request Creation Branch of `aib-analysis.md`. When that branch runs for the first time with non-empty `input.md`, it currently writes `request-<request_id>.md` (all 10 sections, with sections 1–6 derived from the developer's raw input text) and generates Q-blocks in the same pass. This means the AI fixes its interpretation of the developer's intent before the developer has had a chance to confirm or redirect it through Q&A — a form of anchoring bias.

The developer's confirmed intent, as resolved through Q&A, is:

**Behaviour when Q-blocks are generated (first pass):** `request-<request_id>.md` MUST NOT be written at all. The register entry and request folder are created by `create-request.py` as before. The analysis document IS generated in this first pass (including the new `## Input Interpretation` section). Q-blocks are written to `input.md`. No request file exists at `.aib_memory/request-<request_id>.md` after the first pass.

**Behaviour on re-run (after Q&A is resolved):** The Answer Application Sub-flow detects that `request-<request_id>.md` is absent and creates it for the first time, populating all 10 mandatory sections from: (1) the Q&A answers, (2) the `## Input Interpretation` section already present in `analysis-<request_id>.md`, and (3) the active request title from the register. This is the authoritative first write of the file.

**Behaviour when no Q-blocks are generated (first pass):** Unchanged from current behavior — `request-<request_id>.md` is written in the single pass as today.

**`## Input Interpretation` section:** A new mandatory section added to `analysis-<request_id>.md` (placed between `## Files Read During This Analysis Run` and `## Research Results`). It contains an AI-rewritten, terminology-correct interpretation of the developer's original `## Input` content, enriched by `context.md` product knowledge and relevant external knowledge. Its primary purpose is to provide the Answer Application Sub-flow on re-run with a faithful, structured interpretation of developer intent without requiring access to the archived `input.md` (which is prohibited by GC-01).

**Preflight step 3 adaptation:** Step 3 currently halts (GC-04) when `request.md` is absent. It must be modified to gracefully handle the deferred-creation state: if `request.md` is absent but `input.md` contains a `## Questions` section, execution continues (the file will be created by the Answer Application Sub-flow). If `request.md` is absent and there is no `## Questions` section, GC-04 halt applies as before.

**`analysis-convention.md` change:** The mandatory section list grows from 4 to 5. The new section `## Input Interpretation` is inserted as section 3, between `## Files Read During This Analysis Run` (section 2) and `## Research Results` (now section 4). `## Implementation Alternatives` becomes section 5.

### Industry Knowledge

**Finding 1 — Lazy initialization and two-phase commit for specification artifacts (IEEE 29148 / BABOK v3)**

IEEE 29148 recommends that requirements documents be baselined only after stakeholder review and ambiguity resolution. BABOK v3 similarly emphasizes that elicitation must precede documentation. The first analysis pass in AIB is the elicitation phase (Q-blocks are elicitation questions); the re-run is the confirmation phase. Writing `request.md` only on re-run after Q&A directly implements the "define before commit" pattern. Applicability: high — this request aligns AIB with established requirements-engineering best practice.

**Finding 2 — Anchoring bias mitigation in AI-assisted workflows (prompt engineering literature)**

Research on large language model behavior (Wei et al., "Chain-of-Thought Prompting", 2022; various prompt engineering studies on self-consistency) documents that LLMs anchor subsequent reasoning to previously produced artifacts. Writing `request.md` in the first pass creates an anchor that the Answer Application Sub-flow then works around rather than replacing. Deferring the write forces the re-run to reason fresh from confirmed Q&A answers and the `## Input Interpretation` section — a structurally cleaner input set. Applicability: directly applicable; the `## Input Interpretation` section serves as a neutral, verified foundation for re-run reasoning.

### Workspace Findings

**Impacted file:** `.aib_brain/prompts/aib-analysis.md` — specifically section 4.7 (Auto-Request Creation Branch), which currently writes all 10 mandatory sections including sections 1–6 from `input.md` content in the same pass as Q-block generation.

**Current behavior (Auto-Request Creation Branch):**
1. `create-request.py` creates the folder and register entry.
2. `request-<request_id>.md` is written with all sections 1–6 fully populated from `input.md ## Input`.
3. `finalize-input.py` archives `input.md` and resets it.
4. Standard flow resumes at step 6; analysis is generated; Q-blocks are written to `input.md`.
5. On re-run (Answer Application Sub-flow): Q&A answers are applied to sections of the already-populated `request.md`.

**Problem:** In step 4, Q-blocks may reveal that the intent in sections 1–6 (as assumed by the AI from raw input.md text) was incorrect. But sections 1–6 have already been written, anchoring the AI's interpretation. The Answer Application Sub-flow in step 5 only applies corrections for answered Q-blocks; it does not regenerate sections 1–6 from scratch.

**Test impact:** `tests/test_analysis_prompt_structure.py` contains assertions about the auto-request creation branch (class `TestAutoRequestCreationBranch`, asserting the 10-section schema reference). Any prompt language changes to section 4.7 that alter how and when sections 1–6 are written will need corresponding test updates.

**GC-01 constraint interaction:** `aib-analysis.md` GC-01 states: "`inputs/input-archive-*.md` files in request folders MUST NOT be read." This means on the re-run (Answer Application Sub-flow), the AI cannot read the archived `input.md` content to reconstruct sections 1–6. The only available material on re-run is the stub `request-<request_id>.md` content and the Q&A answers. This creates a design gap: if the stub contains no original input content, sections 1–6 must be derived solely from Q&A answers and request title — which may be insufficient for a complete Goal/Background/Scope.

**Prior solution pattern:** The current `## Plan` deferral rule (section 7.2) follows the same principle: when Q-blocks exist in a first pass, the Plan is replaced with a stub. This is the closest existing analogue for the requested behaviour change, and the pattern is already well-tested.

## Implementation Alternatives

### Alternative A — Full stub (no preserved input content)

Write a 10-heading stub with all section bodies empty. On re-run, sections 1–6 are generated from Q&A answers + request title + context.md only. The original `## Input` content is effectively discarded for section generation purposes.

- **Trade-offs:**
  - Pro: Simple stub; no special non-mandatory section needed.
  - Con: Sections 1–6 on re-run are derived only from Q&A answers, which may be narrower in scope than the original input. Background and Scope may be less complete.
- **Codebase impact:** Section 4.7 simplified; Answer Application Sub-flow (section 4.8) extended to generate sections 1–6 from scratch.

---

### Alternative B — Preserved-input stub

Write a 10-heading stub where a special non-mandatory section (e.g., `## _Input Snapshot` or similar) stores the verbatim original `## Input` content. On re-run, sections 1–6 are generated from the preserved input + Q&A answers. The special section is removed after sections 1–6 are written.

- **Trade-offs:**
  - Pro: Re-run has full original context; sections 1–6 are as complete as the current flow.
  - Con: Adds a non-standard section to request.md (even temporarily). The 10-section constraint in `request-convention.md` must accommodate this. Adds complexity to the Answer Application Sub-flow (must detect and remove the special section).
- **Codebase impact:** Section 4.7 must write the special section; Answer Application Sub-flow must read and remove it.

---

### Alternative C — No stub; request.md file absent until Q&A resolved

Do not write any `request-<request_id>.md` file in the first pass. The folder and register entry exist, but no request file. On re-run, request.md is created for the first time.

- **Trade-offs:**
  - Pro: Maximum clarity — request.md only exists when requirements are confirmed.
  - Con: Menu state detection (`request_incomplete` state) and other tooling that reads request.md may fail or produce incorrect guidance when the file is absent. `validate_request_md` in tool scripts may need to handle absence gracefully. Risk of breaking existing flows that assume request.md exists while the request is Active.
- **Codebase impact:** Most disruptive; tool scripts and menu state logic may need changes. This likely exceeds the stated constraint of not changing tooling scripts.
