# Analysis: R-20260524-1303

## Overview

**Request ID:** R-20260524-1303  
**Request title:** Rename Active request section and add status tracking

### Background

The developer requests two independent but related improvements to the AIB workflow:

1. The `## Active request` section heading in `.aib_memory/input.md` should be renamed to `## Status`, and a new state-tracking line should be added underneath. The state line should be updated by `aib-analyze.md` during its run. Allowed state values are `idle`, `analysis_ready`, `implementation_ready`, `unknown`. The aim is that a single read of `input.md` gives a clear indication of the current workflow state and what the next step is.

2. The `aib-analyze.md` prompt should emit a short step-completion note after each of its numbered steps, so the user can follow progress and see each step's outcome in the conversation output.

### Scope

- Rename the `## Active request` heading in the `input.md` seed template across all tooling that generates or resets `input.md`:
  - `.aib_brain/tools/finalize-input.py` (`_SEED_TEMPLATE` constant + `_is_stub_equivalent` regex)
  - `.aib_brain/tools/close-request.py` (`input_seed` string)
  - `.aib_brain/tools/initialize.py` (seed template string)
- Add a `State: <value>` line to the `## Status` section in the seed templates and update the tools to inject the correct state value at reset time.
- Update `aib-analyze.md` to: (a) inject or set the state line at appropriate points in its execution flow; (b) emit a short step-completion note after each of steps S01–S10.
- Update all tests that reference the `## Active request` heading or the seed template constants.

### Out of scope

- Changes to `menu.py` display label `"Active request: {req_text}\n"` — this is a display string in menu output, not the section heading in `input.md`, and the developer did not request menu output changes.
- Changes to `user_guide.html` beyond what is strictly needed to reflect the renamed heading (no end-user visible behavioral change is introduced that requires guide rewriting).
- Changes to archived input files under `.aib_memory/requests/*/inputs/input-archive-*.md` or `.aib_memory/archives/` — GC-04 and GC-01 prohibit modifying closed-request artifacts.
- Introducing new state values beyond the four specified (`idle`, `analysis_ready`, `implementation_ready`, `unknown`).
- Changes to the `aib-implement.md` prompt (the `implementation_ready` state is reserved for future use by the implement prompt, not in scope for this request).

## Input Interpretation

The developer requests two independent improvements to the AIB workflow:

**Improvement 1 — Section rename and state tracking:**  
The `## Active request` section in `.aib_memory/input.md` is to be renamed `## Status`. A second line is to be added under this section that records the current workflow state as a machine-readable key-value. The state is updated by `aib-analyze.md` at appropriate points during execution. Allowed values are: `idle` (no active work), `analysis_ready` (plan generated, implementation can begin), `implementation_ready` (reserved for post-implement), `unknown` (Q-blocks pending or indeterminate state). Because `finalize-input.py`, `close-request.py`, and `initialize.py` all write or reset `input.md` using embedded seed template strings, those strings must also be updated. Tests that assert on the heading text or seed template structure must be updated accordingly.

**Improvement 2 — Per-step output notes in `aib-analyze.md`:**  
After each numbered step (S01 through S10) in the execution procedure, a sub-instruction must be added requiring the AI agent to emit a short step-completion note to the conversation output. The note confirms the step is done and summarizes the outcome. This is a prompt-file-only change; no runtime artifacts or tool scripts are modified.

## Research Results

### Workspace Pattern-Scan Findings

1. **`## Active request` heading is embedded as a string literal in four Python files:**
   - `finalize-input.py` line 35: `"## Active request\n"` (seed template)
   - `finalize-input.py` line 107: regex `r"(## Active request\n)[^\n]*"` (stub-equivalence check)
   - `close-request.py` line 116: `"## Active request\n"` (reset seed)
   - `initialize.py` line 82: `"## Active request\n"` (initialization seed)
   All four must be updated to `## Status`.

2. **The `_is_stub_equivalent` function in `finalize-input.py`** compares the normalized content of `input.md` against the seed template, replacing the first line after `## Active request` with a placeholder to ignore the variable request-ID line. If a `State: <value>` line is added as a second line under `## Status`, the existing regex `r"(## Active request\n)[^\n]*"` only neutralizes one line. The state value line would then participate in the comparison, causing stub-equivalence failures when the state differs between a freshly seeded `input.md` (with `State: idle`) and a post-analysis reset (with `State: analysis_ready`). The regex must be updated to also neutralize the state line.

3. **Tests referencing `## Active request`:**
   - `test_finalize_input.py` lines 28, 37: `_SEED_TEMPLATE` and `_NON_STUB_INPUT` constants include `"## Active request\n"`.
   - `test_close_request.py` line 117: constructs mock input.md with `"## Active request\nR-… Title"`.
   - `test_initialize.py` line 277: asserts `"## Active request" in content`.
   - `test_menu.py` lines 202, 209: `_INPUT_MD_SEED` and `_INPUT_MD_WITH_CONTENT` constants include `"## Active request\n"`.
   All must be updated to use `## Status`.

4. **`menu.py` is not affected** by the heading rename. Its string `"Active request: {req_text}\n"` at line 667 is a display label in menu output, derived from `state.active_request_id`, not read from the heading in `input.md`. No change required.

5. **`user_guide.html`** references "active request" in prose but does not embed the Markdown heading `## Active request` as a code literal. The state values `idle`, `analysis_ready`, etc. are new to `input.md` and are separate from the menu-level guidance states already documented in the guide (`idle`, `input_ready`, `questions_pending`, etc.). The user guide may benefit from a note about the new `State:` line but does not require structural changes.

6. **Step-note scope in `aib-analyze.md`:** Steps S01–S10 are defined in section `## 5. Execution Procedure`. Each step currently ends without any output instruction. Adding a step-completion output sub-step is purely additive and does not conflict with any existing rule. The Appendix A and Appendix B sections have no numbered steps (they are branch descriptions), so they are not in scope for the note additions.

### Industry Knowledge

1. **State-machine patterns in developer workflow tools** (based on established CI/CD tooling conventions, e.g., GitHub Actions job status, Makefile phased targets): Representing workflow state as a machine-readable key-value in a human-editable file is a proven pattern. Tools like `make` use target status files; GitHub Actions writes `job.status` to environment variables. AIB's approach of a `State: <value>` line in `input.md` aligns with this convention. Applicability: high — the pattern is simple, deterministic, and survives partial execution.

2. **Structured logging / step-annotation in AI agent prompts** (from OpenAI function-calling conventions and LangChain agent logging best practices): Requiring the AI agent to emit step-completion notes after each step creates an implicit structured log visible in the conversation. This is consistent with the "chain-of-thought with explicit checkpoints" pattern recommended for multi-step agents to aid debugging and user trust. Applicability: high — the aib-analyze.md prompt is a 9-step linear workflow, making per-step notes directly useful.

3. **Monotonic state progression in workflow engines** (from BPMN and state-machine design literature): State values should represent observable, deterministic milestones rather than transient in-progress states. The four proposed values (`idle`, `analysis_ready`, `implementation_ready`, `unknown`) follow this principle. `unknown` functions as a catch-all for indeterminate states (unanswered Q-blocks), which is a common pattern in workflow engines (e.g., `PENDING` or `BLOCKED` states). Applicability: medium — the AIB state model is simpler than BPMN, but the monotonic progression principle supports the proposed value set.

## Decision Register

### Decision Points

#### Decision Point: State line injection mechanism

**Where it applies:** Step S07 of `aib-analyze.md` (finalize-input.py invocation) and Step S08 (Q-block generation path).

**Why alternatives existed:** After `finalize-input.py` resets `input.md`, the state line must reflect whether the run completed with a plan (`analysis_ready`) or halted with pending questions (`unknown`).

- **Option A — Hardcode state in seed templates; prompt overwrites in S08** *(chosen)*
  - Description: `finalize-input.py`, `close-request.py`, and `initialize.py` each hardcode their respective default state in the seed template (`analysis_ready` for finalize, `idle` for close/initialize). When S08 generates Q-blocks, `aib-analyze.md` writes a direct edit to `input.md` to change `State: analysis_ready` → `State: unknown`.
  - Benefits: No tool CLI interface change; minimal new logic in finalize-input.py; state defaults are self-documenting in the seed strings.
  - Drawbacks: Requires the analyze prompt to perform a direct in-place edit of `input.md`; creates implicit coupling between the hardcoded default and the prompt's override logic.
  - Codebase impact: Update seed strings in three tool files; add one prompt sub-step in S08.

- **Tag:** `resolved-by-user`
- **Resolution outcome:** Option A is implemented. Seed templates hardcode per-tool state defaults; `aib-analyze.md` S08 directly edits `input.md` to override state to `unknown` when Q-blocks are generated.

#### Decision Point: Step-note instruction format

**Where it applies:** End of each step S01–S10 in `aib-analyze.md ## 5. Execution Procedure`.

**Why alternatives exist:** The developer requested that the AI "write in the output short note that the step is executed and the outcome from it" but did not specify the format of the instruction added to the prompt or the format of the note emitted to the user.

**Alternatives:**

- **Option A — Fixed-format instruction with tag**
  - Description: Add a sub-step at the end of each step: `S0N.X. Output the literal step tag followed by a one-sentence outcome summary, e.g., \`[S0N done] <outcome>.\``
  - Benefits: Deterministic format; easy to scan in conversation output; consistent across all steps.
  - Drawbacks: The `[S0N done]` tag notation is not established elsewhere in the workspace.
  - Codebase impact: Add one sub-step per step (10 additions to aib-analyze.md).

- **Option B — Free-form one-sentence note**
  - Description: Add a sub-step: `S0N.X. Output a one-sentence note confirming step completion and summarizing its outcome.`
  - Benefits: Simpler instruction; allows natural-language variation.
  - Drawbacks: No consistent format; harder to grep or programmatically parse if needed.
  - Codebase impact: Same as Option A.

- **Tag:** `resolve-autonomously` — The developer's instruction "write in the output short note that the step is executed and the outcome from it" is unambiguous about what content is required. Option A (fixed tag format) is chosen: it satisfies the request and produces scannable output without adding unnecessary verbosity. The tag `[S0N]` reuses the step identifiers already present in the prompt, maintaining consistency with existing notation.
- **Rationale:** Source: `.aib_memory/input.md ## Input` — "after each step the AI agent to write in the output short note that the step is executed and the outcome from it." The step identifiers (S01, S02, etc.) already exist in the prompt and provide a natural anchor for the tag.
- **Resolution outcome:** Each step gets a final sub-step: `Output a short step-completion note in format: \`[S0N done] <one-sentence outcome summary>.\``
