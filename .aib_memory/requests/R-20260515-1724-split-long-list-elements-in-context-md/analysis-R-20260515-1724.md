## Input Interpretation

Request R-20260515-1724 originally scoped a structural compliance repair of `.aib_memory/context.md`: every bullet item exceeding two sentences was to be split into two or more shorter bullets, with original content preserved verbatim and all other formatting rules (Rule 12: blank line between top-level list items; Rule 14: grammatically complete sentences ending with a period) maintained. The developer's updated input identifies a root-cause gap: `.aib_brain/prompts/aib-context.md` does not enforce Rule 16 as an explicit validation gate during generation. Because `context.md` is fully replaced on each `aib-context.md` execution, any surgical splits applied in this request would be overwritten by the next regeneration run, restoring all violations. The developer requests that `aib-context.md` be fixed so that Rule 16 compliance is verified and corrected before the file is written to disk, making the enforcement durable across future regenerations.

## Research Results

### Workspace Findings

Rule 16 is defined in `.aib_brain/conventions/context-convention.md` (Formatting Rules, item 16): "Each bullet or list item MUST NOT exceed two sentences. If the content of an entry requires more than two sentences to express, it MUST be split into two or more separate bullet items rather than combined into a single long entry."

`.aib_brain/prompts/aib-context.md` Phase 4 already contains the following formatting requirement: "Each bullet or list item MUST NOT exceed two sentences. If an entry requires more than two sentences, split it into two or more separate bullet items." This rule is embedded as a passive inline constraint within synthesis guidance — there is no dedicated verification step or explicit checkpoint that runs after synthesis and before the write action in Phase 5.

The active plan's Task 1 Risk Notes contain a complete violations catalogue: 29 specific Rule 16 violations across five sections of `context.md` (Requirements Summary — Functional Requirements, Acceptance criteria, Architecture — Component Map, Technical Design — Module Breakdown, Key Algorithms, Workspace File Inventory). Each violation is listed with its section, item identifier, sentence count, and split target count.

`.aib_memory/context.md`'s Technical Design section contains the densest violations — the `aib-analysis.md`, `menu.py`, `initialize.py`, and FR-series entries all exceed two sentences, having been generated by prior `aib-context.md` runs without a post-synthesis compliance check.

**Additional out-of-scope discrepancies identified during audit (for future requests):**

- Duplicate entry for `.aib_brain/conventions/requirements-analysis-convention.md` in the Workspace File Inventory (two separate bullets covering the same file path).

- Stale artifact path names in Data Architecture — Data Storage: `.aib_memory/request-<request_id>.md` should be `.aib_memory/plan-<request_id>.md`; `.aib_memory/analysis.md` should be `.aib_memory/analysis-<request_id>.md`; `.aib_memory/UAT_scenarios.md` should be `.aib_memory/UAT_scenarios-<request_id>.md`.

- Workspace File Inventory entry for `aib-context.md` is missing the `.aib_brain/prompts/` path prefix.

These discrepancies are noted as findings and MUST NOT be fixed within this request.

### Industry Knowledge

**Finding 1 — Two-pass generate-then-validate pattern (IEEE 29148, Software Requirements Engineering):** Requirements engineering frameworks distinguish between the act of drafting requirements and the act of validating them against quality criteria. Applied to AI prompt design, separating the synthesis phase from a distinct validation phase is the standard approach for ensuring constraint satisfaction when generation is complex. The applicable practice is: complete the synthesis pass first (Phase 4), then perform a dedicated Rule 16 compliance pass before persisting the output.

**Finding 2 — Pre-commit validation hooks pattern (software development practices):** Many development tools implement a discrete validation step that must pass before a state change is committed. Applied to prompt-driven document generation, adding an explicit pre-write validation step creates an auditable checkpoint and prevents non-compliant states from being persisted — more reliable than embedding a passive constraint within generation guidance.

### Feasibility Notes

Adding a mandatory Rule 16 verification step to `aib-context.md` is straightforward. Phase 5 currently begins with a direct write instruction; inserting a named pre-write verification step (as a new numbered step before the write) requires no restructuring of existing phases. The verification logic is well-defined: scan all generated bullet items, count terminal sentence-ending marks per item, split any item exceeding two sentences at sentence boundaries, and confirm zero violations before proceeding. No technical uncertainty exists.

### Expert Observations

The root cause of recurring Rule 16 violations is that the constraint is embedded as a passive inline formatting guideline within synthesis guidance, which AI models apply inconsistently when generating large volumes of structured content across many sections. A discrete, named verification pass creates a mandatory checkpoint that is distinct from generation, making compliance a verifiable outcome rather than a best-effort constraint. This scope extension — fixing `aib-context.md` in addition to splitting the current violations — is proportionate and necessary; without it, the context.md splits delivered in this request would be silently undone by the next `aib-context.md` run.

## Implementation Alternatives

**Alternative A — Dedicated pre-write verification step in Phase 5 of `aib-context.md` (selected)**

A new numbered step is inserted at the beginning of Phase 5 of `aib-context.md`, before the write instruction. The step explicitly instructs the AI to scan all generated bullet items, count sentences per item, split any item exceeding two sentences at sentence boundaries, and confirm zero violations before writing. This is a discrete, auditable checkpoint structurally separated from the synthesis guidance in Phase 4.

- Trade-offs: Most reliable approach; creates an unambiguous obligation distinct from generation; adds minimal prompt text. Slightly increases prompt length.

- Codebase impact: Modifies only `.aib_brain/prompts/aib-context.md` Phase 5.

**Alternative B — Strengthen the inline rule text in Phase 4**

The existing Phase 4 formatting requirement is expanded with stronger wording, a counter-example, and a mandatory self-check instruction inline within the synthesis guidance.

- Trade-offs: Minimal structural change. Does not address the root cause — the existing inline rule is already present but has not prevented violations. Equally unreliable.

- Codebase impact: Modifies only `.aib_brain/prompts/aib-context.md` Phase 4.
