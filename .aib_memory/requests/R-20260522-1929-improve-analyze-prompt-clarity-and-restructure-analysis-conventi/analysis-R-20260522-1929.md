# Analysis: R-20260522-1929

## Overview

**Request ID:** R-20260522-1929

**Request title:** Improve analyze prompt clarity and restructure analysis convention

### Background

The developer identified multiple clarity and structural issues in the current `aib-analyze.md` prompt and `analysis-convention.md`. The prompt's execution sequence is not as clear as desired, structural content is mixed with behavioral rules in the prompt, Q-block formatting has unnecessary restrictions, and the "Implementation Alternatives" section in the convention needs to become a "Decision Register" with richer resolution semantics. Additionally, several prohibitions (external hyperlinks, embedded images) are no longer desired.

### Scope

- `.aib_brain/conventions/analysis-convention.md` — rename section 4.5 to "Decision Register" with new structure, remove section 5 (Maintenance Rules), remove determinism bullet from section 7, remove hyperlink/image restrictions from section 8, absorb structural content from the prompt's section 6.

- Changes to tool scripts (`create-request.py`, `finalize-input.py`, etc.).

## Input Interpretation

The developer requests a comprehensive restructuring of the AIB analysis prompt (`aib-analyze.md`) and its companion convention (`analysis-convention.md`) to achieve the following outcomes:

1. **Execution sequence clarity** — The prompt's execution procedure shall follow a clear linear sequence:
   - Pre-Read and State Resolution (including auto-request creation check).
   - Brownfield context check (execute `aib-refresh-context.md` if `context.md` is absent or empty).
   - Read `input.md`, attachments, and existing `analysis.md` if present.
   - Halt if unanswered questions exist in `input.md`.
   - Generate/regenerate `analysis.md` with best-effort research, internet research, expert knowledge. Record all human answers without losing previously provided answers unless overwritten by new ones.
   - Archive `input.md` content and reset to initial state.
   - Run quality check against `requirements-analysis-convention.md`.
   - Generate Q-blocks if quality check identifies gaps requiring human input; halt and inform user.
   - Generate/regenerate `plan.md` only from `analysis.md` — only when input is in initial state, analysis is complete, and quality check is 100% satisfied.

2. **Remove Context-Window Management** — Section 3.3 of the prompt shall be removed entirely; AI agent capabilities are trusted to manage context.

3. **Separation of concerns** — Structural/formatting content in the prompt's section 6 (Output Specifications) that describes document structure shall be moved to `analysis-convention.md`. The prompt shall retain only process, behavior, rules, and conditions.

4. **Remove link restriction** — The instruction "Do NOT embed external links. This section is for human review and auditability only — `implement` MUST NOT read or act on it." shall be removed from the Research Results section.

5. **Remove Q-block count limitation** — The "Soft limit: 9 Q-blocks per run" restriction shall be removed.

6. **Free-text question type** — A new question type shall be added for cases where no options can be offered by the AI agent. Multiple-choice questions remain preferred when possible. Free-text questions shall explain what information is needed, why, and its implementation impact.

7. **Remove `> Answer:` line** — The `> Answer:` row in Q-block format shall be removed. Only `- [ ] Other: ___` remains for freeform answers.

8. **Multiple options** — The Q-block format and documentation shall make clear that questions may have more than two proposed options (not just A and B as the current example implies).

9. **Decision Register** — Section 4.5 "Implementation Alternatives" in `analysis-convention.md` shall be renamed to "Decision Register" with the detailed structure provided by the developer: identification, alternatives with trade-offs, resolution classification (`resolve-autonomously` with source citation or `ask`), resolution outcome (chosen alternative only, discard others), and a `### Decision Points` summary.

10. **Remove Maintenance Rules** — Section 5 of `analysis-convention.md` shall be removed entirely.

11. **Remove prohibited content restrictions** — "External hyperlinks" and "Embedded images or diagrams" shall be removed from section 8 (Prohibited Content) of `analysis-convention.md`.

## Research Results

### Workspace Findings

- **Prompt structure (current):** `aib-analyze.md` has 8 top-level chapters. Section 6 (Output Specifications) currently contains both structural definitions and behavioral rules for each analysis section. Structural content includes heading formats, required sub-sections, and content schemas that properly belong in the convention file.

- **Convention structure (current):** `analysis-convention.md` has 8 sections. Section 4 (Mandatory Structure) already partially defines structure. Section 4.5 (Implementation Alternatives) defines the current decision-tracking approach with `ask`/`resolve-autonomously` tags. Section 5 (Maintenance Rules) contains idempotence, change drivers, and closure rules. Section 7 (Determinism Rules) contains a line about identical output from same inputs. Section 8 (Prohibited Content) lists 4 prohibited items including external hyperlinks and embedded images.

- **Q-block format (current):** The prompt defines Q-blocks with `Option A`, `Option B`, `- [ ] Other: ___`, and `> Answer:` lines. A "Soft limit: 9 Q-blocks per run" exists. The example implies exactly 2 options.

- **Cross-reference concern:** The prompt section 6.3.2 references `analysis-convention.md` section 7.3.2 for classification rules — but 7.3.2 does not exist in the convention file; the classification rules are in the prompt itself (section 6.3.2). This is a stale cross-reference in the convention file (section 4.5 says "per the classification rules in `aib-analyze.md` section 7.3.2" but the correct reference is section 6.3.2).

- **Test files:** `tests/test_analysis_prompt_structure.py` and `tests/test_requirements_analysis_convention.py` may need updates if they assert specific section numbers or content.

### Industry Knowledge

1. **Separation of specification and process (IEEE 29148):** IEEE 29148 advocates for clear separation between requirements specification (what to produce) and requirements process (how to produce it). Moving structural definitions to the convention file (specification) while keeping behavioral rules in the prompt (process) aligns with this principle. *Applicability:* Directly applicable — the developer's requested separation maps cleanly to this standard.

2. **Decision records in agile frameworks (ADR patterns):** Lightweight Architecture Decision Records (ADRs) commonly use a "Status: proposed/accepted/deprecated" lifecycle with clear "Context → Decision → Consequences" structure. The developer's requested "Decision Register" pattern enriches this with resolution classification (`ask` vs `resolve-autonomously`), which adds traceability. *Applicability:* The developer's design is more structured than typical ADRs, which is appropriate for AIB's automation-driven context.

3. **Requirements elicitation question design (BABOK):** Business analysis literature recommends that elicitation questions should be open-ended when the solution space is unknown and closed-choice when bounded options exist. The developer's dual question type (multiple-choice preferred, free-text when no options can be inferred) aligns with this best practice. *Applicability:* Directly applicable — supports the new free-text question type.

### AI Agent Critique

- The current prompt section 6 mixes structural definitions (what headings to produce, what content goes where) with behavioral rules (when to generate, how to classify decisions). This makes the prompt harder to maintain.

- The `> Answer:` line in the Q-block format is redundant with `- [ ] Other: ___` since both serve as freeform answer mechanisms.

- The "Soft limit: 9 Q-blocks per run" is artificial and may suppress genuinely needed questions.

- The stale cross-reference in `analysis-convention.md` section 4.5 (references "section 7.3.2" which doesn't exist) needs correction during the rename to Decision Register.

- Section 5 (Maintenance Rules) in the convention largely duplicates section 7 (Determinism Rules) — removing section 5 is appropriate.

- The prohibition on external hyperlinks and embedded images is overly restrictive for a reasoning artifact that humans read for auditability — removing it is reasonable.

- The current prompt's execution sequence interleaves context enrichment, amendment detection, and analysis generation in a way that's harder to follow than the developer's proposed linear sequence.

- The "Do NOT embed external links" instruction appears in two places: once in the analysis convention (section 4.4 Research Results rules) and once in the prompt (section 6.1). Both need removal.

- The convention section 7 (Determinism Rules) mentions creating Q-blocks in `plan-<request_id>.md` → `## Decisions`, which conflicts with the current prompt behavior of writing Q-blocks to `input.md ## Questions`. This inconsistency should be corrected.
