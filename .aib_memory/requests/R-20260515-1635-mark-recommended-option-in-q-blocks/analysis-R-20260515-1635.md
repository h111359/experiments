## Input Interpretation

The developer requests that when the AI Automation Agent generates Q-blocks and writes them to `.aib_memory/input.md ## Questions`, each Q-block must include a visual indicator on the option the AI considers recommended. Currently, the `aib-analysis.md` prompt explicitly prohibits any `*(recommended)*` marker on Q-block options, on the grounds that AI should not steer the developer toward a specific answer. The developer is intentionally overriding this prohibition, directing that the recommended option be marked so that the developer can make a faster, better-informed decision.

The change affects the Q-block format specification and the associated behavioral rule inside `aib-analysis.md`. The developer has also directed (via Q001 answer) that the recommended option must be placed first in the option list and explicitly marked `*(recommended)*`. Additionally, the developer directed (via Q002 answer, Option A) that the Answer Application Sub-flow's tie-break logic be updated from "apply the first listed option" to "apply the recommended-marked option" when a question is left unanswered.

The request does not specify changes to any other prompt, convention, or tool script. The primary artifacts to modify are `.aib_brain/prompts/aib-analysis.md` (section 7.3.3 Q-block format and section 4.8 tie-break), `tests/test_questions_in_input_md.py` (update `test_analysis_references_recommended_marker`), and `.aib_memory/context.md` (FR-007). Context documentation must be updated to reflect the behavioral change.

## Research Results

### Workspace Findings

**Current prohibition:** `.aib_brain/prompts/aib-analysis.md` line 393 contains the explicit rule: _"Do NOT include a `*(recommended)*` marker on any option — the developer chooses without AI steering. If the developer leaves a question unanswered on re-run, the Answer Application Sub-flow applies the first listed option."_ This is the single authoritative rule governing the absence of a recommended marker.

**Marker format already named:** The current prohibition itself names the marker format as `*(recommended)*`. This means the exact format string is already established in the workspace and documented; it was reserved as the "would-be" format before being prohibited. Introducing the marker can reuse this exact string without ambiguity.

**Answer Application Sub-flow tie-break rule:** Section 4.8, step 1 states _"If unanswered: apply the first listed option"_. The developer has directed (Q002, Option A) that this be changed to apply the recommended-marked option. With the recommended option always placed first (per Q001 resolution), the two rules effectively agree — but the explicit "recommended-marked" wording makes the behavior unambiguous and eliminates any dependence on list position.

**Failing automated test:** `tests/test_questions_in_input_md.py` method `test_analysis_references_recommended_marker` currently asserts `"Do NOT include" in source and "*(recommended)*" in source`. After the prohibition is removed and replaced with the new requirement, this test will fail unless updated to assert the new rule.

**Context.md FR-007:** Contains the phrase `mutually exclusive options (no \`*(recommended)*\` marker — the developer chooses without AI steering)` and `If left unanswered on re-run, the first listed option is applied.`. Both substrings require targeted replacement.

**No other prompts involved:** `aib-implement.md` is not a consumer of Q-block generation logic. No other prompt file references the Q-block format spec. The change is isolated to `aib-analysis.md`, the test file, and `context.md`.

### Industry Knowledge

**Finding 1 — Progressive disclosure in decision support (UX/HCI literature):** In decision-support interfaces (forms, configuration wizards, recommendation engines), providing a visually distinguished "recommended" or "default" option reduces cognitive load and decision fatigue without removing user agency (Nielsen Norman Group: "Defaults" principle; Material Design: outlined vs. filled button hierarchy). The applicability to AIB Q-blocks is direct: marking one option as recommended aligns with the developer's request and follows established UX conventions for guided selection.

**Finding 2 — Default-option design and its behavioral influence (behavioral economics / software design):** Thaler and Sunstein's "Nudge" framework, widely applied in software defaults (e.g., RFC 2119's "SHOULD" vs. "MUST"), distinguishes between a _default_ (what happens when no explicit choice is made) and a _recommendation_ (a signal that does not block alternatives). Best practice is to align the recommendation marker with the default so behavior matches the visual signal. If the recommended option is marked but the tie-break applies the first listed option (which may differ), this creates an inconsistency between the visible signal and the actual behavior. The marker and the tie-break should agree.

### Feasibility Notes

The change is confined to text modifications in a single Markdown prompt file (`aib-analysis.md`). No script changes, no schema changes, no test-fixture changes for anything outside the analysis prompt's behavioral tests. The only question is which option earns the recommended marker (AI-chosen vs. always-first) and whether the tie-break logic changes accordingly. Both paths are feasible within a single implementation task.

### Expert Observations

The scope is deliberately small and appropriate. The developer's intent is clear: add the marker. The only genuine complexity is the interaction between the new marker and the existing tie-break behavior. If the marker is informational only and the tie-break stays as "first listed", then the AI must always place the recommended option first — which is a structural constraint that should be documented explicitly. If the tie-break is updated to apply the recommended option, list ordering no longer matters and implementation becomes simpler.

The prohibition on `*(recommended)*` was a principled no-steering rule; the developer has consciously overridden it. The Q001 resolution ("place recommended first AND mark it") is a clean combination: structural ordering provides predictability while the explicit marker provides unambiguous visual confirmation. The replacement rule should be stated with the same authority as the prohibition it replaces to prevent re-introduction in future prompt revisions.

## Implementation Alternatives

### Alternative 1 — AI-chosen recommended option (AI picks best, marks it)

The AI identifies which option it would choose if resolving the fork autonomously and marks that option `*(recommended)*`. The marker moves with the AI's judgment. List order is not constrained.

- **Trade-offs:** Most informative to the developer (sees AI's rationale in action); introduces genuine AI steering, which the developer explicitly requested. Requires a deterministic selection criterion for the AI (what makes an option "recommended"?).
- **Codebase impact:** Modify Q-block format rule in section 7.3.3 of `aib-analysis.md`. Update tie-break guidance in section 4.8 to apply the recommended option (for consistency) OR explicitly state that the tie-break still uses first-listed position.

### Alternative 2 — First-listed option always recommended (structural convention)

The AI always places its preferred option first in the Q-block and marks Option A with `*(recommended)*`. The recommendation is implicitly the first option; the format enforces this structurally.

- **Trade-offs:** Simpler rule — no separate "which option is recommended" decision needed; always the first one. The existing tie-break behavior (apply first listed) naturally aligns with the marker without change.
- **Codebase impact:** Modify Q-block format rule in section 7.3.3 of `aib-analysis.md` to state that the first option is always marked recommended.

## Request Context

### Background

The AIB `aib-analysis.md` prompt generates Q-blocks and writes them to `.aib_memory/input.md ## Questions` when decision forks are identified during analysis. Currently, all Q-block options are displayed without any visual distinction — no option is marked as preferred or default. An explicit rule prohibits adding a `*(recommended)*` marker on any option, on the grounds that the AI should not steer the developer's choice. The developer has determined that this constraint is counterproductive: without a recommendation signal, the developer must evaluate all options from scratch every time a Q-block is presented. The request overrides the prohibition and requires that one option per Q-block be marked as recommended.

### Scope

- `.aib_brain/prompts/aib-analysis.md` — Q-block format specification in section 7.3.3 (marker rule and template); Answer Application Sub-flow tie-break rule in section 4.8.
- `tests/test_questions_in_input_md.py` — `test_analysis_references_recommended_marker` updated to assert the new rule.
- `.aib_memory/context.md` — FR-007 entry updated to reflect the new Q-block format and tie-break behavior.

### Out of scope

- `aib-implement.md` — does not generate Q-blocks; not affected.
- `close-request.py`, `create-request.py`, `finalize-input.py`, `move-request-artifacts.py` — no Q-block generation logic; not affected.
- Convention files other than as documentation consumers — not affected.
