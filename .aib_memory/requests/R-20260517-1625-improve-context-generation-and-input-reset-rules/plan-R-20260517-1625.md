## Goal

Introduce three targeted improvements to the AIB framework: (1) mandate a unique identifier on every requirement bullet in `context-convention.md` Section 5; (2) add a repetitive-content grouping rule to `context-convention.md` Section 12 and a corresponding note to `aib-refresh-context.md`; and (3) enforce a strict all-questions-answered gate in `aib-analyze.md` section 4.8 that halts the Answer Application Sub-flow and prevents `input.md` from being archived or reset when any Q-block remains unanswered.

## Constraints

- Changes are restricted to `.aib_brain/` convention and prompt files, one test file, and `.aib_memory/context.md` (refreshed via `aib-refresh-context.md`). No tool scripts are modified.

- All edits to `.aib_brain/` files must be made during implementation; this prompt is a reasoning-only artifact (GC-07).

- The auto-apply-recommended fallback in `aib-analyze.md` section 4.8 step 1 must be removed — it becomes dead code once the all-answered pre-check is in place.

- No changes to `finalize-input.py`, `close-request.py`, or Q-block format rules.

## Success criteria

- SC-1: `context-convention.md` Section 5 contains a normative rule stating that every requirement bullet must begin with a unique FR-NNN or NFR-NNN identifier followed by a colon, and that continuation content must be nested as a sub-bullet under its identified parent.

- SC-2: `context-convention.md` Section 12 contains a normative rule stating that directories with three or more items sharing a repeating naming pattern must not be listed individually; a single summary bullet describing the pattern must replace all individual entries.

- SC-3: `aib-refresh-context.md` Phase 2 references the Section 12 grouping rule for repetitive folder content.

- SC-4: `aib-analyze.md` section 4.8 contains an all-answered pre-check that halts the sub-flow and leaves `input.md` unchanged when any Q-block is unanswered. The "If unanswered: apply recommended" fallback is absent from section 4.8 step 1. GC-02 documents the unanswered-Q-block trigger as a second reason the Standard Flow Final Step does not execute.

- SC-5: `tests/test_context_formatting_rules.py` and `tests/test_questions_in_input_md.py` contain at least one new passing assertion each, covering SC-1/SC-2 and SC-4 respectively.

- SC-6: `context.md` is regenerated via `aib-refresh-context.md` and the `## Requirements` section has unique identifiers on every top-level bullet.

### Task 4: Update `aib-analyze.md` — all-questions-answered gate

#### Intent

Add an all-answered pre-check to the beginning of section 4.8 that halts the sub-flow when any Q-block is unanswered, update GC-02 to document the second trigger, remove the dead-code auto-apply-recommended fallback from step 1, and add the halt condition to the Failure Handling table.

#### Outputs

- `.aib_brain/prompts/aib-analyze.md` — GC-02 updated, Failure Handling table updated, section 4.8 updated.

#### Procedure

**Step 1** — Open `.aib_brain/prompts/aib-analyze.md`. Locate `**GC-02 — Single input reset:**` in the `## Global Constraints` section. Replace the sentence beginning with "**Exception:**" with:

```
**Exception:** (a) When Q-blocks are written to `input.md ## Questions` during this run, the Standard Flow Final Step (section 8) MUST NOT execute — the reset is deferred so the developer can read and answer the Q-blocks before the next run. (b) When the Answer Application Sub-flow (section 4.8) halts because any Q-block is unanswered, the Standard Flow Final Step (section 8) MUST NOT execute and `input.md` MUST remain unchanged.
```

**Step 2** — Locate the `## Failure Handling` table. Add a new row after the last existing row:

```
| Answer Application Sub-flow detects one or more unanswered Q-blocks | `Note: <N> of <M> questions in input.md are unanswered. Answer all questions before re-running analysis. Execution halted.` |
```

**Step 3** — Locate `### 4.8 Answer Application Sub-flow` and the `**Procedure:**` block. Renumber the existing step 0 to step 1 (if there is one) and insert the following as the new step 0 before all existing steps:

```
0. **All-answered pre-check:** Count the total number of Q-blocks in `input.md ## Questions` (M). Count the number of answered Q-blocks (N) — a Q-block is answered when at least one checkbox is marked `[x]` OR the `> Answer:` line has non-empty text after the colon. If N < M: output `Note: <N> of <M> questions in input.md are unanswered. Answer all questions before re-running analysis. Execution halted.` and HALT. MUST NOT modify `input.md`, `plan.md`, or any other file. MUST NOT continue to any subsequent step of this sub-flow or to the standard analysis flow.
```

**Step 4** — In the existing step 1 of the Answer Application Sub-flow procedure, remove the "If **unanswered**" branch entirely. After the pre-check gate passes, every Q-block is guaranteed to be answered, so only the "If **answered**" branch remains. The updated step 1 should read:

```
1. For each Q-block in `## Questions`: apply the chosen option (the checked `[x]` option or the non-empty `> Answer:` text) to the relevant `plan.md` section (`## Goal`, `## Constraints`, etc.) based on what the answer addresses. If the target section is ambiguous, apply to `## Goal`. Append a resolved entry to `plan.md` `## Decisions` in the format `**Q<nnn>:** <question text> → **Chosen:** <chosen option text>`.
```

#### Done criteria

- `aib-analyze.md` GC-02 contains the text "unanswered Q-block" or equivalent in the exception clause.

- `aib-analyze.md` Failure Handling table contains a row referencing "unanswered Q-blocks" and a `Note:` message.

- `aib-analyze.md` section 4.8 contains a step labelled "All-answered pre-check" before the Q-block processing step.

- The phrase "apply the recommended-marked option" does NOT appear in section 4.8 step 1.

#### Dependencies

#### Risk notes
