## Input Interpretation

The developer submitted three improvement directives for the AIB context-generation and input-handling workflow.

**Directive 1 — Requirements bullet identifier mandate.** When `aib-refresh-context.md` generates or updates the `## Requirements` section of `context.md`, each individual bullet must carry a unique, distinct identifier (FR-NNN for functional requirements, NFR-NNN for non-functional requirements). No continuation content for a single requirement may appear as a separate, unlabelled bullet; all continuation content for the same requirement must be nested as a sub-bullet under its identified parent. This brings the requirements section into full compliance with IEEE 29148 traceability norms.

**Directive 2 — Workspace File Inventory grouping rule.** When `aib-refresh-context.md` generates or updates the `## Workspace File Inventory` section of `context.md`, any directory whose contents consist of multiple items sharing a repeating structure (e.g., per-request subfolders under `.aib_memory/requests/`, versioned log files under `logs/`, versioned archive files under `versions/`) must NOT be listed item-by-item. Instead, a single summary bullet must replace all individual entries, describing the naming pattern and the purpose of the repeating items.

**Directive 3 — Input.md reset gate.** The Answer Application Sub-flow in `aib-analyze.md` must, before executing any other step, verify that ALL Q-blocks in `input.md ## Questions` are explicitly answered by the developer (at least one checkbox marked `[x]` or a non-empty `> Answer:` field). If any Q-block is unanswered, the sub-flow must halt immediately, leave `input.md` unchanged (preserving the Q-blocks for the developer to see and answer), and output an informational message. `input.md` must not be archived or reset in this case. The existing auto-apply-recommended-option fallback for unanswered Q-blocks is superseded by this gate.

## Research Results

### Workspace Findings

**Requirements section inconsistency in `context.md`.**
The `## Requirements` section currently contains FR-NNN–labelled bullets alongside unlabelled continuation bullets. For example, FR-003 is followed by a separate bullet beginning with "After reset, the `## Active request` line…" with no identifier of its own. FR-004 similarly has a follow-on bullet beginning with "The `analysis-<request_id>.md` artifact contains…" without an FR identifier. This is the inconsistency that Directive 1 targets.

**`## Workspace File Inventory` section absent from current `context.md`.**
The current `context.md` ends at `## Requirements`; sections 6–12 (Architecture, Technical Design, Data Architecture, Security, Operations, Development Practices, and Workspace File Inventory) are absent. This means `context-convention.md` Section 12 grouping rules will apply when `aib-refresh-context.md` is next run.

**`aib-analyze.md` section 4.8 auto-apply-recommended behaviour.**
Step 1 of the Answer Application Sub-flow contains the clause: "If **unanswered**: apply the recommended-marked option to the relevant `plan.md` section." With the new gate introduced by Directive 3, any run of the sub-flow is guaranteed to have all Q-blocks answered before reaching step 1, making the "If unanswered" branch dead code. It must be removed.

**GC-02 reset exception in `aib-analyze.md`.**
GC-02 currently states the exception as: "Exception: when Q-blocks are written to `input.md ## Questions` during this run, the Standard Flow Final Step (section 8) MUST NOT execute." This covers the case where Q-blocks were generated in the current pass. Directive 3 adds a second trigger: the sub-flow halts without running if any Q-block is unanswered, which also suppresses the reset. GC-02 must be updated to document both triggers.

**Test coverage gap.**
No existing test covers: (a) the requirements identifier rule in `context-convention.md`, (b) the grouping rule in `context-convention.md`, or (c) the unanswered-Q-block halt behaviour in `aib-analyze.md`. New test assertions must be added.

### Industry Knowledge

**Unique-identifier mandates in requirements engineering (IEEE 29148:2011 — Systems and Software Requirements Engineering).**
IEEE 29148 requires that every requirement be uniquely identifiable so that forward and backward traceability can be maintained throughout the lifecycle. The practice is directly applicable: assigning a distinct FR-NNN or NFR-NNN identifier to every bullet in `context.md ## Requirements` ensures each requirement is independently referenceable in plans, tests, and decision records.

**Progressive disclosure in structured documentation (Diátaxis Documentation Framework, Nüst et al. 2018).**
Items of the same abstraction level that appear in a structured list must be presented consistently. Having some requirement bullets with identifiers and some without creates reader friction and undermines the list's utility as a lookup structure. Consistent identifier prefixes enable scannable, auditable requirements sections.

**Grouped-summary documentation for repetitive directory contents (developer portal and man-page conventions).**
It is a well-established practice in developer documentation to summarise repetitive file collections with a pattern description rather than enumerating every instance (e.g., `/proc/<pid>/` patterns in Linux man pages, per-entity log directories in cloud platform docs). Listing 20 individual `version_vX.Y.Z_log.md` files or 30 per-request subfolders adds noise without information gain. A single summary bullet (e.g., `` - `logs/` — Contains per-version log files named `version_vX.Y.Z_log.md`; individual files are not listed. ``) conveys the same information more efficiently.

**Incremental confirmation in multi-step workflows (Nielsen Norman Group UX research on form completion).**
When a workflow requires explicit user input at multiple steps, silently substituting defaults for missing answers removes the developer's ability to make an informed choice and creates a "surprise" experience upon review. Blocking the workflow until all inputs are explicitly provided — and leaving the prompts visible — is consistent with established UX guidance on progressive confirmation. The existing auto-apply-recommended behaviour was a convenience shortcut that the developer now judges as too opaque.

### Feasibility Notes

All three directives involve text changes to convention and prompt files only; no tooling or schema migrations are required. The only non-trivial interaction is the removal of the auto-apply-recommended fallback from section 4.8: this is safe because the pre-check gate ensures the fallback path can never be reached, making it clean dead-code removal rather than a risky behavioural mutation.

### Expert Observations

- The three changes are orthogonal and can be implemented as independent edits to four files: `context-convention.md` (two sections), `aib-refresh-context.md` (one generation note), and `aib-analyze.md` (GC-02, Failure Handling table, and section 4.8).

- Removing the auto-apply fallback trades convenience for explicitness. This is a sound trade-off for a developer-facing tool where silent defaults can produce undetected plan drift. The developer's explicit statement ("answered by the user") signals a deliberate preference for strict confirmation.

- The requirements identifier mandate will require a `context.md` refresh (via `aib-refresh-context.md`) after the convention is updated to bring the existing content into compliance. The plan must include this step.

- The grouping rule needs a clear, operational definition of "repetitive structure." The plan should specify criteria such as: "a directory qualifies for grouping if it contains three or more items that follow a shared naming pattern." This prevents ambiguous application of the rule.

## Implementation Alternatives

### Alternative A — Convention-first (recommended)

Update `context-convention.md` with the two new rules (requirements identifier mandate, inventory grouping rule), update `aib-refresh-context.md` with the grouping note, and update `aib-analyze.md` with the sub-flow pre-check and GC-02 extension. All three directives are implemented as rule/prompt text changes; no tooling changes.

- **Trade-offs:** Minimal scope, directly aligned with developer directives, fully testable via text assertions. Does not change how existing `context.md` looks until `aib-refresh-context.md` is re-run.

- **Expected codebase impact:** Four files modified: `context-convention.md`, `aib-refresh-context.md`, `aib-analyze.md`, plus new test assertions in `test_context_formatting_rules.py` and `test_analysis_prompt_structure.py` (or `test_questions_in_input_md.py`).

### Alternative B — Convention + immediate context.md regen

Same as Alternative A, but also immediately re-runs `aib-refresh-context.md` as part of implementation to bring `context.md` into compliance with the new requirements-identifier rule.

- **Trade-offs:** More visible immediate improvement; adds `context.md` to the changed-files list; extends implementation scope slightly.

- **Expected codebase impact:** Same as A plus `context.md` full replacement.

### Alternative C — Prompt-only (no convention update)

Add the three rules only to `aib-analyze.md` and `aib-refresh-context.md` as inline notes, without updating `context-convention.md`.

- **Trade-offs:** Lower source-of-truth quality; convention files would not reflect the new rules, risking drift. Not recommended.

- **Expected codebase impact:** Two files modified; no convention hygiene improvement.

## Request Context

### Background

The AIB framework's `context.md` generation and `input.md` reset workflows have accumulated three small inconsistencies:

- The `## Requirements` section in `context.md` allows unlabelled continuation bullets, breaking per-bullet traceability.

- The `## Workspace File Inventory` section, when generated, lists every file individually, including repetitively structured items (log files, request subfolders, version archives), creating a noisy and hard-to-scan inventory.

- The Answer Application Sub-flow auto-applies recommended options to unanswered Q-blocks and then resets `input.md`, which can cause the developer to miss questions they intended to answer.

### Scope

- `context-convention.md` Section 5 — add requirements identifier rule.

- `context-convention.md` Section 12 — add inventory grouping rule.

- `aib-refresh-context.md` — add grouping note to Phase 2 inventory step.

- `aib-analyze.md` — add all-questions-answered pre-check to section 4.8, remove auto-apply-recommended fallback from section 4.8 step 1, update GC-02, and add halt condition to the Failure Handling table.

- `tests/test_context_formatting_rules.py` — add test assertions for both new convention rules.

- `tests/test_questions_in_input_md.py` — add test assertion for the unanswered-Q-block halt behaviour in `aib-analyze.md`.

- `context.md` — refresh via `aib-refresh-context.md` as the documentation step.

### Out of scope

- Changes to `finalize-input.py` or any other tool script.

- Changes to `close-request.py` reset behaviour.

- Changes to how Q-blocks are formatted or generated (format stays the same; only the pre-check before the sub-flow changes).

- Backfilling FR identifiers into prior closed-request plan files or analysis files.
