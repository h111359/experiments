# Analysis — R-20260528-0952

## Overview

**Request ID:** R-20260528-0952

**Request title:** Add context.md format verification script and rules

### Background

The developer requests a Python verification script in the `tools/` folder that validates `context.md` against the context convention. Additionally, the request asks for integration into the modification workflow (triggering verification on every context.md write), a rule for handling non-applicable areas with informational stub statements, and a new step in the analyze prompt to review context.md completeness and ask questions when information is missing.

### Scope

- New Python tool script `.aib_brain/tools/verify-context.py` that validates `context.md` format against `context-convention.md`.
- Integration rule: calling prompts must invoke the verification script after modifying `context.md` (separation of concerns — script is independent).
- Convention amendment: replace the `*Not yet documented.*` stub format with required atomic informational statements in every subsection.
- Update `context-convention.md` to reflect the new stub rule.
- Update existing `context.md` stubs (Analytics, Performance) to use atomic informational statements.
- New dedicated step in `aib-analyze.md` between S05 and S06: review `context.md` and ask the user questions if information is missing. Requires renumbering S06-S10 to S07-S11.
- Add verification invocation instruction to `aib-refresh-context.md` and `aib-implement.md` (where edit-context.py is called).

### Out of scope

- Changes to `edit-context.py` internal logic (verification is orchestrated externally by the calling prompt).
- Automated self-correction within the verification script (it reports only; the calling prompt corrects).
- Changes to the analysis convention or plan convention.

## Input Interpretation

The developer requests the following capabilities to be added to the AIB framework:

1. **Format verification tool:** A new Python script at `.aib_brain/tools/verify-context.py` that reads `.aib_memory/context.md` and validates its structure, formatting, and content against the rules defined in `.aib_brain/conventions/context-convention.md`. The script must execute a predefined checklist of checks derived from the convention's quality gates and formatting rules, and output structured text with pass/fail per check (exit code 0 if all pass, 1 if any fail).

2. **Prompt-orchestrated verification:** Calling prompts (not the tools themselves) are responsible for invoking `verify-context.py` after any modification to `context.md`. This maintains separation of concerns — `edit-context.py` remains a pure CRUD tool.

3. **Convention amendment — atomic stubs:** Replace the `*Not yet documented.*` stub format entirely. Every subsection in Section 2 must have at least one atomic informational statement. For non-applicable areas: `<AREA>-I-<HASH>: This area is not applicable because <reason>.` For areas awaiting information: `<AREA>-I-<HASH>: No information found yet for this area.`

4. **Context review step in aib-analyze.md:** A new dedicated step inserted between current S05 (Generate Analysis) and S06 (Quality Check), requiring renumbering of subsequent steps. The step reviews `context.md` for missing information relevant to the active request and generates questions to the user to fill gaps.

## Research Results

### Workspace Pattern-Scan Findings

- **Existing tool patterns:** All tools in `.aib_brain/tools/` use Python standard library only, follow a `main()` function structure with `argparse` for CLI, and return exit codes (0 success, 1 failure). The `common.py` module provides shared helpers. The `edit-context.py` already defines `SUBSECTIONS` (ordered area list), `VALID_TYPES`, and `STATEMENT_PATTERN` regex — these can be imported or duplicated by the new verification script.

- **Context convention quality gates:** Section "Quality Gates" in `context-convention.md` defines 12 quality criteria that the verification script should implement: structure completeness, non-empty populated sections, stub sections present, statement uniqueness, statement format compliance, rebuildability, no verbatim reproduction, determinism, no external hyperlinks, no `.aib_brain/` content, workspace inventory present, and no structural definitions in prompt.

- **Current context.md state:** Two subsections use the `*Not yet documented.*` stub format (`### 2.13 Analytics - AN` and `### 2.16 Performance - PF`). Per user decision, these must be replaced with atomic informational statements.

- **Existing verification gap:** No automated validation exists for `context.md` format compliance. The `edit-context.py` tool validates individual insertions (uniqueness check on insert) but does not perform full-file structural validation.

- **Prompt modification points:** `aib-refresh-context.md` performs full replacement of `context.md`. `edit-context.py` performs targeted line-level operations. Calling prompts will invoke `verify-context.py` after these operations.

- **aib-analyze.md current step count:** 10 steps (S01-S10). Inserting between S05 and S06 means the new step becomes S06 and existing S06-S10 become S07-S11.

### Industry Knowledge

1. **Schema validation for structured Markdown (linting approach):** Projects like `markdownlint` and custom schema validators use rule-based checklist validation. The pattern of defining checks as individual functions that return pass/fail with a message is standard. Applicability: directly applicable — the verify script should follow this function-per-check pattern.

2. **Convention-as-code pattern:** In regulated environments (ISO 27001, SOC2 documentation), automated compliance checks against document conventions are standard practice. Tools like `vale` (prose linter) validate documents against style guides programmatically. Applicability: the approach of deriving checks from the convention file is aligned with industry best practice for document quality automation.

3. **Separation of concerns in tool orchestration:** Unix philosophy and modern CI/CD design favor small, focused tools composed by an orchestrator rather than monolithic tools with built-in verification. This pattern is evident in `make`, `just`, and GitHub Actions workflows. Applicability: the user's choice (Q003 Option B) aligns with this principle — the prompt orchestrates verification, not the tool itself.

### AI Agent Critique

- **Convention amendment required:** The user chose to replace `*Not yet documented.*` entirely with atomic informational statements. This means `context-convention.md` Section "Stub Notice Format" must be amended — the existing normative text must be updated, not just extended.

- **Renumbering impact in aib-analyze.md:** Inserting a step between S05 and S06 requires renumbering S06-S10 to S07-S11 and updating all internal cross-references within the prompt (step-completion note formats, etc.).

- **Verification checks — subset vs. full:** Not all 12 quality gates from the convention are programmatically verifiable (e.g., "rebuildability" and "determinism" require semantic judgment). The script should implement the structurally verifiable subset and document which gates require manual/AI review.

- **Test coverage needed:** A test file for `verify-context.py` should be added to `tests/` following the existing pattern (`test_<tool>.py`).
