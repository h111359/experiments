The original developer input specified restructuring `context-convention.md` and `context.md` to a 12-section mandatory order. The change dissolves the standalone `## Glossary` section by merging its content into `## Domain Knowledge` (Section 2) under a `### Glossary` sub-heading. ## Research Results

### Workspace Findings

`context-convention.md` already contains a complete "Formatting Rules" section with numbered rules 1-16. Rules 11 (no tables), 12 (blank line between bullets), 14 (complete sentences), 15 (prose before bullets), and 16 (two-sentence limit) directly correspond to the four bullets in `aib-context.md` Phase 4 "Formatting requirements". Rule 6 (bold only for term definitions and critical callouts) corresponds to the Phase 4 bullet prohibiting bold as a section or field label. The convention is already the authoritative source for these rules; the Phase 4 block is purely redundant.

`tests/test_context_formatting_rules.py` contains two test classes. `TestContextConventionFormattingRules` asserts Rules 11, 12, 13, and 16 are present in `context-convention.md` — these assertions remain valid after this change. `TestContextPromptFormattingChecklist` asserts that `aib-context.md` Phase 4 contains the literal strings "Do NOT use Markdown tables anywhere in the document", "Insert one blank line between each top-level list item", and "MUST NOT exceed two sentences". These three assertions will fail after the formatting block is removed from the prompt; the test class must be updated.

`aib-context.md` Phase 5 contains a "Rule 16 verification pass" step: before writing, scan every generated bullet for two-sentence violations and correct them. This is a workflow procedure (when and how to apply the rule), not a formatting definition. It is appropriate to keep in the prompt.

`context.md` currently has sections in this order: Product Identity, Business Context, Requirements Summary, Architecture & Key Decisions, Technical Design, Data Architecture, Security & Compliance, Operations, Development Practices, Constraints & Assumptions, Glossary, Workspace File Inventory. The `## Constraints & Assumptions` section is currently at position 10 (after Development Practices) and needs to move to position 4. The standalone `## Glossary` section needs to be dissolved into `## Domain Knowledge`.

`aib-context.md` Phase 1 reads `context-convention.md` before any synthesis begins. Once the formatting rules are removed from Phase 4, the Phase 1 convention read ensures the AI agent has the authoritative rules in context before writing `context.md`.

### Industry Knowledge

**Separation of Concerns (Parnas, 1972; SOLID principles):** The single responsibility principle applied to documentation artifacts states that each artifact should own one category of knowledge without redundancy. In AIB's architecture, the convention owns structural and formatting rules; the prompt owns workflow procedures (when and how to apply the rules). Duplicating formatting definitions in both places violates this principle and creates a maintenance surface where the two can diverge. The fix — remove definitions from the prompt and reference the convention — is the canonical application of this principle.

**DRY — Don't Repeat Yourself (Hunt & Thomas, *The Pragmatic Programmer*, 1999):** Every piece of knowledge must have a single, unambiguous, authoritative representation. When the same formatting rule exists in both `aib-context.md` and `context-convention.md`, any future update to the rule must be made in two places. If one copy is updated and the other is not, the system becomes inconsistent. Removing duplication reduces the risk of silent semantic drift and makes the convention the single source of truth for all formatting constraints.

### Feasibility Notes

The change is low-risk and mechanically straightforward. The convention already contains all rules that are being removed from the prompt. The only work is removing the Phase 4 block (approximately 10 lines), adding a one-sentence convention reference, and updating three test assertions. No behavioral change is introduced: the AI agent still reads the convention in Phase 1 and applies the same rules when generating `context.md`.

### Expert Observations

The scope is minimal and well-contained. The change does not alter any formatting rule — it only relocates where those rules are defined. Removing the Phase 4 block from the prompt shortens the prompt without any loss of enforced behavior, because Phase 1 already guarantees the convention is in context.

The test update is necessary and correct. The tests were written at the time the formatting rules were intentionally duplicated in the prompt. Now that the convention is the intended single source, the test coverage goal shifts from "prompt contains the rule text" to "prompt references the convention for formatting". Updating `TestContextPromptFormattingChecklist` to assert for the convention reference achieves the same test intent with the correct architectural posture.

No simplification opportunities for the existing Tasks 1-3 (convention and context.md structural changes) were identified during this re-run. Those tasks remain unchanged and correct.

## Implementation Alternatives

### Alternative A — Remove block and add convention reference (selected)

Replace the four-bullet "Formatting requirements" block in `aib-context.md` Phase 4 with a single sentence referencing the convention's Formatting Rules section. Update `TestContextPromptFormattingChecklist` in `tests/test_context_formatting_rules.py` to assert that the prompt contains this convention reference rather than the specific formatting strings.

- **Trade-offs:** Minimal change; eliminates redundancy; Phase 1 convention read already guarantees the rules are in context before synthesis. The single reference sentence preserves traceability without duplication.

- **Codebase impact:** Approximately 10 lines removed from `aib-context.md`; one sentence added; three test assertions updated.

### Alternative B — Remove block entirely without any reference

Remove the formatting block from Phase 4 without adding any replacement text, relying entirely on the Phase 1 convention read.

- **Trade-offs:** Slightly shorter prompt, but loses explicit cross-reference. A future reader of the prompt would need to know to consult the convention for formatting rules without a prompt-level cue.

- **Codebase impact:** Same as A but without the reference sentence.

### Alternative C — Create a separate "Prompt Formatting Guidance" section in the convention

Move the Phase 4 block into a new section in `context-convention.md` explicitly scoped to prompt-facing formatting instructions.

- **Trade-offs:** Adds convention complexity; the existing Formatting Rules section (Rules 1-16) already covers all the same rules. No added value over Alternative A.

- **Codebase impact:** Approximately 10 lines added to convention; approximately 10 lines removed from prompt; more invasive.
