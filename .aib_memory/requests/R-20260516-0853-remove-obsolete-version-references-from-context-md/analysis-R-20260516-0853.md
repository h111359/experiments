## Input Interpretation

The Developer requests that the `aib-context.md` prompt be augmented with a content currency policy rule governing what the AI Automation Agent may write into `.aib_memory/context.md`. Specifically, the generated product memory document must represent the current state of the product exclusively. It must not contain version history annotations of the form "introduced in vX.Y.Z", "added in vX", "deprecated as of vX", or "removed as of"; and it must not include glossary entries for deprecated or removed concepts, since the presence of such entries implies those concepts are still active. Any existing content in `.aib_memory/context.md` that violates this policy must be removed as part of the same implementation cycle by re-executing the updated prompt.

## Research Results

### Workspace Findings

**Current `context.md` violations:** Three categories of version-history content are present in the current `.aib_memory/context.md`:

1. Glossary entry `Auto-request` — contains the phrase "introduced in v1.2.0", which is a version-stamp annotation on a current feature rather than a description of the feature's current behavior.

2. Glossary entry `Iteration` — opens with "(Deprecated)" and closes with "Removed as of R-20260414-1421." This entry describes a concept that no longer exists in the product; its presence implies the concept is still relevant to readers of current-state documentation.

3. Glossary entry `instructions.md` — contains "Currently populated with the curated-change-log directive (R-20260422-1308)", embedding a specific request ID (a change-history artifact) inside what should be a capability description.

**Prompt gap in `aib-context.md`:** The synthesis prompt's Core Requirements section mandates "workspace/tool/model/vendor agnostic" output and "full content replacement on each execution", but contains no rule prohibiting version history annotations. Phase 4 — Synthesis instructs the agent to "keep content consistent with workspace evidence" and "prefer concise, deterministic wording" — neither of which prohibits historical annotations. The gap is a missing synthesis constraint, not a structural defect.

**Convention gap in `context-convention.md`:** The convention governs section structure, required headings, and formatting rules. It does not restrict content style beyond mandating definition-list format for glossary entries. No existing rule in the convention prohibits version history references. The convention is not the correct location for this rule (see Implementation Alternatives).

### Industry Knowledge

**Living documentation principle** (from Cyrille Martraire's *Living Documentation* and Dan North's BDD community): Reference documentation that reflects only the current system state is more reliable and less cognitively expensive for consumers than documentation that interleaves current state with change history. Historical annotations belong in changelogs, commit messages, or Architecture Decision Records — document types with an explicit historical purpose — not in current-state reference documentation. Applicability: high. `context.md` is consumed by the AI Automation Agent as its product memory; interleaving version annotations creates noise that can cause the agent to treat historical information as current constraints.

**Diataxis documentation framework** (Procida, 2021): The Diataxis framework classifies documentation into four types — tutorials, how-to guides, reference, and explanation. Reference documentation (the category `context.md` belongs to) must describe the current state of the system in a neutral, factual tone. Changelog content (what changed and when) is an orthogonal concern that must reside in a dedicated artifact. Applicability: directly applicable; the current `context.md` mixes reference and changelog concerns in glossary entries.

### Feasibility Notes

The implementation risk is low. Adding a synthesis constraint to `aib-context.md` is a text-only addition to an existing section. Re-executing the prompt to regenerate `context.md` is a standard AIB workflow operation. The only non-trivial judgment required is identifying every violation in the current `context.md` — three instances were identified by workspace scan; no additional instances are suspected but the regeneration will be the authoritative cleanup.

### Expert Observations

The three violations in `context.md` represent a common drift pattern: documentation authors add historical context annotations to help readers understand lineage, but these annotations accumulate and eventually make the document harder to use as a current-state reference. A single preventive rule in the synthesis prompt is the minimal effective intervention. Removing the `Iteration` entry entirely (not just stripping its deprecated tag) is the correct action — a concept that does not exist in the current product has no place in a current-state glossary. The `instructions.md` entry improvement (removing the request ID reference) is a quality improvement to the entry's precision, not a structural change.

## Implementation Alternatives

**Alternative A — Prompt-only content currency rule (selected):** Add a new synthesis constraint to `.aib_brain/prompts/aib-context.md` that prohibits version history annotations and deprecated-concept entries. Then re-execute the prompt to regenerate `context.md`.

- Trade-offs: Minimal scope. The rule lives where synthesis behavior is defined. No convention change needed. Effective immediately on next run.

- Codebase impact: 1 file modified (`aib-context.md`); 1 file regenerated (`context.md`).

**Alternative B — Prompt + convention rule:** Add the rule to both `aib-context.md` and `context-convention.md`, making it a normative convention requirement.

- Trade-offs: Convention becomes more complete, but conflates structural rules (what sections exist) with content-policy rules (what style of text is forbidden in those sections). Creates maintenance overhead — two files must be kept consistent.

- Codebase impact: 2 files modified + 1 regenerated.

A prohibition on version history annotations is a synthesis behavior rule, not a structural rule. Adding it to the convention conflates two concerns. Resolved: prompt only. |
| Strip version markers from existing entries vs. remove entire deprecated concept entries | resolve-autonomously | Developer input: "no need of references that something is stopped to be valid after specific version." ## Request Context

### Background

`.aib_memory/context.md` is the AIB product memory file consumed by the AI Automation Agent during every analysis and implementation run. Its content shapes AI decisions about scope, terminology, and constraints. The current file contains glossary entries annotated with version history information (e.g., "introduced in v1.2.0") and a deprecated concept entry ("Iteration"). These annotations are change-history artifacts that belong in version logs, not in a current-state product reference document. The `aib-context.md` prompt that generates this file has no explicit rule preventing such annotations, allowing them to accumulate over time.

### Scope

- `.aib_brain/prompts/aib-context.md` — add a content currency policy rule to the synthesis phase.

- `.aib_memory/context.md` — regenerate with the updated prompt to remove existing violations.

### Out of scope

- `.aib_brain/conventions/context-convention.md` — no structural changes required.

- All other prompts, conventions, and tool scripts.

- Historical version logs and changelogs (`logs/`).

- Any workspace file outside `.aib_brain/prompts/aib-context.md` and `.aib_memory/context.md`.
