# Analysis — R-20260528-0859

## Overview

- **Request ID:** R-20260528-0859
- **Request title:** Restructure context.md to atomic statements format

### Background

The developer requests a fundamental structural redesign of `.aib_memory/context.md`. The current format uses prose-based sections with bullet points. The new format replaces this with atomic statements — each a single-line entry with an area prefix, type identifier, hash code, statement text, and optional relationship suffix. The goal is to enable precise, machine-editable knowledge management where every piece of workspace knowledge is individually addressable, insertable, and deletable via a CLI tool.

### Scope

- Redesign the structure of `.aib_memory/context.md` to use atomic statements grouped by 22 topic areas.
- Modify `.aib_brain/conventions/context-convention.md` to reflect the new section structure and atomic statement format.
- Create `.aib_brain/tools/hash-text.py` — a script that generates an 8-character hash from input text.
- Create `.aib_brain/tools/edit-context.py` — a CLI script for select, insert, and delete operations on atomic statements in `context.md`.
- Modify `.aib_brain/prompts/aib-analyze.md` to add a plan task that invokes `edit-context.py` with specific context modifications.
- Modify `.aib_brain/prompts/aib-refresh-context.md` to perform multi-pass enrichment and verification of `context.md`.

### Out of scope

- Direct modification of `.aib_memory/context.md` content as part of this request (the scripts will do that when invoked).
- Changes to the implementation prompt (`aib-implement.md`) beyond what's needed to support the new plan task structure.
- Migration of existing `context.md` content to the new format (that will happen on next `aib-refresh-context.md` execution).

## Input Interpretation

The developer requests a complete overhaul of the `.aib_memory/context.md` knowledge representation model. The specification-grade interpretation is:

1. **New context.md structure.** The file shall contain three top-level sections:
   - `## 1. Product Identity` — retained from current format.
   - `## 2. Statements` — the core section containing all knowledge as atomic statements grouped into 22 subsections (PO, CM, DO, CO, BP, FN, AR/TD, TS, NW, DS, DF, PR, AN, UI, SC, PF, OP, DV, DP, DR, OB, DM).
   - `## 3. Workspace File Inventory` — retained from current format.

2. **Atomic statement format.** Each statement is a single line: `<area_prefix>-<type_letter>-<hash>: <text> {Related to: <list>}`. The hash is generated by `hash-text.py`. Statement types are: Definition (N), Requirement (R), Constraint (C), Reference (E), Relationship (L), Rule (U), Assumption (A), Decision (D), Information (I).

3. **Uniqueness invariant.** The combination of area prefix + type letter + hash (the "statement index") MUST be unique within the file.

4. **hash-text.py script.** Takes input text as a parameter, returns an 8-character hex hash (similar to a Git short commit hash).

5. **edit-context.py script.** Supports three operations:
   - `select` — returns statement text by area, type, and hash.
   - `insert` — creates a new entry given area, type, hash (or auto-generates it), and text.
   - `delete` — removes an atomic statement by area, type, and hash.
   - Validates no duplicate statement indices exist.

6. **Analyze prompt modification.** Add a plan task that invokes `edit-context.py` with specific modifications derived from analysis decisions.

7. **Refresh-context prompt modification.** Add multiple enrichment and verification passes:
   - Verify all analysis decisions are reflected in context.
   - Verify all plan task results are reflected in context.
   - Verify all modified file changes are reflected in context.

8. **Granularity principle.** Composite and complex statements should be split into multiple smaller atomic statements. The statements must be detailed enough to enable full workspace recreation from the context file alone.

### Industry knowledge

- **Knowledge graphs as atomic facts (Zettelkasten method):** The atomic statement approach parallels Zettelkasten-style knowledge management — each note is self-contained, uniquely identified, and linked to others. This pattern is well-established in PKM (Personal Knowledge Management) literature and is proven effective for machine-processable knowledge bases. Applicability: high — the atomic format enables surgical updates and cross-referencing.

- **Content-addressable hashing (Git model):** Git uses SHA-1 truncated to 7-8 hex characters for short commit hashes. For non-cryptographic content addressing, CRC32 (8 hex chars) or truncated SHA-256 are standard approaches. Applicability: high — the 8-char hash provides sufficient collision resistance for a bounded statement set (thousands of entries).

- **Structured knowledge representation (RDF triples / semantic web):** The `subject-predicate-object` pattern from RDF is analogous to the area-type-statement model. Industry practice shows that explicit relationship annotations (like the `{Related to:}` suffix) enable graph traversal without requiring a full triple-store. Applicability: medium — validates the relationship suffix design.

## Decision Register

### Decision Points

#### Decision Point: Hash algorithm for hash-text.py

- **Why:** Multiple valid hashing approaches exist for generating an 8-character hex identifier from text.
- **Alternatives:**
  - **Truncated SHA-256**: Use `hashlib.sha256(text.encode()).hexdigest()[:8]`. Benefits: well-known, deterministic, good distribution. Drawbacks: overkill for non-cryptographic use, 8-char truncation has collision probability at ~50k entries.
  - **CRC32**: Use `zlib.crc32(text.encode()) & 0xffffffff` formatted as 8-char hex. Benefits: fast, produces exactly 8 hex chars naturally, commonly used for content fingerprinting. Drawbacks: higher collision probability than SHA-256 truncation, not cryptographic.
  - **Truncated MD5**: Use `hashlib.md5(text.encode()).hexdigest()[:8]`. Benefits: fast, good distribution. Drawbacks: MD5 is deprecated for security but fine for content addressing.
- **Tag:** `resolve-autonomously`
- **Rationale:** The input states "returns an 8 symbol hash value (similar to git commit hash value)". Git uses truncated SHA-1. Following the Git analogy, truncated SHA-256 is the modern equivalent. The developer explicitly referenced "similar to git commit hash value" in `.aib_memory/input.md ## Input`.
- **Resolution:** Truncated SHA-256 — `hashlib.sha256(text.encode()).hexdigest()[:8]`.

#### Decision Point: How hash-text.py handles statement text for hashing

- **Why:** The hash must be deterministic. The input says "by provided input text as parameter returns an 8 symbol hash value" — it's unclear whether the hash input includes the relationship suffix or only the statement body.
- **Alternatives:**
  - **Hash only the statement body text (excluding relationship suffix)**: This keeps the hash stable even if relationships change.
  - **Hash the complete text including relationship suffix**: Simpler but means adding/removing a relationship changes the hash.
- **Tag:** `resolve-autonomously`
- **Rationale:** The input examples show `FN-R-dd0db0e4: The system must download only files that are new.` where the hash `dd0db0e4` clearly derives from the statement text "The system must download only files that are new." (without prefix or suffix). The script is described as "by provided input text as parameter" — the text is the statement content. Source: `.aib_memory/input.md ## Input`, examples section.
- **Resolution:** Hash only the statement body text (the text after the colon-space, before any `{Related to:}` suffix).

#### Decision Point: How edit-context.py generates the hash on insert

- **Why:** The input says the hash is retrieved by executing `hash-text.py`, but `edit-context.py` has an `insert` operation that needs a hash. Should insert require the hash as a parameter, or auto-generate it?
- **Alternatives:**
  - **Require hash as parameter**: The caller must invoke `hash-text.py` separately and pass the hash. More explicit, separates concerns.
  - **Auto-generate hash internally**: `edit-context.py` calls the hashing logic internally on the provided text when no hash is supplied. More convenient.
  - **Both (optional parameter)**: Accept an optional `--hash` parameter; if omitted, auto-generate from text.
- **Tag:** `resolve-autonomously`
- **Rationale:** The input states "Insert – creates new entry by provided area, type, hash and text" — explicitly listing hash as a required parameter. Source: `.aib_memory/input.md ## Input`.
- **Resolution:** Require hash as a mandatory parameter for insert operations.
