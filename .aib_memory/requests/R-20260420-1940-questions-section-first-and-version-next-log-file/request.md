## Goal

1. Move the `## Questions & Decisions` section in `request.md` to position 2, immediately after `## Goal`, so users see open decision points before reading the full request body.

2. Create a permanent file `logs/version_next.md` that accumulates human-authored implementation change summaries. The `aib-implement.md` prompt must append a summary entry for each completed request.

3. Update `scripts/release_bookkeeping.py` and `.github/workflows/aib-semver-patch-bump-and-log.yml` to read the content of `logs/version_next.md` and use it to form the changes section of the per-version log, then clear `logs/version_next.md` so it is ready for the next release cycle.

## Background

The `## Questions & Decisions` section is currently placed at position 10 of 12 mandatory sections in `request.md`, requiring users to scroll past Goal, Background, Scope, Out of scope, Constraints, and Success criteria before discovering pending AI-generated questions. Placing it directly after `## Goal` makes it immediately visible to any reviewer opening the request file.

The per-version log generated by CI (`logs/version_vX.Y.Z_log.md`) currently derives its content from PR commit subjects, which are often terse one-liners. Accumulating semantically rich implementation summaries in `logs/version_next.md` at implementation time—written by the AI agent using full request context—produces more informative release notes for reviewers and changelog consumers.

## Scope

- Update `request-convention.md` to reorder the 12 mandatory sections so that `## Questions & Decisions` appears at position 2 (immediately after `## Goal`), shifting the former positions 2–9 down by one.

- Update `aib-analysis.md` Part 2 output instructions to write/update the `## Questions & Decisions` section in the new position (after `## Goal`).

- Create `logs/version_next.md` as a permanent VCS-tracked accumulator file in the repository root's `logs/` directory.

- Update `aib-implement.md` to append a change summary entry to `logs/version_next.md` after each successful request implementation (before invoking `close-request.py`).

- Update `scripts/release_bookkeeping.py` to read `logs/version_next.md` content and use it as the changes section of the per-version log when the file is present and non-empty; fall back to PR commit subjects when `logs/version_next.md` is absent or empty (current behavior preserved). After writing the log, clear `logs/version_next.md` by resetting it to the seed template: `# Pending changes for next version\n`. In `--dry-run` mode, the file MUST NOT be cleared.

- Update `.github/workflows/aib-semver-patch-bump-and-log.yml` as needed to ensure the cleared `logs/version_next.md` is committed alongside the version log and SemVer marker rotation.

## Out of scope

- Retroactive updates to existing closed `request.md` files (section reordering applies only to newly generated requests going forward).

- Changes to any other sections or the total count of mandatory sections in `request.md`.

- Adding `logs/version_next.md` to `references.md`.

- Changing the format of the per-version log beyond the changes section content.

- Installing additional Python libraries in `release_bookkeeping.py` or the tool scripts.

- Changes to `initialize.py` to seed `logs/version_next.md`; the file is created only by `aib-implement.md` on first write.

## Constraints

- `request-convention.md` defines normative section ordering; the reorder must maintain all 12 mandatory sections with non-empty requirements preserved for Goal, Background, Scope, Out of scope, Constraints, and Success criteria.

- Tool scripts must remain Python 3.10+ using the standard library only.

- The CI workflow must remain idempotent on re-runs.

- `logs/version_next.md` must be tracked in VCS (not gitignored).

- `.aib_brain/` assets MUST NOT be modified by tool scripts at runtime.

- The clearing of `version_next.md` must not break CI workflow idempotency on repeated runs for the same PR.

## Success criteria

- SC-1: A newly generated `request.md` produced by `aib-analysis.md` has `## Questions & Decisions` as the second top-level heading, immediately following `## Goal`.

- SC-2: The file `logs/version_next.md` exists in the repository and is tracked by VCS.

- SC-3: After running `aib-implement.md` for a completed request, `logs/version_next.md` contains a summary entry for that request.

- SC-4: After a CI run triggered by a PR, the per-version log (`logs/version_vX.Y.Z_log.md`) incorporates the content from `logs/version_next.md` rather than (or in addition to) bare commit subjects.

- SC-5: After a CI run, `logs/version_next.md` is cleared and the cleared state is committed to the PR branch.

## Questions & Decisions

## Assumptions

- A1: No existing tool script (`common.py`, `create-request.py`, `close-request.py`, `menu.py`) parses `request.md` sections by numeric position; all section references use heading names. Confirmed by code scan: no position-based section parsing found in any tool script. Moving `## Questions & Decisions` to position 2 therefore does not break any automated tooling.
  - Risk if false: Tool scripts would mis-read request sections after the reorder, causing silent errors or incorrect behavior.

- A2: The `git add .aib_brain logs versions` in the CI workflow's "Commit and push" step already stages the entire `logs/` directory, including `logs/version_next.md`. Confirmed by reading the workflow YAML. No structural workflow change is needed to commit the cleared file.
  - Risk if false: The cleared `logs/version_next.md` would not be committed, accumulating entries across multiple releases.

- A3: `logs/version_next.md` uses one bullet entry per implemented request, format: `- <request_id>: <summary>`. The CI script reads the file's non-empty, non-header lines as the changes list.
  - Risk if false: If the format differs from what the script expects, the version log may be malformed or empty.

- A4: When `logs/version_next.md` is non-empty, its content completely replaces the commit-subject list in the changes section (not supplemented by commit subjects). When absent or empty, commit subjects are used as the fallback.
  - Risk if false: Duplicated or mixed-content changes section in the version log.
