# OBS-01 - Logging

Source location: `04 Technology / Observability`

## Overview

AIB provides two categories of persistent logs:

1. **Release version logs** — generated by CI release bookkeeping (`logs/version_vX.Y.Z_log.md`). Documented in CMP-ART-0007.
2. **Action execution logs** — generated by the AIB Command Menu (`logs/aib-action-<YYYYMMDD>-<HHmmss>-<action-id>.log`). Introduced by R-20260403-1651.

This document specifies the action execution logging standard.

## Log Levels

Action execution logs use a flat marker-based format rather than traditional log levels:

- `[START]` — action execution begins.
- `[CMD]` — the full command being executed.
- `[OUT]` — a line from subprocess stdout.
- `[ERR]` — a line from subprocess stderr.
- `[THREAD-ERROR]` — an exception raised inside a streaming thread, capturing type and message.
- `[EXIT]` — subprocess termination with return code.

## Log Event Schema

Each action execution log file contains the following entries in order:

| Field | Format | Required | Description |
| --- | --- | --- | --- |
| START marker | `[START] <ISO-8601-timestamp> — <action-title>` | Y | Timestamp and action name |
| CMD marker | `[CMD] <full-command-string>` | Y | The exact command executed |
| OUT lines | `[OUT] <line-content>` | N | Zero or more stdout lines |
| ERR lines | `[ERR] <line-content>` | N | Zero or more stderr lines |
| THREAD-ERROR lines | `[THREAD-ERROR] <ExceptionType>: <message>` | N | Exception raised in a streaming thread; zero or more per execution |
| EXIT marker | `[EXIT] <return-code>` | Y | Integer subprocess exit code |

## Structured Logging Format

- Encoding: UTF-8 plain text.
- One marker per line.
- No JSON wrapping; human-readable flat text.

## Collection & Transport

- Logs are written directly to the local filesystem by the AIB Command Menu (`menu.py`).
- No remote transport, agents, or sidecars are used.
- Writes use Python standard library file I/O with explicit flush after each line.

## Destinations & Storage

| Destination | Purpose | Path pattern | Access pattern |
| --- | --- | --- | --- |
| Local filesystem | Developer reference and debugging | `logs/aib-action-<YYYYMMDD>-<HHmmss>-<action-id>.log` | Manual file read |

## Retention & Lifecycle

- Retention policy: unbounded (no automatic cleanup).
- Developer manages manual deletion as needed.
- Execution log files are excluded from version control via `.gitignore` pattern `logs/aib-action-*.log`.
- Release version logs (`logs/version_*.md`) are committed and retained indefinitely.

## Taxonomy & Categories

| Category | Marker prefix | Description |
| --- | --- | --- |
| Execution start | `[START]` | Action invocation timestamp |
| Command | `[CMD]` | Full command string |
| Standard output | `[OUT]` | Subprocess stdout content |
| Standard error | `[ERR]` | Subprocess stderr content |
| Thread error | `[THREAD-ERROR]` | Exception caught inside a streaming thread |
| Execution end | `[EXIT]` | Return code |

## Correlation & Tracing

- Each log file corresponds to exactly one action execution.
- The filename timestamp and action-id serve as the correlation key.
- No cross-file trace propagation is implemented.

## Privacy & Security

- Log files MUST NOT contain secrets, credentials, tokens, or sensitive PII.
- The tee helper writes only subprocess stdout/stderr content; no environment variables or system secrets are logged.
- User keyboard input passed via stdin inheritance is NOT logged.

## Alerting & Usage Expectations

- No automated alerting is configured for action execution logs.
- Developers review logs manually when diagnosing action failures.

## Quality Gates & Validation

- Every action execution MUST produce a log file with `[START]` and `[EXIT]` markers.
- Log file content is validated by automated tests in `tests/test_menu.py`.

## Operational Procedures

- To inspect a recent action execution: open the most recent `logs/aib-action-*.log` file.
- To clean up old logs: delete `logs/aib-action-*.log` files manually.
- To investigate a failure: check the `[ERR]` lines and the `[EXIT]` return code in the log file.
