## Executive Summary

- **Request ID:** R-20260507-2209

- **Title:** Improve request.md readability and remove redundant sections

- **Purpose:** Reduce cognitive overhead when humans review AI-generated `request.md` files by removing two redundant sections (`## Code and Asset Scan for Impacted Components` and `## Internal Review of Request and Product Docs`), keeping `## Outputs` as a labeled task field while embedding other task metadata inline, requiring explicit file-path references in every Procedure step, and adding an explicitness standard for documentation steps that mirrors the code step standard.

- **Key inputs processed this run:** `input.md` carried four developer amendments that materially altered the prior scope: (1) every Procedure step must cite the exact file path it operates on; (2) Internal Review pre-flight flags must be redistributed into Plan task Risk Notes or Q&D blocks; (3) `## Outputs` must remain a labeled field per task (reversing the prior intent to embed it in Procedure steps); (4) documentation steps must carry the same explicitness standard as code steps (target file, change description, acceptance test).

- **`request.md` sections updated this run:** `## Scope` and `## Success criteria` amended to reflect kept `## Outputs` field and new per-step file-path requirement; `## Assumptions`, `## Plan`, `## Documentation`, and `## Questions & Decisions` fully replaced.

## Domain Knowledge Essentials

- **AIB (AI Builder):** A minimal, file-first, model-agnostic framework for specification-driven software development inside a single repository workspace. Its workflows are defined by Markdown prompt files and normative conventions.

- **`request.md`:** The primary structured work-item artifact. Generated by `aib-analysis.md`, read by `aib-implement.md`, archived upon closure. Single source of truth for scope, constraints, and the implementation plan.

- **`request-convention.md`:** The normative specification for `request.md` structure, section order, validation rules, and formatting. Changes here propagate to all future generated `request.md` files.

- **`aib-analysis.md`:** The prompt that runs the analysis workflow. Generates `analysis.md`, updates `request.md` optional sections (Assumptions, Plan, Documentation, Q&D, Code Scan, Internal Review), and auto-creates a request when none exists.

- **Mandatory section:** A `request.md` heading that must exist and appear in exact order. Currently 12; the target after implementation is 10.

- **Code and Asset Scan section:** Current section 11 in `request.md`; an AI-generated table listing files impacted by the request. Identified as redundant because the same information should live in Plan tasks.

- **Internal Review section:** Current section 12 in `request.md`; an AI-generated list of factual findings (ambiguities, contradictions, cross-reference issues). Identified as redundant because its findings should be in Plan task Risk Notes or surfaced as Q-blocks.

- **Plan task schema:** The structured format for each task in `## Plan`. Currently includes labeled fields: Intent, Inputs, Outputs, External Interfaces, Environment & Configuration, Procedure, Done Criteria, Dependencies, Risk Notes. After implementation: `Outputs` remains a labeled field; `Inputs`, `External Interfaces`, and `Environment & Configuration` are embedded in Procedure steps.

- **Developer:** The human who authors request content and reviews analysis artifacts. Wants faster review of `request.md`.

- **AIB Maintainer:** The role that owns `.aib_brain/` assets including prompts and conventions.

## External Benchmarking

- **Lightweight specification formats (ADR / RFC patterns):** Architecture Decision Records commonly strip ancillary metadata and place it in procedure steps or footnotes. The prevailing recommendation (Thoughtworks Technology Radar, Michael Nygard ADR format) is that each section should add unique value — repeated or derivable content reduces signal-to-noise. This aligns with removing the Code Scan table (derivable from Plan tasks) and Internal Review (derivable from Q&D and Risk Notes).
  - Takeaway: The ADR principle of "one source of truth per fact" is directly applicable. Adopted.

- **Work Breakdown Structure schemas in PM tooling (PMI PMBOK, Linear, Shortcut):** Modern agile WBS formats embed file/component scope into task procedures rather than maintaining a separate impact registry. The separate "impacted files" list is a legacy waterfall-era pattern.
  - Takeaway: Moving file-impact information into Plan task Procedure steps and Risk Notes aligns with modern WBS practice. Adopted.

- **Test-driven specification (BDD / Given-When-Then):** Specification-by-example patterns (Gherkin) require each step to reference its subject (file, endpoint, component) explicitly. The developer amendment to require exact file paths in Procedure steps is consistent with this traceability principle.
  - Takeaway: Requiring file-path citations in Procedure steps increases traceability and reduces ambiguity. Adopted.

## AI Copilot Suggestions

**Observation 1 — Scope creep risk on "embed metadata in Procedure steps"**
The developer has partially reversed the original scope (keep `Outputs`; embed the rest). Keeping `Outputs` labeled is correct — it is the externally visible output contract. However, embedding `Inputs`, `External Interfaces`, and `Environment & Configuration` into Procedure steps may produce verbose steps that are harder to scan than the labeled fields. Evaluate whether the readability goal is sufficiently achieved by section removal alone without touching the schema.
- Suggestion: If post-implementation review shows steps are now harder to scan, consider retaining all four metadata fields and relying solely on section removal for the readability win.

**Observation 2 — No automated validation for `Outputs` field retention**
The success criteria will verify schema changes, but there is no automated test planned that verifies `Outputs` remains a labeled field in a generated plan after changes. A future analysis run that omits `Outputs` from a task block would not be caught.
- Suggestion: Extend the regression test (T6/T7) to assert that `**Outputs:**` appears as a labeled field in the schema definition in both `aib-analysis.md` and `request-convention.md`.

**Observation 3 — Documentation step explicitness is a prose-enforced rule**
The requirement for documentation steps to carry "target file, what to change, acceptance test" is an excellent traceability improvement, but it is a prose rule in the prompt — no automated test can verify runtime compliance. Future analysis runs may silently omit required detail.
- Suggestion: Embed this rule explicitly in `request-convention.md` under the Plan section description so it appears in the normative contract and can be referenced by manual reviewers.

**Observation 4 — Scope is appropriately sized**
Removing two sections from prompt and convention, adjusting the schema, adding file-path citation rules, and updating tests is a well-bounded single-sprint unit of work. No scope creep detected.

## Multi-Perspective Stakeholder Review

### Senior Solution Architect

This request is technically coherent and low-risk in isolation. The two removed sections are genuinely redundant — their information is a proper subset of what appears in Plan tasks and Q&D blocks. Removing them reduces the contract surface without losing fidelity. The partial schema change (keep `Outputs`, embed the rest) is architecturally sound because `Outputs` is the externally visible contract of a task, whereas `Inputs`, `External Interfaces`, and `Environment & Configuration` are better expressed procedurally. The file-path citation requirement in Procedure steps is a meaningful traceability improvement.

- The risk of prompt-convention drift is the primary architectural concern: both files must be updated atomically to prevent generated `request.md` files failing convention validation.
- The test coverage gap (no existing test for section names) is a latent risk that this request partially addresses.
- The `context.md` regeneration dependency must not be skipped; stale context misleads future analysis runs.
- The change to the auto-request creation validation step (10 sections instead of 12) is a correctness fix, not a feature addition.

### Product Owner

This request directly addresses a stated developer pain point. The two removed sections add length without adding unique business value during review. The amended plan (keeping `Outputs` labeled, requiring file paths in steps) is an improvement over the original draft.

- Success criteria are measurable and cover the key behavioral changes.
- Scope is well-contained with no user-facing UI or API impact.
- The `context.md` update (SC-8) ensures product documentation remains accurate after implementation.
- Risk: if tests are not updated alongside prompt/convention, silent regressions could reintroduce removed sections in future runs.

- Concern: if the prompt is not updated correctly, developers may see a silent regression where Code Scan or Internal Review sections reappear. The regression test (T11) addresses this.
