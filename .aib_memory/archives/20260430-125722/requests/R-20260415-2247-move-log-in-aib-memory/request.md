# Request

## Goal

Currently the scripts in .aib_brain create logs in folder `logs` in the root folder of the workspace. These logs should be moved in a folder `.aib_memory/logs` and no logs of AIB framework shall be created outside `.aib_memory/logs` folder anymore.

Check all the documentation, prompts, conventions, scripts and apply the change of logs location.

## Background

AIB currently writes two categories of logs to `logs/` at the workspace root:

1. **Action execution logs** (`logs/aib-action-<timestamp>-<action-id>.log`) — transient, written by `menu.py`, excluded from VCS via `.gitignore`.
2. **Release version logs** (`logs/version_vX.Y.Z_log.md`) — generated by `release_bookkeeping.py` during CI, committed to VCS. These logs are repository-specific release artifacts and must remain in the `logs/` folder at the workspace root.

Keeping action execution logs in the workspace root creates a mixed-concern directory: transient AIB operational artifacts sit alongside product source files and committed release records. Moving only action execution logs to `.aib_memory/logs/` aligns with the framework's architecture where all AIB runtime artifacts reside in `.aib_memory/`, while preserving the established `logs/` location for committed release version logs.

## Scope

- Change `_make_log_path()` in `.aib_brain/tools/menu.py` to write action execution logs to `.aib_memory/logs/`.

- Update `.gitignore` to exclude `.aib_memory/logs/aib-action-*.log` (replace old `logs/aib-action-*.log` pattern).

- Update `tests/test_menu.py` `TestMakeLogPath` assertions to reference `.aib_memory/logs`.

- Regenerate `.aib_memory/context.md` via `aib-context.md` after all changes are applied.

## Out of scope

- Changes to log file content or format.
- Changes to `release_bookkeeping.py` or its `--log-dir` default value. Release version logs remain in the root `logs/` folder — they are repository-specific committed artifacts, not AIB runtime artifacts.
- Adding `--log-dir` arguments to `.github/workflows/aib-semver-patch-bump-and-log.yml`.
- Physical migration of release version log files (`logs/version_vX.Y.Z_log.md`) from `logs/` to `.aib_memory/logs/`.
- Removal of the root `logs/` directory from VCS (it must remain because it holds committed release version logs).
- Updates to `README.md` (all existing `logs/` references describe release version log behaviour, which is unchanged).
- Updates to `docs/Development_and_Deployment_Specification.md` (all existing `logs/` references describe the root `logs/` directory, which is preserved).
- Any non-AIB logs (Python virtual-environment logs, OS-level logs, test runner output).

## Constraints

- All changed tool scripts must remain compatible with Python 3.10+ standard library (NFR-004).
- The `.gitignore` update must be committed in the same batch as the code changes to prevent transient action logs from being accidentally staged.
- `.aib_brain/` files (`menu.py`) must not be modified by AIB tool scripts — only humans or this explicit implementation request may modify them.

## Success criteria

1. After implementation, running the AIB menu (`run.bat`) writes action logs to `.aib_memory/logs/aib-action-*.log` and never to `logs/`.
2. After implementation, the CI release bookkeeping workflow writes version logs to `logs/version_vX.Y.Z_log.md`.
4. `.gitignore` excludes `.aib_memory/logs/aib-action-*.log` and no longer contains the old `logs/aib-action-*.log` pattern.
5. All tests in `tests/test_menu.py` pass, including the updated `TestMakeLogPath` assertions.
7. `.aib_memory/context.md` reflects the new log location after regeneration.

## Assumptions

- A1: `_make_log_path()` already calls `log_dir.mkdir(parents=True, exist_ok=True)`, so `.aib_memory/logs/` will be auto-created on first use. No change to `initialize.py` is required.
  - Risk if false: First action execution will fail with `FileNotFoundError` if the directory is not pre-created.

- A2: The root `logs/` directory must remain in VCS; it holds committed release version log files that must not be moved. Treating it as removable would silently delete release history.
  - Risk if false: Low — removal is explicitly out of scope and not performed; assumption is a guard against drift.

- A3: `README.md` and `docs/Development_and_Deployment_Specification.md` require no edits. All their `logs/` references describe release version log behaviour or the `logs/` directory preservation requirement, both of which remain accurate after this change.
  - Risk if false: Stale documentation would be introduced, but the described behaviour (version logs in `logs/`) remains correct.
