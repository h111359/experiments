# Request

## Goal

Refactor the AIB analysis and planning workflow so that:

1. `request.md` becomes the **single implementation-driving artifact** — all information needed by `implement` lives there.
2. Open questions, solution-option alternatives, assumptions, plan (WBS), testing intent, and documentation touchpoints are written directly into `request.md` by the `create-analysis` action instead of into separate questionnaire or plan files.
3. The `create-analysis` action absorbs all planning responsibilities previously held by `create-plan`.
4. `<ITERATION_ID>-questionnaire.md` and `<ITERATION_ID>-plan.md` are no longer generated; their source prompts and the plan convention are removed from the framework.
5. The analysis document becomes a reasoning/knowledge-capture log only — not an implementation driver.
6. **Newly Added after the last analysis - adapt the rest of the analysis accordingly:** To be removed iterations of a request. No more prefixed analysis, questionnaire and plan files. Hence the iterations are not needed. Remove from the application menu options, scripts, prompts, conventions, contexts, documentation Readme and Concepts.md.
7. **Newly Added after the last analysis - adapt the rest of the analysis accordingly:** Create a prompt which applies the answered questions in the request and removes the answered questions (replace the answered questions with instructions)
8. **Newly Added after the last analysis - adapt the rest of the analysis accordingly:** Add notation if the user wants to add/modify something in the request and wants to ask the AI to revise the request accordingly. This could be by adding a special type instruction in the request file and running the analysis prompt again.

## Background

The current AIB workflow produces up to three iteration artifacts per analysis cycle: `<ITERATION_ID>-analysis.md`, `<ITERATION_ID>-questionnaire.md`, and `<ITERATION_ID>-plan.md`. The questionnaire contains decision-point questions that duplicate Decision Points sub-section (4.6.6) captured in analysis. The plan contains execution tasks and testing strategy derivable from the request. This multi-file structure increases maintenance effort, splits implementation-relevant information across three files, and requires the `implement` action to read and reconcile multiple artifacts. Consolidating everything into `request.md` — the already-canonical specification artifact — eliminates redundancy, ensures `implement` has a single authoritative source, and reduces the overhead of AIB operation.

  - `## Documentation` — list of documentation files that must be revised because of the request.
  - `## Questions & Decisions` — open questions and solution-option forks as QID-tagged inline question blocks.
- Re-run behaviour (idempotent):
  - `## Questions & Decisions`: preserve all questions whose QID exists AND at least one checkbox is `[x]` or free-text block is non-empty; append new unanswered questions; append `> [!NOTE] DECISION REVIEW NEEDED: <reason>` to any answered question whose prior answer conflicts with updated analysis context.
  - `## Assumptions`, `## Plan`, `## Testing`, `## Documentation`: fully replace on each re-run (AI-generated sections; no user-entered data).
- Add these optional sections **only when they have content**; never add empty shell sections.

## Constraints

- `request-template.md` MUST NOT be changed. The seed template keeps exactly 6 sections.
- The `create-analysis` action MAY add optional sections 7–11 to `request.md`, but MUST NOT add empty shell sections.
- When analysis is re-executed (idempotent re-run), answers in `## Questions & Decisions` already entered by the user MUST be preserved unless an explicit conflict is detected and flagged with `> [!NOTE] DECISION REVIEW NEEDED: <reason>`.
- `## Plan`, `## Testing`, `## Documentation`, `## Assumptions` sections in `request.md` are AI-generated and may be fully replaced on re-run.
- The change MUST NOT break the existing test suite (`tests/*.py`) or alter the behavior of any tool script.
- Tool scripts that validate `request.md` must not require exactly 6 sections after this change (the convention allows additional optional sections).

## Success criteria

- Running `create-analysis` on an active request produces `<ITERATION_ID>-analysis.md` only as an iteration artifact; no `<ITERATION_ID>-questionnaire.md` or `<ITERATION_ID>-plan.md` is generated.
- The updated `request.md` after analysis contains `## Assumptions`, `## Plan`, `## Testing`, `## Documentation`, and (if open questions exist) `## Questions & Decisions` sections.
- Re-running `create-analysis` on a request where the user has answered questions in `## Questions & Decisions` preserves all answered questions unmodified.
- `implement` reads `request.md` as the sole specification source; it does not read analysis, questionnaire, or plan files.
- The revised `analysis-convention.md` contains 8 sections (down from 13); all implementation-relevant sections are absent.
- `aib-questionnaire.md`, `aib-plan.md`, and `plan-convention.md` are deleted from the framework.
- The existing test suite passes without modification.
- `request-convention.md` and `analysis-convention.md` pass a manual review confirming backward compatibility for all closed requests.

## Assumptions

- A1: The Disambiguation Questionnaire sub-section (4.12) and section 18 (Canonical Disambiguation Questionnaire) of `analysis-convention.md` should be removed because their purpose was to feed the plan's Decision Gates section, which no longer exists as a separate artifact. Resolved disambiguation decisions are captured in `## Plan` task descriptions; unresolved ones become Q&D items in `request.md`.
  - Risk if false: Disambiguation coverage of canonical technical questions may decrease; mitigated by the `## Questions & Decisions` mechanism in `request.md`.

- A2: `## Plan`, `## Testing`, `## Documentation`, and `## Assumptions` sections in `request.md` are fully AI-generated and may be replaced entirely on re-run. Only `## Questions & Decisions` contains user-entered data requiring preservation.
  - Risk if false: Users who manually edit Plan/Testing/Documentation/Assumptions would lose their edits on re-run. Mitigated by doc note.

- A3: The WBS task schema in `## Plan` preserves the existing plan-convention schema (Intent, Inputs, Outputs, Procedure, Done Criteria, Dependencies, Risk Notes) and adds `External Interfaces` and `Environment & Configuration` per task, as explicitly specified.
  - Risk if false: If other plan sections (milestones, acceptance handover) are also expected, the Plan section would be incomplete.

- A4: Removing `plan-convention.md` does not require changes to tool scripts. The `create-plan` action was AI-only (no backing script exists).
  - Risk if false: Any script referencing `plan-convention.md` would error. Verified via grep: no such script references found.

- A5: `aib-documentation.md` prompt does not depend on plan or questionnaire artifacts and requires no changes.
  - Risk if false: If it reads plan/questionnaire as inputs, it would break after deletion. Verify before implementing.

- A6: The `## Testing` section defines test cases at intent level only (what to test and expected observable outcome), not executable scripts. Test scripts are produced during `implement`.
  - Risk if false: If intent-level test descriptions are insufficient for implementation, a more detailed format may be needed.
