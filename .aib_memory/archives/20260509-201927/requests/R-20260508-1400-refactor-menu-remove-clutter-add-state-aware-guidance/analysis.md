- **Title:** Refactor menu: remove clutter, add state-aware guidance

  - *(new)* `unknown` — catch-all defensive fallback when no other state matches

- **No Q-blocks generated:** All decision points were deterministically resolvable from existing workspace documentation and the input.md amendment content.

## Domain Knowledge Essentials

**AIB (AI Builder):** A minimal, model-agnostic, file-first framework for specification-driven development inside a local repository workspace. Its workflows are deterministic and convention-governed.

**Active request:** A request row in `requests_register.md` whose `state` column equals `Active`. At most one Active request may exist at any time (FR-001).

**`input.md`:** The ephemeral input channel between developer and AI agent. Contains `## Active request`, `## Options`, and `## Input` structural sections. A `## Questions` section may be temporarily appended by `aib-analysis.md` and is cleared after the Q&A cycle.

**`request.md`:** The canonical artifact indicating that analysis has been performed for the active request. Its presence at `.aib_memory/request.md` is the lifecycle indicator — not `analysis.md`. Created by `aib-analysis.md`; absent until analysis runs.

**`context.md`:** The product context document generated by `aib-context.md`. Empty (or absent) until the first context generation run. An empty `context.md` indicates the workspace has not been fully initialised from an AIB product-knowledge standpoint.

**Workspace lifecycle state machine:** An enumerated value derived at runtime from the combination of: active request presence, `request.md` presence, `input.md ## Input` content, and `input.md ## Questions` content. Seven named states:

| State key | Active request | request.md | ## Questions | ## Input |
|---|---|---|---|---|
| `idle` | No | — | — | Empty |
| `input_ready` | No | — | — | Non-empty |
| `request_incomplete` | Yes | Absent | Empty | Empty |
| `questions_pending` | Yes | Any | Non-empty | Any |
| `implementation_ready` | Yes | Present | Empty | Empty |
| `amendment_pending` | Yes | Present | Empty | Non-empty |
| `unknown` | — | — | — | Exception/fallback |

**Detection order (evaluation sequence within `_detect_guidance_state()`):**
1. `has_active_request == False` → `idle` or `input_ready` (from `## Input` content)
2. `has_active_request == True`, `## Questions` non-empty → `questions_pending` *(priority check — must precede request.md check)*
3. `has_active_request == True`, no questions, `request.md` absent → `request_incomplete`
4. `has_active_request == True`, no questions, `request.md` present, `## Input` non-empty → `amendment_pending`
5. `has_active_request == True`, no questions, `request.md` present, `## Input` empty → `implementation_ready`
6. Any unhandled exception in the detection body → `unknown`

**Context.md empty check:** An orthogonal check, independent of state detection. If `.aib_memory/context.md` is absent or empty (after stripping whitespace), an additional guidance line is displayed alongside the state guidance: `"Context file is empty — run context generation first: Execute \`.aib_brain/prompts/aib-context.md\`"`. Shown for any state.

**Allow-list vs. deny-list:** An allow-list enumerates what is permitted; everything else is excluded by default. A deny-list (`EXCLUDE_SCRIPTS`) enumerates what is blocked. Allow-lists are safer for a small, fixed action surface — new items are excluded automatically unless explicitly registered.

**Impacted roles:** Developer (daily menu user), AIB Maintainer (owns `menu.py`), AI Automation Agent (generates Q-blocks processed by `questions_pending` state).

## Technical Knowledge & Terms

**`menu.py`** — Interactive launcher at `.aib_brain/tools/menu.py`. Python 3.10+, standard library only (NFR-004). Renders the menu in a 3-second polling loop, accumulates output into `io.StringIO`, and flushes to stdout in a single write (blink-free).

**`MenuState`** — Frozen dataclass: `active_request_id`, `active_request_folder`, `active_request_title`, `.has_active_request`. Derived from `requests_register.md` by `resolve_menu_state()`. **Unchanged by this request.**

**`_detect_guidance_state(state: MenuState, workspace: Path) -> str`** — New helper. Returns one of seven state key strings. Reads `input.md` for section content extraction. Reads `request.md` presence. Entire body wrapped in `try/except Exception` returning `"unknown"` on unexpected error. `OSError` on any file read treated as empty string.

**`_GUIDANCE_MESSAGES: dict[str, list[str]]`** — Module-level dict mapping state key → `list[str]`. Single-line states: one-element list. Two-line states (`input_ready`, `questions_pending`, `implementation_ready`): two-element list. `render_menu()` iterates the list and writes each string as a line.

**`_is_context_empty(workspace: Path) -> bool`** — New helper. Reads `.aib_memory/context.md`; returns `True` if file absent or content empty after stripping. Used by `render_menu()` to decide whether to append the context.md extra guidance line.

**`build_script_actions(tools_dir: Path)`** — Body replaced from dynamic auto-discovery to a hard-coded one-entry list: `reverse-engineer.py` titled "Reverse Engineer".

**`EXCLUDE_SCRIPTS`** — To be deleted entirely. No callers remain after `discover_tool_scripts()` removal.

**`discover_tool_scripts(tools_dir)`** — To be deleted. Only caller is `build_script_actions()`.

**`_REFRESH_ACTION`** — Sentinel dict causing a "Refresh" menu item. To be deleted.

**`print_prompt_reference()`** — Zero callers. To be deleted.

**`render_menu(state, script_actions, selected_index, workspace)`** — Gains `workspace: Path` parameter. Inline prompt block removed. Guidance block added (state-aware lines + optional context.md extra line).

## Research Results

**Pattern scan — codebase:**

- `_detect_guidance_state()` can reuse the same `## section` extraction approach used conceptually in `aib-analysis.md`'s toggle detection. The pattern: read file text, split on `\n## ` to isolate sections, take the segment following the target heading marker, strip whitespace. Same algorithm serves both `## Input` and `## Questions` extraction.

- `discover_tool_scripts()` is called exclusively from `build_script_actions()`. No other production callers. `render_menu()` is not called directly in any test. These confirm the two deletions and the signature change are low-risk.

- `_REFRESH_ACTION` injection occurs at `choose_action()` line ~635: `filter_visible_actions(all_script_actions, state) + [_REFRESH_ACTION]`. This is the sole construction point. The ENTER and DIGIT handlers that check `action.get("type") == "refresh"` are the two consumption points. All three must be removed atomically.

- `filter_visible_actions()` docstring mentions `EXCLUDE_SCRIPTS` — must be updated.

- `amendment_pending` is novel: no prior code pattern exists. Detection: active + request.md present + `## Input` non-empty. Same `## Input` extraction as the `input_ready` / `idle` branch confirms shared helper approach.

- `unknown` as a try/except catch-all is a clean pattern. The function returns a string in all branches; the caller never sees an exception.

## External Benchmarking

**Reference 1 — git contextual status display (Git SCM, open-source)**
- `git status` adapts output text to exact repository state (merge conflict, rebase in progress, detached HEAD, clean). For states with multiple valid next steps, it lists both.
- Takeaway: State-specific guidance with inline command invocations reduces developer cognitive load. Two-line format for multi-path states is a documented CLI UX convention.
- Adoption: Directly adopted — `questions_pending`, `input_ready`, and `implementation_ready` use two-line format; single-path states use one line.

**Reference 2 — GitHub CLI `gh pr status` branching guidance (GitHub CLI, open-source)**
- `gh pr status` shows different content per PR state; for two-option states, secondary option is prefixed with "Or" to communicate relative weight (primary recommendation first).
- Takeaway: "Or" prefix for secondary options is idiomatic in developer CLIs and communicates optionality without implying equal weight.
- Adoption: Adopted — `questions_pending` secondary option uses "Or run implement directly —"; `input_ready` and `implementation_ready` secondary options follow the same pattern.

**Reference 3 — npm CLI workspace-not-initialised checks**
- npm detects uninitialised workspaces and displays a targeted warning before allowing further commands. Warning is shown alongside (not instead of) normal command output.
- Takeaway: Cross-cutting "system not ready" warnings as additional lines (not interrupting normal flow) is a standard pattern.
- Adoption: Adopted — context.md empty check adds an extra line below the state guidance without suppressing it.

## AI Copilot Suggestions

> This section is for human review only. `implement` MUST NOT read or act on this section.

**1. State key consistency should be enforced by a module-level assertion.**
The seven state keys are string literals threaded through `_detect_guidance_state()`, `_GUIDANCE_MESSAGES`, and `test_menu.py`. A typo produces a silent `KeyError` at render time — caught by `unknown` fallback, but invisible to tests unless explicitly covered. Suggestion: Add a module-level assertion or dedicated test verifying `set(_GUIDANCE_MESSAGES.keys()) == {"idle", "input_ready", "request_incomplete", "questions_pending", "implementation_ready", "amendment_pending", "unknown"}`.

**2. The `unknown` state guidance needs visual differentiation.**
A message like "Workspace state is unknown" looks like any other guidance line. Suggestion: Prefix with a warning sigil (`"⚠ Workspace state could not be determined. Check .aib_memory/ for inconsistencies."`). One character that significantly improves discoverability of an error condition.

**3. `_is_context_empty()` should be an independently testable helper.**
Embedding the context.md check inside `render_menu()` makes it hard to test without mocking `sys.stdout`. Suggestion: Extract `_is_context_empty(workspace: Path) -> bool` as a named helper, directly testable without rendering overhead.

**4. Scope is well-calibrated — no over-engineering.**
The seven-state model covers the full AIB lifecycle plus two defensive guards and one catch-all. No unnecessary states, no changed existing behavior. The context.md check fills a real onboarding gap without scope creep.

**5. `questions_pending` priority must be tested under simultaneous conditions.**
SC-12 must include a test where both `request.md` IS present AND `## Questions` IS non-empty simultaneously, asserting `questions_pending` (not `implementation_ready`). This is the only test that validates priority ordering vs. isolated detection.

## Multi-Perspective Stakeholder Review

### Senior Solution Architect

The refactoring replaces a fragile auto-discovery model with a deterministic allow-list, eliminating a category of silent menu expansion failures. The state machine expansion from 5 to 7 states is well-motivated: `amendment_pending` fills an existing gap (the developer is literally in this state right now, as they wrote amendments to `input.md` with an active request), and `unknown` is a textbook defensive fallback pattern for a function that reads from the filesystem. The evaluation order is architecturally sound — `questions_pending` is checked first, preventing it from being masked by `request.md` presence.

Key findings:
- State machine is finite, deterministic, and non-overlapping given the evaluation order — correctly designed.
- `_detect_guidance_state()` wrapped in try/except is the right defensive pattern for a filesystem-dependent function called on every render cycle.
- Extracting `_is_context_empty()` as a named helper improves testability at negligible cost.
- `render_menu()` signature change (`workspace: Path`) has zero test breakage.

### Product Owner

The request delivers measurable developer experience improvements. The menu no longer shows irrelevant items; the guidance block answers "what do I do next?" without requiring lifecycle memorization. The renamed states are semantically clearer: `implementation_ready` communicates intent more directly than `active_with_analysis`. `amendment_pending` fills a real gap: previously a developer who wrote to `input.md` with an active request had no visible indicator of what to do next.

Key findings:
- `amendment_pending` is immediately observable in the current session — the developer is demonstrating it.
- Context.md empty check is a valuable first-run guard for new users.
- All 17 SCs are measurable and map to automated tests.
- Scope is proportional to the stated goal.

### User (Developer)

The guidance block solves the most common pain point: knowing what to do next after returning to the workspace. The renamed states use natural language phrases the developer would use to describe their situation. `amendment_pending` is the highest-value new state — it makes explicit that "I wrote something new to input.md; I should re-run analysis." The context.md empty check is a one-time onboarding aid that disappears once context has been generated.

Key findings:
- `questions_pending` priority is the right user-visible behavior — Q-blocks must not be silently ignored.
- `unknown` state message should be visually distinct (warning prefix) to avoid confusion with normal guidance.
- Two-line guidance messages may wrap on narrow terminals; manual verification warranted.

### Security Officer

All changes are confined to UI rendering logic in a local Python CLI tool with no network access. New file reads are workspace-relative, read-only, and wrapped in `OSError` handlers. No user-supplied input is interpolated into file paths or shell commands. Hard-coded allow-list reduces attack surface compared to auto-discovery.

Key findings:
- No new attack surface. File reads are local, read-only, and guarded.
- Guidance strings are display-only; no execution or deserialization occurs.
- Hard-coded allow-list is a minor integrity improvement over auto-discovery.

### Data Governance Officer

All data accessed is local workspace engineering documentation. No PII, no regulated data, no external data providers. The context.md empty check reads file content only to determine emptiness (boolean); no content is displayed, logged, or transmitted.

Key findings:
- No data leaves the workspace filesystem.
- No retention, classification, or compliance concerns introduced.
- Hard-coded allow-list removes a surface where unexpected script execution could occur.
